# 亮点二：积分抽奖高并发链路

## 这条亮点的核心价值

这条亮点要讲出的不是“我做了一个抽奖功能”，而是：

1. 我把一个强业务属性的功能，做成了高并发交易链路。
2. 我同时处理了算法性能、风控规则、并发一致性和异常补偿四类问题。
3. 这条链路能体现我对设计模式、缓存、锁、事务和容灾的综合理解。

## 30 秒答法

我在项目里设计过一条高并发积分抽奖链路，抽奖算法上用了 Alias Method，把单次抽奖复杂度做到 O(1)；架构上通过策略模式支持不同抽奖策略，通过责任链串联积分校验、限流、冷却期和黑名单等风控规则；并发控制上使用 Redisson 分布式锁、Redis 库存预热、数据库扣减和失败回滚机制，保证抽奖在高并发下既快又一致。

## 1 分钟展开答法

抽奖看起来是一个很简单的业务，但真正做起来难点很多。第一个难点是性能，如果每次抽奖都遍历奖池做概率计算，在高并发场景下开销会很高；第二个难点是风控，不能让用户无限点、刷接口或者通过异常行为套利；第三个难点是一致性，像积分扣减、奖品库存扣减、中奖记录落库这些步骤一旦中间失败，就容易出现扣了积分但没中奖、或者库存超发的问题。

所以我把它拆成四层来做。算法层用 Alias Method 优化抽奖采样；业务层用策略模式支持别名算法、动态权重、保底等策略切换；风控层用责任链把积分校验、频率限制、冷却期、黑名单这些规则串起来；一致性层则通过 Redisson 用户锁防止并发重入，库存采用 Redis 预热 + 数据库同步扣减 + 异常回滚，最终保证接口吞吐、风控可扩展和数据一致性。

## 你在项目里具体做了什么

### 1. 为什么抽奖算法用 Alias Method

- 普通加权随机通常是每次抽奖都扫描概率区间，时间复杂度接近 O(n)。
- 如果奖品数量不小、抽奖请求又很频繁，这部分会成为热点。
- Alias Method 的核心思路是：
  - 先做一次 O(n) 预处理，构建 `prob` 和 `alias` 两张表
  - 每次抽奖只做一次随机列选择和一次概率判断
  - 单次采样复杂度降到 O(1)

你可以这样概括：

“我把复杂度前移到了预处理阶段，用空间和初始化成本换运行时吞吐，这种 trade-off 很适合高频抽奖接口。”

### 2. 为什么要用策略模式

这个点体现的是扩展性。

- 不同运营活动对抽奖规则要求不一样。
- 有的场景追求纯概率公平，有的需要动态权重，有的要保底。
- 所以我没有把抽奖逻辑写死在一个大方法里，而是抽成 `LotteryStrategy` 接口。
- 这样后续新增策略时，不需要改主流程，只要新增实现类并在工厂里注册即可。

你可以讲一句：

“策略模式让我把抽奖算法从业务交易主链路里拆开，避免一个方法里堆满 if-else，后续扩展活动玩法会更稳。”

### 3. 风控为什么用责任链

这是面试里很好讲的地方。

- 抽奖前要做的检查很多，比如积分够不够、用户今天次数是否超限、是否还在冷却期、是否在黑名单。
- 这些规则有明显的先后顺序，而且未来还可能继续加。
- 所以我用了责任链模式，把每个规则拆成一个独立处理器，再由链构建器按顺序拼起来。
- 这样规则之间低耦合，也方便按成本高低排序，先做便宜检查，减少无效请求继续进入主流程。

一个比较加分的说法是：

“责任链解决的不只是代码结构问题，更是风控规则持续膨胀后的维护成本问题。”

### 4. 如何保证高并发下不重复抽奖

- 我在用户粒度上加了 Redisson 分布式锁，锁键通常按 `userId` 维度构建。
- 这样能防止用户重复点击、前端重试、网络抖动导致的并发重入。
- 获取不到锁就快速失败，避免同一用户并发扣积分或重复扣库存。

如果面试官继续问“为什么不是数据库锁”，你就说：

“用户抽奖是高频短事务，先在应用层做分布式锁会更轻，能在进入数据库前就拦掉大量重复请求。”

### 5. 如何保证库存一致性

这是最重要的一段。

- 奖品库存我没有只信数据库，也没有只信 Redis，而是做了双层控制。
- 先把库存预热到 Redis，抽奖时优先用 Redis 做快速判断和扣减。
- 同时再同步扣数据库库存，作为最终一致性的持久化保障。
- 如果数据库扣减失败，就回滚 Redis 库存，避免缓存和数据库不一致。
- 如果后续流程异常，也有显式的库存回滚逻辑。

你可以总结成一句话：

“我做的是 Redis 抗并发、数据库保底、异常路径可回滚，而不是把一致性完全赌在某一层。”

### 6. 如何处理异常补偿

这个点很容易加分，因为很多人只会讲成功路径。

- 抽奖链路至少包含：风控校验、积分扣减、抽奖、库存扣减、发奖、记录落库。
- 只要中间一步失败，都可能出现业务不一致。
- 所以我在关键节点设计了补偿逻辑，比如库存已扣但后续异常时回滚库存。
- 同时通过事件发布把统计、监控等后置逻辑从主链路剥离，降低核心事务复杂度。

## 为什么这样设计

### 为什么不用一个大事务包住所有逻辑

因为抽奖是高频接口，大事务会把锁持有时间拉长，而且很多后置逻辑不应该进入强事务范围。

### 为什么积分和库存都要单独考虑一致性

因为抽奖本质上是“扣用户资源，发系统资源”的交易链路，两个资源都必须有明确失败路径。

### 为什么风控和抽奖策略要解耦

因为风控回答的是“这个请求能不能做”，策略回答的是“允许做之后怎么抽”，这两个关注点不同。

## 高频追问与参考回答

### Q1：Alias Method 为什么快？

答法：

“因为它把每次抽奖都要做的概率累加，提前变成了两张查找表。运行时不需要遍历奖池，只要随机选列再抛一次硬币，复杂度就是 O(1)。”

### Q2：如果 Redis 和数据库不一致怎么办？

答法：

“我不会只做单向扣减，而是把数据库作为持久化底座，Redis 作为高并发前置层。数据库扣减失败时立即回滚 Redis，另外还能通过定时同步或补偿逻辑做二次修正。”

### Q3：为什么用户锁和库存锁都可能需要？

答法：

“用户锁是为了防止同一用户重复提交，库存锁是为了防止同一奖品被并发超发，它们保护的对象不同，所以有必要分层控制。”

### Q4：责任链顺序怎么定？

答法：

“一般先做成本低、过滤能力强的检查，比如黑名单、次数限制、冷却期，再做积分和库存这类更贴近交易的数据校验，这样能减少无效请求进入后续链路。”

### Q5：如果让我继续优化，你会做什么？

你可以答：

- 把奖品配置和概率表进一步缓存化
- 引入更细粒度的幂等控制
- 对高风险用户做动态限流或降权
- 对事件消费者增加更完整的重试和死信治理

## 面试时容易踩的坑

### 不要只讲“用了分布式锁”

只讲锁会显得很浅。一定要把锁和“防重复提交”“库存一致性”这两个业务问题绑定起来。

### 不要忽略异常路径

抽奖最容易被问的就是失败补偿，必须提前准备好。

### 不要说成“绝对强一致”

更稳的说法是：

“核心交易步骤尽量保证同步一致，后置统计和监控通过异步事件解耦；缓存与数据库之间通过回滚和同步机制控制不一致窗口。”

## 一句话背诵版

我把积分抽奖做成了一条高并发交易链路：前面用责任链做风控，核心用 Alias Method 做 O(1) 抽奖采样，用户侧用 Redisson 锁防重入，库存侧用 Redis 预热 + 数据库扣减 + 回滚机制保证一致性，同时把统计和监控通过事件异步解耦出去。
