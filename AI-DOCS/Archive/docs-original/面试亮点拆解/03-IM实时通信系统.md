# 亮点三：IM 实时通信系统

## 这条亮点的核心价值

这条亮点要体现的是：

1. 你做过长连接系统，不只是普通 HTTP 接口。
2. 你理解连接鉴权、在线状态、心跳保活、消息广播和多节点同步。
3. 你把即时通信能力做成了“实时链路 + 管理能力 + 扩展能力”的组合。

## 30 秒答法

我在项目里做过一个基于 WebSocket + Sa-Token 的 IM 实时通信系统，连接建立阶段就完成登录态鉴权，支持在线状态维护、消息撤回、禁言、踢人和系统广播等能力。为了解决多实例部署下的消息同步问题，我又引入 Redis Pub/Sub 做跨节点广播桥接，同时把心跳、在线用户和房间关系维护放到 Redis，保证聊天室的实时性和可扩展性。

## 1 分钟展开答法

这个模块核心难点不在于“把 WebSocket 连起来”，而在于如何把连接、鉴权、在线状态和多节点广播稳定串起来。我的做法是把系统拆成四部分：第一部分是连接层，在 WebSocket 握手阶段通过 Sa-Token 校验 token，把用户身份放进连接上下文；第二部分是会话层，服务端维护 session，并同步在线用户、房间关系和心跳时间；第三部分是业务层，支持消息发送、撤回、禁言、踢人和系统广播；第四部分是集群层，通过 Redis Pub/Sub 同步多节点消息，避免单机内存广播在扩容后失效。

## 你在项目里具体做了什么

### 1. 如何做 WebSocket 鉴权

- HTTP 接口鉴权和 WebSocket 鉴权不是一回事。
- WebSocket 建连后是长连接，不能只依赖前端自己传 userId。
- 我在握手拦截器里读取 token，并通过 Sa-Token 的登录态能力反查登录用户。
- 校验通过后，把 `userId`、`username` 等信息放到 WebSocket session attributes 中，后续消息处理直接从上下文拿用户身份。

这个点面试里一定要讲清楚：

“我不是在消息体里信任前端传过来的用户身份，而是在握手阶段完成身份绑定，后续所有消息都基于服务端连接上下文处理。”

### 2. 如何维护在线状态

- 在线状态不是简单的一个内存 map。
- 单机下我会维护 session 容器，便于快速广播。
- 同时把在线用户、房间成员和心跳时间写到 Redis：
  - 在线用户信息放 Hash
  - 房间成员关系放 Set
  - 心跳时间单独维护
- 这样做的好处是：
  - 查询在线人数更方便
  - 多节点下共享状态更容易
  - 节点重启后更容易做恢复和清理

### 3. 如何做心跳保活

- 长连接最大的问题之一就是“假在线”。
- 所以我设计了心跳机制：客户端定期发送心跳包，服务端只更新心跳时间，不做重数据库写。
- 超时未上报的连接由定时任务清理，并同步做离线处理。

你可以这样讲：

“心跳更新我尽量走 Redis，不走数据库，是为了避免高频心跳把数据库打成热点。”

### 4. 如何实现撤回、禁言、踢人

这部分是业务能力，但很能体现系统完整性。

- 撤回消息：校验是否本人、是否在允许时间窗口内，然后更新消息状态并广播撤回通知。
- 禁言：发消息前先检查用户禁言状态，管理员可按时长或永久禁言。
- 踢人：管理员触发后给指定用户定向发送踢出消息，并主动关闭对应连接。
- 系统广播：管理员发布系统消息后，统一推送到在线用户。

一句话讲法：

“我把 IM 设计成实时通信能力和后台治理能力一体化，而不是只有一个单纯的消息收发通道。”

### 5. 多节点下怎么广播

这是最容易体现难度的地方。

- 单机时可以直接遍历本机 session 广播。
- 但部署成多实例后，本机 session 容器只知道自己节点的连接。
- 所以我增加了 Redis Pub/Sub 广播桥：
  - 本节点产生广播事件后先发到 Redis channel
  - 各节点订阅该 channel
  - 收到消息后再转发给自己节点上的连接

这句话很关键：

“Redis Pub/Sub 在这里不是替代 WebSocket，而是做 WebSocket 集群化下的节点间消息桥接。”

## 为什么这样设计

### 为什么不用纯内存维护在线状态

因为单机可以，多机不行；一旦扩容，纯内存状态就天然不共享。

### 为什么集群广播选 Redis Pub/Sub

因为聊天室消息强调低延迟和轻量同步，这类广播型消息用 Pub/Sub 足够直接，接入成本也低。

### 为什么心跳不落数据库

因为心跳是高频行为，不适合直接把数据库当实时状态存储。

## 高频追问与参考回答

### Q1：WebSocket 如何鉴权？

答法：

“我是在握手阶段通过 Sa-Token 校验 token，校验通过后把用户身份写进 session 上下文，后续消息都信任服务端上下文，不信任前端透传身份。”

### Q2：多实例部署后，A 节点的消息怎么发给 B 节点上的用户？

答法：

“我在节点内仍然走 WebSocket，本节点外通过 Redis Pub/Sub 做广播桥接。消息先发到 Redis，再由每个节点消费后转发给本地连接。”

### Q3：怎么处理掉线但服务端还以为在线的问题？

答法：

“通过心跳时间和定时清理机制来剔除超时连接，避免出现假在线状态。”

### Q4：为什么撤回和踢人要走实时通知？

答法：

“因为这类操作对聊天室一致性要求高，不能只改数据库不通知前端，否则用户看到的消息状态会不一致。”

## 面试时容易踩的坑

### 不要把 IM 讲成“发消息 + 收消息”

一定要讲连接鉴权、在线状态、心跳、集群广播。

### 不要忽略集群问题

如果你讲了 Redis Pub/Sub，就说明你考虑过从单机到多机的演进。

### 不要说“WebSocket 自带可靠性”

更稳的说法是：

“WebSocket 解决的是长连接和低延迟，不自动解决鉴权、假在线、集群广播和状态同步，这些都需要服务端自己设计。”

## 一句话背诵版

我做的 IM 系统核心不是 WebSocket 本身，而是围绕它补齐了握手鉴权、在线状态、心跳保活、消息治理和 Redis Pub/Sub 跨节点广播，最终让聊天室从单机通信能力升级成了可扩展的实时系统。
