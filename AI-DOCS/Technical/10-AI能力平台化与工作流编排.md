# 亮点十：AI 能力平台化与工作流编排

## 简历可直接贴

- 搭建统一 AI 服务平台，按 Agent/Skills 思路编排多阶段 AI 工作流，沉淀 Prompt Engineering、上下文注入、结构化输出解析与 fallback 兜底能力，支撑 AI 出题、面试评估、JD 解析、简历匹配、SQL 诊断等核心场景闭环。

## 这条亮点的核心价值

这条亮点要讲出来的是：

1. 你做的不是单点 AI 接口，而是一套平台化 AI 能力接入方案。
2. 你不是只会“调提示词”，而是能把 Prompt、上下文、输出协议和异常治理串成工程体系。
3. 你讲这条亮点时，既能往 AI 工程落地上靠，也能往 Agent / Skills / 工作流编排上拔高。

## 30 秒答法

我在项目里做过统一 AI 服务平台，不是把大模型零散接到几个业务里，而是按 Agent/Skills 的思路把多类 AI 能力抽象成可编排的工作流。具体上，我沉淀了 Prompt Engineering、上下文注入、结构化输出解析和 fallback 兜底能力，支撑 AI 出题、面试评估、JD 解析、简历匹配和 SQL 诊断等核心场景，让 AI 从单次调用升级成了可复用、可扩展的业务闭环能力。

## 1 分钟展开答法

这个亮点我最想表达的是，我做的不是“某个接口调了 AI”，而是一套 AI 能力平台化方案。因为项目里 AI 模拟面试、求职作战台和 SQL 优化工作台这些场景都不是单步任务，而是天然带有链路属性的，比如模拟面试会经历出题、答题、评价、追问、总结，求职作战台会经历 JD 解析、简历匹配、差距识别、行动计划，SQL 优化也会经历诊断、重写、收益对比和结果沉淀。所以我的做法是，先把这些能力按 skill 维度拆开，再通过统一 AI 服务平台完成工作流注册、场景路由、上下文注入、结构化输出约束、结果解析和 fallback 兜底。这样做之后，不同业务场景就不是各自零散接 AI，而是共享同一套能力底座，后续扩展新场景也会更快。

## 你在项目里具体做了什么

### 1. 为什么要做“统一 AI 服务平台”

- 如果每个模块自己接大模型，最后一定会出现参数不统一、Prompt 风格不一致、返回格式不可控、异常处理不可复用的问题。
- 所以我把 AI 能力统一收口，按业务场景注册 workflow，再由统一服务层去编排调用。
- 这样做的核心价值不是“代码更整齐”，而是 AI 能力可以被平台化治理。

你可以这样说：

“我不是在业务模块里硬塞 AI，而是先搭能力底座，再让业务场景复用这套底座。”

### 2. Prompt Engineering 具体做了什么

很多人一说 Prompt 优化就显得很虚，所以一定要讲具体：

- 我先拆清业务目标，到底是出题、评价、总结、匹配还是诊断。
- 再补齐关键上下文，比如面试方向、难度、风格、追问轮次、问答记录、JD 文本、简历内容、SQL 和 EXPLAIN。
- 然后在 Prompt 里明确角色设定、评分规则、输出字段、边界约束和禁止项。
- 目标不是让模型“写得更花”，而是让它“输出更稳、更准、更容易被系统消费”。

一句话：

“我做 Prompt Engineering 的重点，是控制上下文和输出协议，而不是只调语气。”

### 3. 为什么要强调 Agent / Skills 思路

这部分是你把项目层级抬高的关键。

- 真实业务里的 AI 场景很多都不是单次问答，而是多个阶段串起来的任务链。
- 所以我把不同 AI 能力拆成 skill，由编排层按场景路由和组合。
- 比如模拟面试场景里，出题和评价是两个 skill；求职作战台里，JD 解析和简历匹配也是两个 skill。
- 每个 skill 都有自己的输入输出协议、边界和兜底策略。

你可以这样说：

“我把 AI 能力按 skill 做了能力拆解，再通过工作流把多个 skill 串成业务闭环，这种组织方式本质上已经很接近 Agent/Skills 的思路。”

### 4. 结构化输出和解析治理为什么重要

- AI 落地的难点不是“能不能生成内容”，而是“生成的结果能不能稳定进入业务系统”。
- 所以我会强约束模型返回 JSON 结构，把评分、优缺点、追问、建议、索引 DDL、收益对比等字段标准化。
- 再通过解析层兼容 output 包装、代码块、杂质文本和异常响应。
- 如果解析失败或工作流波动，再通过 fallback 返回基础结果，保证整条业务链路不至于中断。

这个点要讲得职业一点：

“我把大模型当成不稳定外部依赖来治理，而不是把它当成天然可信的黑盒。”

### 5. 这条亮点最后落到了哪些业务场景

- AI 模拟面试：AI 出题、答案评价、追问生成、总结报告。
- 求职作战台：JD 解析、简历匹配、能力差距识别、行动计划生成。
- SQL 优化工作台：结构化诊断、SQL 重写建议、收益对比和案例沉淀。

所以你一定要强调：

“我做的不是 AI 点缀功能，而是 AI 驱动的核心业务链路。”

## 高频追问与参考回答

### Q1：你说的“多阶段 AI 工作流”是什么意思？

答法：

“就是把一个完整业务目标拆成多个阶段，比如先解析，再匹配，再生成计划。前一个阶段输出的结构化结果，会成为后一个阶段的输入。这样比单次问答更适合真实业务闭环。”

### Q2：你说的 Prompt Engineering，优化的到底是什么？

答法：

“我主要优化三类内容：一是输入上下文的完整性，二是输出字段的结构化约束，三是规则边界，比如评分规则、追问条件和建议范围。目的不是让模型回答更像人，而是更稳定、更可消费。”

### Q3：你这算不算 Agent？

答法：

“如果从广义上看，它已经具备 Agent 的一些关键特征，比如能力拆解、上下文管理、技能路由和结果收敛；但我更愿意把它定义成面向业务场景的 Agent 式工作流编排，而不是通用智能体平台。”

### Q4：你这是不是 MCP？

答法：

“底层不是标准 MCP 协议实现，但设计思路和 MCP/Skills 很接近，比如能力注册、输入输出协议定义、统一编排和调用边界隔离。所以我一般会说这是类 MCP 的技能化抽象，而不是直接说我实现了标准 MCP Server。”

### Q5：为什么要做 fallback？

答法：

“因为 AI 在线上本质上是不稳定依赖，可能出现格式漂移、超时或者结果不可解析。fallback 的意义不是替代模型，而是保证业务链路不中断，平台仍然能给用户一个基础可用结果。”

## 面试时容易踩的坑

### 不要把这条亮点讲成“我会调提示词”

一定要讲：

- 平台化能力接入
- 多阶段工作流编排
- 结构化输出治理
- 异常回退和稳定性

### 不要上来就说“我做了 MCP Server”

除非你真的能讲清楚协议、transport、tool schema 和 resource，否则更稳的说法是：

“我做的是类 MCP / Skills 的能力抽象和业务工作流编排。”

### 不要把 Agent 讲成万能自动体

更好的讲法是：

“我做的是面向具体业务目标的 Agent 式编排，不是通用大而全智能体。”

### 不要只讲 AI，不讲业务落地

一定要落回：

- 模拟面试
- 求职作战台
- SQL 优化工作台

因为面试官最终看重的是你有没有把 AI 真正接进业务链路。
