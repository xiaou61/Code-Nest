# 亮点九：AI 工作流编排与技能化封装

## 这条亮点的核心价值

这条亮点要体现的是：

1. 你不是把 AI 当成单次调用能力，而是当成一条可编排的任务链路来设计。
2. 你能把不同 AI 能力抽象成可复用的“技能单元”，按场景路由和组合。
3. 你讲这条亮点时，会有明显的 Agent / Skills / 平台化味道，容易把项目层次抬高。

## 30 秒答法

我在项目里把多类 AI 能力做成了工作流化、技能化封装。比如模拟面试场景里，出题、答案评价、追问生成、总结报告本质上是连续任务；求职作战台里，JD 解析、简历匹配、能力差距识别、行动计划生成也是一条链路。我把这些能力拆成可复用的 skill，由统一编排层完成上下文注入、工作流路由、结果解析和异常回退，让 AI 从单点能力升级成了类似 Agent 的业务执行系统。

## 1 分钟展开答法

这个模块我不是按“接口”思路做的，而是按“任务编排”思路做的。因为很多 AI 场景本身就不是一步完成的，比如模拟面试不是只生成一道题，而是会经历出题、答题、评价、追问、总结多个阶段；求职作战台也不是只做 JD 分析，而是会继续做简历匹配、差距分析、行动计划和复盘建议。所以我的设计方式是，先把不同场景拆成一组可独立执行的技能单元，再通过统一编排层根据场景把前一步输出作为后一步输入，串成完整业务闭环。这样做的好处是，不同场景之间可以共享输入协议、输出结构和兜底策略，后续新增能力也只是在现有编排体系里扩展节点，而不是重新堆一套零散接口。

## 你在项目里具体做了什么

### 1. 为什么要做“技能化封装”

- 如果每个 AI 场景都只写成一个大而全接口，后续复用会很差。
- 所以我把能力拆成多个 skill，例如：
  - 出题 skill
  - 评价 skill
  - 总结 skill
  - JD 解析 skill
  - 简历匹配 skill
  - SQL 诊断 skill
  - SQL 重写 skill
  - 收益对比 skill
- 每个 skill 都有自己的输入、输出、业务边界和异常策略。

这句话很好用：

“我不是把模型能力写死在业务代码里，而是把它拆成了可复用的能力单元。”

### 2. 工作流编排层到底做了什么

- 根据业务场景选择合适的 workflow。
- 把前一阶段产出的结构化结果传给下一阶段继续处理。
- 统一做参数注入、上下文拼装和结果分发。
- 对失败节点做 fallback，不让单点失败拖垮整条链路。

你可以这样说：

“我把 AI 场景做成了有状态的任务流，而不是无状态的单次问答。”

### 3. 这为什么能往 Agent 方向去讲

面试时可以适当拔高，但要讲得像回事：

- Agent 的本质不是 UI 聊天框，而是任务拆解、上下文管理、能力调用和结果收敛。
- 我的这套设计里已经有了：
  - 场景识别
  - 技能路由
  - 上下文传递
  - 多阶段执行
  - 结果标准化
  - 异常回退
- 所以它虽然不是一个通用 Agent 平台，但在业务层已经具备了明显的类 Agent 编排特征。

推荐说法：

“底层不是标准化 Agent 框架，但业务抽象已经是 Agent/Skills 的组织方式。”

### 4. 如果你想往 MCP / Skills 靠，怎么讲更稳

这里不要一上来就说“我自己实现了完整 MCP 协议”，风险太大。

更好的讲法是：

- 我做过类 MCP 的能力抽象思路。
- 我把不同 AI 能力按 skill 维度定义了输入协议、输出协议和调用边界。
- 编排层负责根据任务场景选择 skill，并把结果标准化返回给业务系统。

也就是说，你可以讲“设计思路接近 MCP/Skills”，而不是说“我从零实现了标准 MCP Server”。

### 5. 多阶段 AI 链路为什么比单次问答更有价值

- 因为真实业务里，用户要的是结果，不是一次回答。
- 比如求职作战台里，JD 解析的结果本身不是终点，而是简历匹配和行动计划的输入。
- SQL 优化工作台里，诊断不是终点，后面还要重写、对比和沉淀。
- 所以真正有价值的是让 AI 参与完整业务链路，而不是只输出一段文本。

一句话：

“我做的是 AI 驱动的业务闭环，不是聊天式能力点缀。”

## 高频追问与参考回答

### Q1：你说的 skills 化封装，和普通 service 拆分有什么区别？

答法：

“普通 service 拆分更多是代码层拆分类，而 skills 化封装强调的是能力边界、输入输出协议和可编排性。也就是说，一个 skill 不只是一个方法，而是一个可被路由、可被组合、可被替换的能力单元。”

### Q2：你这算不算 Agent？

答法：

“如果从广义上看，它已经具备类 Agent 的关键特征，比如任务拆解、上下文管理、能力路由和结果收敛；但如果从严格定义上说，它不是通用 Agent 平台，而是面向业务场景的 Agent 式编排。”

### Q3：你这是不是 MCP？

答法：

“底层不是标准 MCP 协议实现，但在设计上借鉴了 MCP/Skills 的思想，比如能力注册、输入输出协议定义、调用边界隔离和统一编排。面向业务落地时，这种抽象已经足够支撑扩展和复用。”

### Q4：为什么要强调可编排性？

答法：

“因为 AI 场景很多都是链式任务，一个阶段的结果会成为下一个阶段的输入。如果没有编排层，就只能写一堆强耦合接口，后面很难扩展，也很难治理。”

## 面试时容易踩的坑

### 不要上来就说“我做了 MCP Server”

除非你真的能讲清楚：

- 协议
- transport
- tool schema
- resource
- client / server 关系

否则建议统一说成：

“类 MCP 的技能化抽象和工作流编排思路。”

### 不要把 skills 讲成“几个方法封装”

一定要强调：

- 输入输出协议
- 业务边界
- 可组合性
- 可替换性
- 编排关系

### 不要把 Agent 讲成万能自动体

更稳的说法是：

“我做的是面向具体业务流程的 Agent 式编排，不是一个无边界的通用智能体。”
