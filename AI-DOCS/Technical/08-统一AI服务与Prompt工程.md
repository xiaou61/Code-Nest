# 亮点八：统一 AI 服务与 Prompt 工程

## 这条亮点的核心价值

这条亮点要讲出来的是：

1. 你做的不是简单“接了个大模型接口”，而是把 AI 能力做成了统一服务层。
2. 你理解 Prompt 工程的核心不是写提示词，而是控制输入上下文、输出结构和稳定性。
3. 你能把 AI 能力接到真实业务闭环里，而不是只做演示型问答。

## 30 秒答法

我在项目里搭建过统一 AI 服务层，把 AI 模拟面试、求职作战台和 SQL 优化工作台这些场景都收敛到同一套调用框架里。具体上，我做了工作流注册、Prompt 模板设计、上下文参数编排、结构化输出约束和结果解析兜底，让 AI 能力从零散接口升级成了可复用、可扩展、可治理的业务服务。

## 1 分钟展开答法

这个模块的难点不在于“把 Coze 调起来”，而在于怎么让大模型输出真正稳定可用。我的思路是把 AI 能力统一抽象成服务层：第一层是能力注册层，把出题、评价、总结、JD 解析、简历匹配、SQL 诊断这类场景统一映射成可管理的工作流；第二层是参数编排层，根据业务场景动态注入方向、难度、风格、问答记录、JD 文本、简历内容、SQL 执行计划等上下文；第三层是 Prompt 工程层，用固定模板约束模型角色、评分规则、字段结构和输出边界；第四层是结果治理层，对模型结果做结构化解析、异常识别和 fallback 兜底，保证 AI 能力可以稳定服务线上业务。

## 你在项目里具体做了什么

### 1. 为什么要做统一 AI 服务层

- 如果每个业务模块各自直连 AI，后续会出现调用方式不统一、错误处理不一致、输出格式难对齐的问题。
- 所以我把 AI 调用统一收口，抽象出工作流枚举、统一入参和统一解析逻辑。
- 这样后面新增业务场景时，不需要重新造调用轮子，只需要扩一个 workflow 和对应 DTO 就行。

一句话：

“我做的不是一个 AI 功能点，而是一层 AI 能力接入中台。”

### 2. Prompt 工程到底做了什么

很多人说 Prompt 优化，容易讲虚。你要把它讲具体：

- 我会先把业务目标拆清楚，比如是出题、评价、总结、匹配还是诊断。
- 再把关键上下文组织进去，比如面试方向、难度、风格、追问轮次、问答记录、JD 结构、简历文本、SQL 和 EXPLAIN。
- 然后在 Prompt 里明确角色设定、打分规则、输出字段和禁止事项，尽量减少模型自由发挥。
- 目标不是让模型“写得更好看”，而是让它“更稳定、更可解析、更贴合业务规则”。

你可以这样说：

“我做 Prompt 工程时，核心关注的是上下文完整性、字段约束和结果稳定性，而不是单纯调措辞。”

### 3. 为什么要强约束结构化输出

- 大模型如果只返回一段自然语言，业务侧几乎没法直接消费。
- 所以我要求不同场景尽量返回固定 JSON 结构，比如评分、优缺点、追问、建议、索引 DDL、收益对比、行动计划等字段。
- 这样前端展示、数据库落库、历史追溯和多阶段编排都会更容易。

这个点一定要强调：

“Prompt 只是入口，真正让 AI 能落地的是结构化输出协议。”

### 4. 结果解析和 fallback 为什么重要

- AI 能力在线上最怕的不是“不够聪明”，而是“不稳定”。
- 所以我在解析层做了几件事：
  - 识别错误响应
  - 兼容 output 包装
  - 兼容代码块和杂质文本
  - 解析失败时走本地 fallback
- 这样即使工作流波动，系统也不会整个失效。

更职业一点的说法：

“我把 AI 集成当成不稳定外部依赖来治理，而不是当成一定正确的黑盒。”

### 5. 统一 AI 服务层支撑了哪些场景

- AI 模拟面试：出题、答案评价、面试总结、追问建议。
- 求职作战台：JD 解析、简历匹配、补短板计划、面试复盘。
- SQL 优化工作台：诊断、重写建议、收益对比和案例沉淀。

这说明你讲的是“多场景能力平台”，不是单一业务点。

## 高频追问与参考回答

### Q1：你说的 Prompt 优化，具体优化了什么？

答法：

“我主要优化三类东西：一是输入上下文，把业务所需信息尽量补全；二是输出结构，要求模型严格按 JSON 字段返回；三是规则约束，比如评分标准、追问条件和建议边界，减少模型自由发挥导致的不稳定结果。”

### Q2：为什么不让每个业务模块自己调 AI？

答法：

“因为那样会导致调用方式、参数协议、异常处理和输出格式都不一致，后面维护成本很高。统一收口后，新增能力和治理能力都会更容易复用。”

### Q3：AI 返回的东西不稳定怎么处理？

答法：

“我会把它当成不稳定依赖来治理，通过结构化输出约束、统一解析、异常识别和 fallback 机制，让系统至少能返回基础结果，而不是直接挂掉。”

### Q4：为什么要强调结构化输出？

答法：

“因为业务系统不只是给用户看一段文字，还要支持前端渲染、历史记录、二次分析和多阶段流程衔接，所以结构化输出是 AI 工程化落地的关键。”

## 面试时容易踩的坑

### 不要把 Prompt 工程讲成“调提示词话术”

更好的表达是：

- 输入上下文设计
- 业务规则约束
- 输出协议设计
- 结果稳定性治理

### 不要把统一 AI 服务讲成“封了个工具类”

一定要强调：

- 多场景复用
- 统一治理
- 统一解析
- 统一兜底

### 不要承诺模型结果百分百准确

更稳的说法是：

“我的目标不是让模型永远正确，而是让结果足够稳定、可消费、可回退。”

## 一句话背诵版

我在项目里做的不是单点 AI 接口，而是统一 AI 服务层和 Prompt 工程体系，通过工作流注册、上下文编排、结构化输出约束、结果解析和 fallback 兜底，把 AI 模拟面试、求职作战台和 SQL 优化工作台这些能力做成了可复用、可治理的业务服务。
