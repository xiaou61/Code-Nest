# 亮点五：OJ 沙箱评测体系

## 这条亮点的核心价值

这条亮点非常适合打技术深度，因为它天然包含：

1. 代码执行安全
2. 多语言编译运行抽象
3. 长耗时任务异步化
4. 竞赛规则计算

也就是说，这条亮点可以同时覆盖“系统设计 + 安全隔离 + 业务规则”。

## 30 秒答法

我在项目里做过一个基于 go-judge 的多语言 OJ 沙箱评测体系，支持 Java、C/C++、Go、Python、JavaScript 等语言的编译与运行。架构上通过策略模式抽象不同语言的源码文件、编译命令和运行命令，并对编译错误、运行错误、超时、超内存等状态统一判定；同时把正式判题链路接入 RabbitMQ 异步处理，编译产物支持缓存复用，赛事模式下还补充了报名校验、时间窗控制和 ACM 榜单规则计算，形成了完整在线评测闭环。

## 1 分钟展开答法

做 OJ 的难点不只是“执行代码”，而是如何安全、统一、可扩展地执行代码。我的实现思路是分三层。第一层是沙箱层，通过 go-judge 把代码编译和运行放到隔离环境中执行，并限制 CPU 时间、内存和进程数。第二层是语言抽象层，用策略模式封装不同语言的源码文件名、编译命令、运行命令和编译产物，从而支持多语言扩展。第三层是调度和业务层，自由运行、自测和正式提交三条链路共用底层沙箱能力，正式判题通过 MQ 异步消费，赛事模式下再加上报名校验、题目归属校验和 ACM 榜单计算。

## 你在项目里具体做了什么

### 1. 为什么要用沙箱

- 用户提交的是任意代码，不能直接在业务进程里执行。
- 这类代码可能死循环、爆内存、起过多进程，甚至尝试访问系统资源。
- 所以必须把执行环境隔离出去，并对资源做硬限制。

你可以这样讲：

“我不是把代码执行当成功能问题，而是优先当成安全和资源治理问题。”

### 2. go-judge 在这里解决了什么

- 提供隔离执行环境
- 支持资源限制，如 CPU 时间、内存、进程数
- 支持标准输入输出收集
- 支持编译产物缓存、文件拷入拷出

这句很关键：

“业务服务不直接执行用户代码，而是通过 REST 客户端调用 go-judge，由沙箱负责真正的编译和运行。”

### 3. 多语言是怎么抽象的

- 不同语言的差异主要在：
  - 源文件名不同
  - 编译命令不同
  - 运行命令不同
  - 是否需要编译不同
- 所以我抽了 `JudgeStrategy` 接口，每种语言一个实现类。
- 主流程只关心“拿到策略 -> 编译 -> 运行 -> 汇总结果”，不用关心语言细节。

你可以把这个点讲成：

“我把语言差异收敛到了策略层，使评测主流程保持稳定，多语言扩展只影响局部实现。”

### 4. 编译产物缓存复用是怎么考虑的

- 正式判题一个提交通常要跑多个测试用例。
- 如果每个测试用例都重新编译一遍，性能会很差。
- 所以我在编译阶段把产物缓存下来，再在多测试用例之间复用。
- 运行结束后再统一清理缓存文件，避免长期堆积。

这个点很值钱，因为它说明你考虑过实际运行成本。

### 5. 为什么正式判题要异步化

- 判题是典型长耗时任务，不适合占用接口线程。
- 用户提交代码后，不需要立刻阻塞等待最终结果。
- 所以我把提交和判题拆开：
  - 提交接口先落提交记录
  - 再投递 RabbitMQ 消息
  - 消费者异步执行判题

你可以总结成：

“提交链路负责记录和入队，判题链路负责异步消费，这样高峰期提交接口也能快速返回。”

### 6. 状态判定怎么做

统一判定的状态包括：

- compile_error
- runtime_error
- time_limit_exceeded
- memory_limit_exceeded
- wrong_answer
- accepted
- system_error

你要强调：

“我做的是标准化评测状态流转，而不是简单返回一段错误日志。”

### 7. 赛事模式补了哪些规则

- 报名资格校验
- 题目是否属于当前赛事校验
- 提交时间窗校验
- ACM 榜单计算
- 错误提交罚时规则

这里面最值得讲的是 ACM 榜单：

- 先按提交时间排序
- 同一题只统计首次 AC
- AC 前的错误提交按固定分钟数累计罚时
- 最后按 solved 数、penalty、最后 AC 时间排序

这段非常适合展示你对“复杂业务规则落地”的理解。

## 高频追问与参考回答

### Q1：为什么不用本地执行，非要引入 go-judge？

答法：

“因为用户代码执行的风险太高，本地执行会把安全、资源隔离和稳定性问题都暴露给业务进程；引入沙箱是为了把执行风险和业务服务解耦。”

### Q2：多语言扩展怎么做？

答法：

“我把语言差异抽成策略接口，新增语言时主要补充文件名、编译参数和运行参数，主判题流程基本不动。”

### Q3：为什么要缓存编译产物？

答法：

“因为一个提交通常会跑多个测试用例，编译是重复成本，把编译结果缓存起来后可以在多个用例间复用，明显减少总评测耗时。”

### Q4：为什么正式判题走 MQ，自测却同步返回？

答法：

“因为两个场景目标不同。自测更强调即时反馈，正式提交更强调吞吐和稳定性，所以前者同步、后者异步是更合理的拆分。”

## 面试时容易踩的坑

### 不要只讲“我用了 go-judge”

一定要继续往下讲：

- 为什么要沙箱
- 多语言怎么抽象
- 状态怎么统一
- 为什么判题要异步化

### 不要忽略安全问题

OJ 最大的技术价值之一就是安全隔离和资源控制，不讲这个会很亏。

### 不要把赛事规则一笔带过

如果你能顺手讲清楚 ACM 榜单逻辑，面试官会觉得你做的不只是底层技术，还有复杂业务规则。

## 一句话背诵版

我做的 OJ 不是简单代码运行，而是一套完整评测体系：底层用 go-judge 做隔离执行和资源限制，中间用策略模式抽象多语言编译运行，正式提交通知走 MQ 异步判题，上层再叠加自测链路和 ACM 赛事规则，最终形成可扩展的在线评测闭环。
