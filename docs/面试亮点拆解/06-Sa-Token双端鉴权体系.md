# 亮点六：Sa-Token 双端鉴权体系

## 这条亮点的核心价值

这条亮点不应该被讲成“我把 JWT 换成了 Sa-Token”，而是要讲成：

1. 我重构了项目的认证鉴权基座。
2. 我解决了管理员端和用户端并存时的登录域隔离问题。
3. 我把 HTTP 鉴权和 WebSocket 鉴权打通了，并完成了平滑迁移。

## 30 秒答法

我在项目里主导了 JWT 到 Sa-Token 的认证体系迁移，核心是基于独立 `StpLogic` 把管理员和用户拆成两个登录域，分别做登录、会话、角色权限和路由鉴权隔离。除了 HTTP 接口，我还把鉴权能力延伸到 WebSocket 握手阶段，并且让 Sa-Token 使用独立 Redis 连接池存储会话数据，避免影响现有业务 Redisson 配置，最终完成了双端账号体系的平滑迁移。

## 1 分钟展开答法

这个改造的真正难点，不是“接一个新框架”，而是如何在不破坏现有业务的前提下，把原来的认证方式统一迁到更稳定、可维护的体系。因为项目里同时存在管理员端和用户端，两类账号不能共用一套登录上下文，否则会出现 token 混用、权限串域等问题。所以我做法上是基于 Sa-Token 抽了两套独立的 `StpLogic`，分别用 `admin` 和 `user` 作为 loginType，管理员和用户各走自己的登录、登出、权限和 session 体系；然后在路由拦截层按 `/auth/**` 和 `/user/**` 做差异化鉴权；最后再把这套鉴权能力扩展到 WebSocket 握手阶段，实现长连接场景下的统一身份校验。

## 你在项目里具体做了什么

### 1. 为什么要从 JWT 迁移到 Sa-Token

你可以这样讲：

- JWT 适合轻量认证，但项目做大之后，登录态控制、踢人下线、会话存储、角色权限和多端并发管理会越来越复杂。
- Sa-Token 更像一套完整的认证框架，登录、session、权限、路由拦截这些能力都有现成支撑。
- 所以迁移的目标不是“换库”，而是“把登录态治理做成统一基础设施”。

### 2. 双端账号体系为什么要拆 loginType

这是这条亮点最值钱的地方。

- 项目里既有管理员端，也有普通用户端。
- 两类账号的权限模型、接口路径和业务边界都不一样。
- 如果共用一套登录上下文，后续权限校验和 session 数据会很混乱。
- 所以我分别定义了：
  - `StpAdminUtil`，底层 `new StpLogic("admin")`
  - `StpUserUtil`，底层 `new StpLogic("user")`
- 这样管理员和用户的 token、session、角色权限天然隔离。

一句话讲法：

“我不是简单分两个 controller，而是在认证基座层做了双域隔离。”

### 3. HTTP 路由鉴权怎么做

- 我在 `SaTokenConfig` 里注册统一拦截器。
- 对 `/auth/**` 走管理员登录校验。
- 对 `/user/**` 走用户登录校验。
- 同时给登录、注册、刷新 token、验证码、Swagger 等接口做白名单放行。

面试里这样讲比较好：

“路由层拦截是第一层，角色权限和业务注解是第二层，我把认证和授权分层处理，而不是在 controller 里到处手写判断。”

### 4. 权限和角色怎么接

- Sa-Token 提供权限接口扩展能力。
- 我通过 `StpInterfaceImpl` 按 `loginType` 区分管理员和用户的权限来源。
- 管理员端可以走菜单、角色、权限树体系。
- 用户端则按业务需要提供基础角色能力。

这个点可以说明你理解 RBAC，而不是只会校验登录。

### 5. 会话为什么要放 Redis

- 登录态不能只放内存，否则服务重启就丢。
- 我把 Sa-Token 的 token/session 数据放到 Redis。
- 而且项目里原本已经用了 Redisson，所以我特意让 Sa-Token 使用独立 Redis 连接池，避免和业务 Redis 配置互相干扰。

这个说法很加分：

“我不仅考虑功能接入，还考虑了基础组件之间的资源隔离，避免认证流量和业务流量互相影响。”

### 6. 怎么打通 WebSocket 鉴权

- 这是很多项目容易漏掉的点。
- HTTP 接口鉴权做完，不代表 WebSocket 天然安全。
- 我在 WebSocket 握手拦截器里读取 token，再调用 Sa-Token 的能力反查登录用户。
- 握手成功后，把用户身份放到连接上下文，后续长连接消息统一用服务端绑定身份处理。

你可以补一句：

“这样做的好处是，HTTP 和 WebSocket 共用同一套登录态来源，权限体系不会分裂。”

### 7. 为什么说这是平滑迁移

因为你不是推翻重做，而是逐步替换：

- 先接入 Sa-Token 基础能力
- 再替换登录、登出和刷新逻辑
- 再替换 controller / service 里读取用户信息的方式
- 最后把 WebSocket、权限切面、异常处理统一纳入 Sa-Token 体系

## 高频追问与参考回答

### Q1：为什么不能让 admin 和 user 共用一套 token 体系？

答法：

“因为两类账号的权限边界完全不同，共用一套上下文后很容易出现角色串域、会话污染和鉴权逻辑混乱，所以我在底层就用不同 loginType 做隔离。”

### Q2：Sa-Token 和 JWT 的核心区别是什么？

答法：

“JWT 更偏 token 本身，Sa-Token 更偏完整登录态治理，除了 token 还有 session、踢下线、权限拦截、多账号体系这些能力。我的场景更适合后者。”

### Q3：为什么 WebSocket 还要单独鉴权？

答法：

“因为 WebSocket 一旦建连，就是长连接通道，如果握手阶段不校验身份，后续消息就会失去可信用户上下文。”

### Q4：Redis 在这里主要存什么？

答法：

“主要是 token 和 session 数据。这样服务重启后登录态不会丢，而且也便于多实例共享会话。”

## 面试时容易踩的坑

### 不要把这条亮点说成纯框架替换

一定要强调：

- 双端登录域隔离
- 路由鉴权收口
- 权限接口适配
- WebSocket 握手打通

### 不要忽略迁移成本

如果你讲“平滑迁移”，就要说明不是一次性重写，而是逐步替换登录和上下文读取方式。

### 不要说“用了 Sa-Token 所以就安全了”

更稳的说法是：

“Sa-Token 提供了统一认证鉴权基础能力，但真正的安全性还取决于登录域隔离、权限边界设计和具体场景接入是否完整。”

## 一句话背诵版

我做的不是单纯换认证框架，而是基于 Sa-Token 重构了一套双端鉴权基座：通过独立 `StpLogic` 隔离管理员和用户登录域，统一接管 HTTP 路由、角色权限、会话存储和 WebSocket 握手鉴权，并把登录态持久化到 Redis，完成了 JWT 到统一认证体系的平滑迁移。
