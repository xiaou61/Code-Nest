## 2. 背景与目标

### 2.1 背景

现有 OJ 已具备题目管理、提交判题、排行榜等基础能力，但缺乏“赛事”层面的组织能力，无法支持周赛/挑战赛等运营场景。

### 2.2 目标

在不重写现有判题链路的前提下，新增完整赛事闭环能力：

1. 管理端可创建赛事、编排题目、管理状态。
2. 用户端可查看赛事、报名、参赛、查看榜单。
3. 提交链路支持赛事上下文并进行参赛资格校验。
4. 提供 ACM 规则榜单计算能力。

### 2.3 非目标（本期不做）

1. Rating/ELO 积分系统。
2. 封榜/揭榜高级机制。
3. 队伍赛与组队功能。

## 3. 角色与核心流程

### 3.1 角色

1. 管理员：创建与运营赛事。
2. 普通用户：报名并参加赛事。
3. 判题服务：复用现有提交评测能力。

### 3.2 核心流程

1. 管理员创建赛事并编排题目。
2. 管理员发布赛事（进入可报名/可查看状态）。
3. 用户浏览赛事详情并报名。
4. 赛事进行期间，用户提交赛事题目代码。
5. 系统校验参赛资格并执行判题。
6. 榜单基于提交记录实时计算并展示。

## 4. 功能需求

### 4.1 管理端

#### 4.1.1 赛事列表

1. 支持分页查询。
2. 支持按关键词/类型/状态筛选。
3. 支持状态变更（发布/下线）。
4. 支持删除与跳转编辑。

#### 4.1.2 赛事编辑

1. 支持填写赛事基础信息（标题、类型、时间窗口、描述）。
2. 支持题目编排（多选题目）。
3. 支持新增赛事与编辑赛事。
4. 保存时校验时间合法性与题目必选规则。

### 4.2 用户端

1. 赛事中心列表页。
2. 赛事详情页（基础信息、题目、榜单入口）。
3. 报名动作与状态反馈。
4. 榜单展示与排序规则说明。

### 4.3 提交校验

当提交请求携带 `contestId` 时，需按顺序校验：

1. 赛事存在且状态允许提交。
2. 当前时间在赛事提交窗口内。
3. 用户已报名该赛事。
4. 提交题目属于该赛事编排题目。

### 4.4 榜单规则（ACM）

1. 排序主键：`解题数 DESC`。
2. 排序次键：`总罚时 ASC`。
3. 排序三键：`最后一次 AC 时间 ASC`。
4. 罚时规则：每题仅计首次 AC，AC 前每次 `wrong_answer` 罚 `20` 分钟。

## 5. 数据模型（MVP）

### 5.1 `oj_contest`

核心字段建议：

1. `id`
2. `title`
3. `contest_type`（`weekly` / `challenge`）
4. `status`（草稿、即将开始、进行中、已结束）
5. `start_time`
6. `end_time`
7. `description`
8. `created_by`
9. `create_time`
10. `update_time`

### 5.2 `oj_contest_problem`

1. `id`
2. `contest_id`
3. `problem_id`
4. `sort_order`

### 5.3 `oj_contest_participant`

1. `id`
2. `contest_id`
3. `user_id`
4. `join_time`
5. `status`

### 5.4 `oj_submission` 扩展

新增字段：

1. `contest_id`（可空，用于区分普通提交和赛事提交）

## 6. 接口规划（MVP）

1. `POST /admin/oj/contests/list`

5. `DELETE /admin/oj/contests/{id}`

### 6.2 用户端接口

## 7. 权限与安全

1. 管理端接口要求管理员权限。
2. 用户端报名/提交要求登录态。
3. 提交接口需二次校验参赛资格，防止绕过前端调用。

## 8. 验收标准

### 8.1 功能验收

1. 管理端可完成赛事新增、编辑、发布、删除。
2. 用户端可查看并报名赛事。
3. 非报名用户提交赛事题目会被拦截。
4. 榜单排序符合 ACM 规则。

### 8.2 技术验收

1. 后端相关模块构建通过。
2. 用户端构建通过。
3. 管理端构建通过。
4. 关键链路有可复现的回归步骤。

## 9. 里程碑计划

### 9.1 Phase 1（MVP）

1. 规则与数据结构落地。
2. 赛事管理服务与 API。
3. 用户端赛事中心与榜单页面。

### 9.2 Phase 2（增强）

## 10. 风险与应对

1. 风险：赛事状态流转复杂，容易出现边界错误。  
   应对：统一状态机校验函数，关键路径补单测。

2. 风险：榜单实时计算压力随提交量上升。  
   应对：先用内存计算 + 分页，后续引入缓存与增量刷新。

3. 风险：提交链路改动可能影响普通题目提交流程。  
   应对：`contestId` 设计为可选，确保普通提交路径不变。
