# 端到端业务链路图

这页用“链路图”的方式解释 Code Nest 的核心功能如何互相连接。它不替代 [核心链路教程](/manuals/core-workflows)，而是帮你先看清全局：一个用户从注册登录开始，如何进入学习、OJ、AI、社区、聊天、积分、文件和后台运营闭环。

读这页时不要急着背接口。先看每条链路的入口、关键能力、写入的数据和验证方式。真正要操作时，再回到模块页和核心链路教程。

## 全局主链路

```text
用户注册/登录
  -> 用户资料和登录态
  -> 成长学习 / 内容创作 / 实时互动 / 工具使用
  -> 文件、敏感词、通知、积分、AI Runtime 等平台能力
  -> 管理端运营审核、配置和日志
  -> 验证记录、版本发布和问题定位
```

这条主链路里有三类能力：

| 类型 | 代表模块 | 作用 |
| --- | --- | --- |
| 用户业务 | 面试题、OJ、模拟面试、学习资产、社区、博客、动态、简历、聊天室 | 用户每天直接使用的功能 |
| 平台能力 | 鉴权、用户、文件、通知、积分、敏感词、AI Runtime、WebSocket、日志 | 被多个业务复用的横切能力 |
| 运营闭环 | 管理端审核、配置、统计、日志、验证记录、版本历史 | 让功能可维护、可排障、可发布 |

一个功能如果只做了用户页面，没有接入平台能力和运营闭环，通常还不能算完整。

## 账号与身份链路

```text
/register 或 /login
  -> /user/auth/register 或 /user/auth/login
  -> Sa-Token 用户端登录态
  -> /profile 当前用户资料
  -> 业务接口读取当前 userId
```

关键点：

| 环节 | 要确认什么 | 先看 |
| --- | --- | --- |
| 验证码 | 注册登录页能加载验证码，后端能校验 | [鉴权与用户体系](/modules/auth) |
| 登录态 | 用户端和管理端是两套登录域 | [权限与安全边界](/guide/security-boundaries) |
| 当前用户 | 后端从登录态取用户 ID，不信任前端传参 | [用户账户与个人中心](/modules/user-account) |
| 验证 | 登录、资料读取、资料更新、登录失效提示 | [核心链路教程](/manuals/core-workflows) |

判断一个用户端接口是否安全时，先问三件事：

1. 是否需要登录。
2. 是否从当前会话取 `userId`。
3. 是否会越权读写别人的数据。

## 学习成长链路

```text
面试题学习
  -> 掌握度和复习记录
  -> 模拟面试或求职作战台
  -> AI 反馈和报告
  -> 学习资产沉淀
  -> 管理端审核
  -> 通知和统计回流
```

主要模块关系：

| 步骤 | 用户动作 | 平台能力 | 管理端或数据回流 |
| --- | --- | --- | --- |
| 刷题 | 学题单、标记掌握度、收藏 | 用户登录、学习记录 | 题库分类、题单、题目管理 |
| 复习 | 查看待复习题目 | 掌握度历史 | 学习统计 |
| 模拟面试 | 创建会话、作答、看报告 | AI Runtime、Prompt、Schema | 会话、方向、AI 治理 |
| 转资产 | 把面试总结沉淀为学习资产 | AI 总结、通知 | 学习资产审核、合并、驳回 |

相关文档：

| 想理解 | 先看 |
| --- | --- |
| 面试题和学习记录 | [面试题库](/modules/interview)、[题库与成长闭环](/modules/interview-and-growth) |
| AI 面试和求职作战台 | [模拟面试与求职作战台](/modules/mock-interview-job-battle) |
| 学习资产审核 | [学习资产](/modules/learning-assets) |
| AI 治理和回归 | [AI Runtime](/modules/ai-runtime)、[AI Schema 与治理](/reference/ai-schemas) |

验证时不要只看报告页是否显示。完整验证应该包含：会话创建、AI 反馈、报告生成、学习资产生成、后台审核和通知回流。

## OJ 判题链路

```text
/oj/problem/:id
  -> 运行 / 测试 / 提交
  -> 后端生成 submission
  -> go-judge 执行代码
  -> 判题结果回写
  -> 我的提交、提交详情、排行榜或积分回流
```

关键节点：

| 节点 | 常见风险 | 验证方式 |
| --- | --- | --- |
| 题面与用例 | 题目、测试用例、输入输出不一致 | 从列表进入题目详情，运行示例 |
| 提交创建 | 未登录、语言不支持、代码为空 | 提交接口返回和提交记录 |
| go-judge | 服务不可达、沙箱超时、资源限制 | 确认 go-judge 可用，跑 AC/WA/CE 样例 |
| 结果回写 | 状态卡住、详情丢失、排行不更新 | 我的提交、提交详情、赛事榜单 |
| 积分奖励 | 首次 AC 判断错误、重复奖励 | 积分流水和余额 |

相关文档：[OJ 判题系统](/modules/oj)、[测试与回归](/guide/testing-regression)、[验证记录与已知问题](/manuals/verified-scenarios)。

## 内容发布链路

```text
用户发布内容
  -> 登录态和权限校验
  -> 敏感词检查
  -> 文件或图片上传
  -> 内容入库
  -> 评论、点赞、收藏、通知
  -> 管理端审核、统计或下架
```

内容类模块很多，但检查方法一致：

| 模块 | 用户入口 | 重点平台能力 | 管理端验证 |
| --- | --- | --- | --- |
| 社区 | 发帖、评论、收藏、个人主页 | 敏感词、通知、AI 摘要 | 帖子、评论、用户状态 |
| 动态 | 发布、点赞、收藏、评论 | 敏感词、文件、通知 | 动态列表、评论、统计 |
| 博客 | 开通博客、写文章、公开博客页 | Markdown 渲染、文件、敏感词 | 文章、分类、标签 |
| CodePen | 发布作品、查看详情 | 代码展示、点赞、收藏 | 作品、模板、标签、统计 |
| 简历 | 编辑、保存、导出 | 文件、模板、分析报告 | 模板、分析、报告 |

内容链路的完整验证是“三段式”：

1. 用户端写入成功。
2. 用户端刷新或重新进入后仍能看到。
3. 管理端能检索、审核、统计或处理。

如果内容会渲染 Markdown、代码、Diff、HTML 片段或 AI 命中内容，一定要额外检查 [前端渲染安全](/reference/frontend-rendering-security)。

## 文件、通知、积分三条横切链路

很多业务问题最后会落到这三条链路上。

### 文件链路

```text
上传组件
  -> /file 上传
  -> 存储策略和 file_record
  -> 业务表保存 fileId 或 URL
  -> /api/files 公开读取或登录读取
```

检查重点：上传是否登录、文件是否公开、URL 前缀是否正确、业务数据是否保存引用。先看 [文件存储](/modules/file-storage)。

### 通知链路

```text
业务事件
  -> 通知模板或通知创建
  -> 用户通知列表
  -> 已读/未读状态
  -> 管理端公告或消息运营
```

检查重点：事件是否触发、接收人是否正确、通知是否可读、已读状态是否回写。先看 [通知中心](/modules/notification)。

### 积分链路

```text
签到 / OJ 首次 AC / 后台发放 / 抽奖消费
  -> 积分规则
  -> 余额变更
  -> 积分流水
  -> 用户端和管理端回流
```

检查重点：是否重复发放、扣减是否幂等、余额和流水是否一致。先看 [积分与抽奖](/modules/points)。

## 实时聊天链路

```text
/chat
  -> /user/chat/ws-ticket
  -> /ws/chat?ticket=...
  -> PING/PONG 保活
  -> MESSAGE / IMAGE / SYSTEM / TYPING
  -> ACK、限流、撤回、禁言、踢出
  -> 管理端消息和在线用户运营
```

聊天链路的验证要同时看 HTTP 和 WebSocket：

| 环节 | 验证点 |
| --- | --- |
| ticket | 先获取一次性 ws-ticket，不能继续使用旧 token 握手 |
| 连接 | Origin、Nginx 转发、WebSocket 地址正确 |
| 发送 | 消息入库，客户端收到 `MESSAGE_ACK` |
| 失败态 | 限流、敏感内容、禁言、踢出都能反馈到页面 |
| 管理端 | 消息检索、公告、禁言、踢人和撤回可用 |

协议细节见 [WebSocket 协议](/reference/websocket)，排障见 [问题定位流程](/operations/diagnosis-flow)。

## AI Runtime 链路

```text
业务场景
  -> Prompt 和 Schema
  -> 模型调用
  -> RAG 或 Graph 增强
  -> 结构化输出
  -> 业务落库或页面展示
  -> 回归样例和治理指标
```

AI 链路不能只看“接口返回成功”。要同时检查：

| 检查项 | 为什么 |
| --- | --- |
| Prompt | 场景意图、变量和边界条件决定输出质量 |
| Schema | 前端和后端依赖结构化字段，字段错了页面也会错 |
| RAG/Graph | sidecar 未启动时结果可能退化 |
| 失败态 | 超时、空结果、模型错误要有可解释反馈 |
| 回归 | Prompt 改动可能影响多个场景 |
| 治理指标 | 质量、错误率、耗时要能持续观察 |

相关文档：[AI Runtime](/modules/ai-runtime)、[AI Schema 与治理](/reference/ai-schemas)、[测试与回归](/guide/testing-regression)。

## 管理端运营链路

```text
业务数据产生
  -> 管理端列表或详情
  -> 审核 / 上下架 / 配置 / 统计
  -> 操作日志或状态回写
  -> 用户端可见结果变化
```

管理端不是“附属页面”，它决定功能是否可运营。检查一个业务模块时，至少确认：

| 问题 | 示例 |
| --- | --- |
| 管理端能看到数据吗 | 帖子、动态、学习资产、聊天消息、积分流水 |
| 管理端能处理状态吗 | 审核、驳回、下架、禁言、配置 |
| 操作后用户端会变吗 | 用户刷新页面后状态、通知或权限变化 |
| 高风险操作有权限吗 | 管理端登录域、管理员权限和操作日志 |

相关文档：[系统运营后台](/modules/system-ops)、[管理端操作手册](/manuals/admin-operations)、[仪表盘与日志](/modules/dashboard-logs)。

## 回归验证链路

```text
变更范围判断
  -> 选择构建和测试命令
  -> 跑核心烟测
  -> 记录已验证和未验证
  -> 更新验证记录或版本历史
```

推荐按这个顺序收口：

1. 先在 [功能开发流程](/guide/feature-development) 判断改动影响面。
2. 再用 [权限与安全边界](/guide/security-boundaries) 检查高风险边界。
3. 用 [测试与回归](/guide/testing-regression) 选择命令和烟测。
4. 发布前按 [发布前验证](/guide/release-verification) 做最终检查。
5. 真实结果写入 [验证记录与已知问题](/manuals/verified-scenarios)。

## 读图后的自测

| 自测问题 | 应该能回答 |
| --- | --- |
| 一个用户端写操作怎样才算完整 | 接口成功、页面回流、数据可查、必要时管理端可见 |
| 一个 AI 功能怎样才算完整 | Prompt、Schema、调用、失败态、回归样例和治理指标都能解释 |
| 一个 OJ 提交怎样才算完整 | 题目、提交、go-judge、结果回写、记录和积分/榜单都验证 |
| 一个内容发布怎样才算完整 | 敏感词、文件、渲染安全、互动、通知和后台运营都考虑 |
| 一个发布怎样才算可交接 | 已验证、未验证、原因、风险和后续补测都写清楚 |
