# 异常路径与失败态索引

这页是给“已经知道出问题了，但还不知道先查哪一层”的人准备的。

它和几页相邻文档的区别是：

- [响应体与错误码](/reference/response-errors)：告诉你返回码和异常类型是什么。
- [问题定位流程](/operations/diagnosis-flow)：告诉你排查顺序怎么走。
- [模块最小回归矩阵](/reference/module-regression-matrix)：告诉你改完模块最低要补哪些验证。
- 本页：告诉你**常见失败态长什么样、第一证据在哪、最可能卡在哪一层**。

## 怎么用这页

建议按这个顺序用：

1. 先看你遇到的是哪种表现：401、403、空数据、文件打不开、WebSocket 断、AI 结构坏、OJ 卡住、审核流不推进。
2. 找到对应行，先看“第一证据”列，不要一上来就猜原因。
3. 再看“最常见根因”，缩小到登录态、权限、参数、状态机、外部依赖还是部署层。
4. 最后顺着“继续看”跳回对应模块页或操作页。

如果一个问题同时满足两类表现，优先选**最早出现失败的那一层**。

## 最常见的 10 类失败态

| 失败类型 | 典型表现 | 第一证据 | 最常见根因 | 继续看 |
| --- | --- | --- | --- | --- |
| 登录态失效 | `701`、`702`、跳登录页、刷新后掉线 | 浏览器 Network、LocalStorage/Cookie、响应体 `code` | 用户端/管理端 Token 混用、Token 过期、请求没带 `Authorization` | [鉴权与用户体系](/modules/auth)、[响应体与错误码](/reference/response-errors) |
| 权限或角色不足 | `703`、后台按钮能看但点了失败、普通账号访问后台报错 | 响应体 `code`、Controller 是否有 `@RequireAdmin` | 只做了前端按钮控制、后端方法漏注解、当前不是 admin 登录域 | [权限注解与角色边界索引](/reference/permission-boundaries)、[权限与安全边界](/guide/security-boundaries) |
| owner 归属不对 | 已登录但看不到自己的数据，或能误操作别人的数据 | Service 查询条件、当前用户 ID、表里 owner 字段 | 后端信任前端传 `userId`、缺少 `selectByIdAndUserId`、只校验登录没校验归属 | [权限注解与角色边界索引](/reference/permission-boundaries)、对应模块页 |
| 参数校验失败 | `400`、`601`、表单提交直接报参数错误 | Response `message`、前端请求体、DTO 字段 | DTO 必填字段没传、类型不匹配、数组/枚举结构变了 | [响应体与错误码](/reference/response-errors)、[API 调用示例](/reference/api-examples) |
| 文件链路失败 | 上传成功但 URL 打不开、`403`、`404`、下载失败 | `/file/**` 接口响应、文件表、物理文件或 URL | `is_public` 不对、文件记录失效、Nginx `/api/files/**` 代理缺失、业务归属没放行 | [文件存储](/modules/file-storage)、[权限注解与角色边界索引](/reference/permission-boundaries) |
| WebSocket 失败 | ws-ticket 获取失败、握手失败、消息没 ACK、乐观消息变红 | ws-ticket 接口、浏览器 WS 面板、`ERROR` 事件 data | ticket 失效、Origin 问题、未登录、限流、禁言、tempId 对不上 | [WebSocket 协议](/reference/websocket)、[IM 聊天室](/modules/chat) |
| AI 结构化输出失败 | 接口是 `200` 但字段缺失、页面空、管理端调试结果形状不对 | Prompt 调试、Schema 校验、回归结果 | Prompt 变量缺失、Schema 变更、模型输出漂移、RAG 空召回 | [AI Runtime](/modules/ai-runtime)、[AI Schema 与治理](/reference/ai-schemas) |
| OJ 判题失败 | 提交一直 `pending/judging`、`system_error`、编译错误异常 | 提交状态、go-judge 健康、测试用例、日志 | go-judge 不可达、编译器异常、赛事约束没过、测试用例问题 | [OJ 判题系统](/modules/oj)、[模块状态机与生命周期索引](/reference/module-state-machines) |
| 状态机或审核流卡住 | 状态不推进、推进后列表不回流、审核结果不一致 | 主表状态、日志表、阶段日志、发布日志 | 只改主表没改日志、条件分支漏写、幂等规则触发、非法回退 | [模块状态机与生命周期索引](/reference/module-state-machines)、对应模块页 |
| 外部依赖或部署失败 | 所有接口慢、CORS、代理错、RAG/OJ/监控不可用 | 容器状态、Nginx、Actuator、依赖健康接口 | 服务没起、端口或代理错、环境变量没配、依赖地址过期 | [问题定位流程](/operations/diagnosis-flow)、[Docker 与服务部署](/operations/docker)、[常见问题排查](/operations/troubleshooting) |

## 不是所有失败都长成错误码

很多新人会先找“有没有一个 code”，但项目里的失败态其实至少有 4 种表现方式：

| 表现方式 | 例子 | 该先看什么 |
| --- | --- | --- |
| HTTP 状态失败 | `404`、`405`、`500`、下载接口直接 `403` | 浏览器 Network、Nginx、后端日志 |
| 业务 `code` 失败 | `701`、`703`、`601`、`805` | 响应体、统一异常处理、前端拦截器 |
| WebSocket 事件失败 | `ERROR` 事件里的 `RATE_LIMITED`、`MESSAGE_REJECTED` | WS 面板、事件载荷、`tempId` |
| 页面静默失败 | 接口成功但页面空、统计不更新、列表没刷新 | Response 结构、前端字段名、缓存/聚合逻辑 |

所以不要把“没看到错误码”误当成“没有失败”。

## 失败态排查的第一原则

先找**第一层不对的证据**，不要从最后看到的页面结果开始倒推。

例如：

- 页面空白，不代表一定是前端问题，可能是接口 `200` 但字段结构变了。
- 文件打不开，不代表一定是文件服务坏了，可能是业务层归属没放行。
- OJ 一直 `pending`，不一定是提交逻辑坏了，可能是异步判题根本没跑起来。

## 高风险模块至少要补一条失败态

下面这些模块，如果只测成功路径，基本都不算真的测过：

| 模块 | 最低失败态 |
| --- | --- |
| 鉴权 | 未登录访问、错误登录域访问 |
| 文件存储 | 私有文件读取失败、非法类型上传 |
| 聊天 | ws-ticket 失效、限流或禁言失败态 |
| AI Runtime | Schema 不匹配、RAG 空召回 |
| OJ | 编译错误、判题依赖失败 |
| 学习资产 | 审核拒绝、目标模块写入失败 |
| 积分抽奖 | 库存不足、重复领取、签到字段不一致 |
| 版本历史 | 草稿态不应公开可见 |

如果你现在是从测试角度补文档，这一页最好和 [测试与回归](/guide/testing-regression) 一起看。

## 最好排查、最容易误判、最值得先补文档的失败态

### 最好排查

- `701/702/703`
- 文件 `403/404`
- 表单 `400/601`

这些通常第一证据比较明确。

### 最容易误判

- 接口 `200` 但页面空
- 聊天消息发出后无 ACK
- AI 返回结构漂移
- OJ 提交长时间卡在中间态

这些问题如果只看最终页面，很容易判断反。

### 最值得先补文档

- owner 归属校验
- 审核流拒绝态
- 外部依赖未启动时的“未验证边界”
- 统计/缓存回流延迟

因为这些最容易在团队交接时丢信息。

## 一张按症状反查的速查表

| 你看到的症状 | 第一动作 | 第二动作 |
| --- | --- | --- |
| 一点按钮就跳登录 | 看请求 `code` 是 `701` 还是 `702` | 看 Token 存储和登录域 |
| 后台页面能打开但保存失败 | 看方法有没有 `@RequireAdmin` | 看是不是缺 admin 角色 |
| 我的数据列表为空 | 看当前用户是否取对 | 看 SQL/Mapper 是否按 owner 查询 |
| 文件上传成功但页面图片裂开 | 看返回 URL 和文件状态 | 看 `/api/files/**` 代理和公开权限 |
| 聊天能连上但消息失败 | 看 WS `ERROR` 事件 | 看 `tempId`、限流、禁言、ACK |
| AI 页能返回但字段不全 | 看 Schema、Prompt 调试结果 | 看回归样例和模型输出 |
| OJ 运行正常但提交卡住 | 看提交状态和 go-judge | 看赛事校验和异步判题 |
| 审核通过了但目标模块没更新 | 看发布日志或阶段日志 | 看目标模块写入和通知回流 |
| 所有接口都慢或不通 | 先看依赖和代理 | 再看代码，不要反过来 |

## 最小失败态记录模板

把失败态记录清楚，后面的人会轻松很多。建议至少记这几项：

```text
失败表现：
第一证据：
影响范围：
最可能根因：
已验证不是的问题：
最终修复：
仍未验证项：
相关文档：
```

这份模板很适合回写到：

1. [验证记录与已知问题](/manuals/verified-scenarios)
2. 模块页“常见坑”
3. [常见问题排查](/operations/troubleshooting)

## 推荐配套阅读

如果你想把“失败态”这件事真正吃透，建议按这个顺序看：

1. 先看本页，认识常见失败分类。
2. 再看 [问题定位流程](/operations/diagnosis-flow)，按排查顺序走。
3. 接着看 [响应体与错误码](/reference/response-errors)，理解失败怎么被表达出来。
4. 然后配合 [权限注解与角色边界索引](/reference/permission-boundaries) 和 [模块状态机与生命周期索引](/reference/module-state-machines)，判断失败是卡在权限层还是状态层。
5. 最后回到对应模块页，把这次排查结论沉淀成长期资产。

## 一句话总结

真正有用的失败态文档，不是把所有错误都列一遍，而是让后来的人能更快回答这三个问题：

```text
最早是在哪一层开始失败的？
我该先去抓哪份证据？
我现在应该回哪一页继续查？
```
