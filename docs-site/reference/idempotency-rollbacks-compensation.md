# 幂等、回滚与补偿索引

这页专门回答一个很现实的问题：

**当一次写操作做了两步、三步，或者还牵涉异步、外部服务、统计和回流时，失败了到底要不要回滚，怎么补。**

它和相邻几页的分工是：

- [事件、通知与回流索引](/reference/event-backflow-index)：回答“成功后要回到哪里”。
- [统计、排行与计数口径索引](/reference/statistics-ranking-counts)：回答“数字怎么算、何时更新”。
- 本页：回答“重复提交怎么挡、失败后哪些要回滚、哪些要补偿、哪些只记日志”。

## 怎么用这页

1. 先判断这是主流程写操作，还是附属副作用。
2. 再判断它是不是会重复触发。
3. 再判断失败点发生在主表之前、主表之后、异步任务里，还是外部依赖里。
4. 最后决定是回滚、重试、补偿，还是只记日志。

## 先分清四种处理方式

| 方式 | 适用场景 | 结果 |
| --- | --- | --- |
| 幂等 | 重复点击、重复提交、重试请求 | 同一动作多次执行，结果保持一致 |
| 回滚 | 主流程失败，且能安全撤销 | 之前的主数据和关键副作用撤回 |
| 补偿 | 主流程已成功，但副作用失败或异步延迟 | 后续任务把缺的那一步补回来 |
| 只记日志 | 附属副作用失败，不应阻断主流程 | 主流程成功，失败留痕，后续排查 |

| [OJ 判题系统](/modules/oj) | 提交异步判题、首次 AC 积分回流、赛事约束 | 主提交先落库，判题和积分异步补回 |

## 幂等怎么读

幂等不是“不要重复执行”，而是“重复执行时结果不变”。

常见幂等手段：

- 唯一键。
- 状态机前置判断。
- 去重 Key。
- `INSERT IGNORE` 或 `ON DUPLICATE KEY UPDATE`。
- 先查再写，但必须配合锁或唯一约束。

### 常见幂等场景

| 场景 | 典型做法 | 说明 |
| --- | --- | --- |
| 重复签到 | 位图 + 当日是否已签到判断 | 重复请求只返回已签到 |
| 重复抽奖 | 用户锁 + 限制表 + 记录表 | 同一用户同一窗口只允许一次有效抽奖 |
| 重复发通知 | 唯一记录或接收人去重 | 同一事件不重复发给同一用户 |
| 重复上传文件 | MD5 去重 | 相同文件直接复用已有记录 |
| 重复提交 OJ | 提交记录先落库 | 同一次点击只生成一次提交 |
| 重复发布学习资产 | 转化记录和候选状态校验 | 已发布候选不再重复发布 |

## 回滚怎么读

回滚适合“主流程还没结束，且能安全撤回”的地方。

| 场景 | 回滚什么 | 不回滚什么 |
| --- | --- | --- |
| 抽奖扣库存后失败 | Redis 库存、数据库库存 | 用户已经看到的错误提示 |
| 上传文件后落库失败 | 物理文件 | 用户端已提交动作本身 |
| 判题流程失败 | 提交状态或结果回写 | 已写入的提交记录 |
| 学习资产发布失败 | 候选状态、目标模块映射 | 来源快照和记录本身 |

判断回滚时有个很关键的原则：

> 能回滚主数据，不代表要回滚所有副作用。

比如通知发送失败、日志写入失败、统计任务失败，通常不应该把主流程一起打回去。

## 补偿怎么读

补偿适合“主流程成功了，但副作用晚了、漏了、或依赖临时不可用”的情况。

### 常见补偿信号

- 主表有记录，明细缺失。
- 主状态已推进，但通知没发。
- 主行为完成，但统计没更新。
- Redis 和数据库短暂不一致。
- 外部依赖失败，但主链路已经返回成功。

### 常见补偿方式

| 方式 | 说明 |
| --- | --- |
| 定时扫描补跑 | 定期扫描差异记录，补齐缺失数据 |
| 失败重试 | 对短暂失败的外部调用做重试 |
| 差异对账 | 对比主表、明细表、缓存和统计表 |
| 人工修复 | 生产事故时由维护者按记录修复 |

## 哪些失败只记日志

这些通常不应该阻断主流程：

- 通知发送失败。
- 操作日志写入失败。
- 登录日志写入失败。
- 某些统计任务失败。
- 某些热榜或排行刷新失败。

这类失败应该做到：

1. 记录日志。
2. 保留补偿入口。
3. 不反向影响主流程结果。

## 最容易混的 6 个判断

| 现象 | 先看什么 | 常见结论 |
| --- | --- | --- |
| “接口成功但结果不完整” | 主表、明细、通知、统计 | 可能是副作用失败，不一定是主流程失败 |
| “点了两次但只想生效一次” | 唯一键、锁、状态机 | 这是幂等问题 |
| “已经成功了但页面没更新” | 回流和异步任务 | 这是补偿或刷新问题 |
| “库存对不上” | Redis、数据库、回滚日志 | 这是库存一致性问题 |
| “积分余额和流水不一致” | 余额表、流水表 | 这是事务或异常回滚问题 |
| “通知缺了但主操作成功了” | 通知表、错误日志 | 这是典型只记日志场景 |

## 对照表

| 业务 | 主流程 | 副作用 | 处理建议 |
| --- | --- | --- | --- |
| OJ 提交 | 写提交记录 | 判题、积分回流、排行榜 | 主流程先成功，副作用异步补齐 |
| 抽奖 | 扣积分、扣库存、写记录 | 统计、告警、通知 | 主流程强一致，统计和告警可补偿 |
| 文件上传 | 上传对象、落库 | 备份、迁移、统计 | 主存储失败可降级，附属任务可补 |
| 聊天发送 | 入库消息 | ACK、在线态、广播 | 消息主记录成功后再推实时事件 |
| 动态发布 | 写内容、评论、点赞 | 通知、热度、统计 | 主内容成功，副作用尽量不阻断 |

## 配套阅读顺序

如果你要把“失败后该怎么处理”这件事系统补齐，建议这样读：

1. 先看 [幂等、回滚与补偿索引](/reference/idempotency-rollbacks-compensation)，先分清处理方式。
2. 再看 [事件、通知与回流索引](/reference/event-backflow-index)，确认成功后副作用该去哪里。
3. 接着看 [统计、排行与计数口径索引](/reference/statistics-ranking-counts)，理解哪些数字允许延迟。
4. 然后看 [异常路径与失败态索引](/reference/failure-paths)，把失败证据和回查入口补齐。
5. 最后回到具体模块页，按模块自己的事务、幂等、补偿规则写验证清单。
