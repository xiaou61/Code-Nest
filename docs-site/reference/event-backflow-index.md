# 事件、通知与回流索引

这页回答的是另一个很常见的问题：

**一个动作做完以后，系统应该在哪些地方“回得来”？**

这里的“回流”不只指消息通知，还包括：

- 用户端页面是否马上可见
- 管理端是否可运营
- 状态是否推进
- 统计是否更新
- 审计日志是否落下
- WebSocket 是否 ACK

它和相邻文档的分工是：

- [异常路径与失败态索引](/reference/failure-paths)：告诉你失败时先看哪层。
- [模块状态机与生命周期索引](/reference/module-state-machines)：告诉你状态怎么推进。
- [模块最小回归矩阵](/reference/module-regression-matrix)：告诉你改完模块至少回归什么。
- 本页：告诉你**成功之后应该出现哪些副作用、哪些副作用不该阻断主流程**。

## 怎么用这页

1. 先确定这次动作是“写入主状态”还是“触发副作用”。
2. 再看应该回流到哪里：用户端、管理端、通知中心、日志、统计、排行榜或实时连接。
3. 区分必须成功和 best-effort。
4. 如果副作用没到位，先看第一证据，再回模块页。
5. 如果你关注的是“数值怎么算、为什么慢半拍”，继续看 [统计、排行与计数口径索引](/reference/statistics-ranking-counts)。
6. 如果你关注的是“失败后到底回滚还是补偿”，继续看 [幂等、回滚与补偿索引](/reference/idempotency-rollbacks-compensation)。

## 五种最常见的回流类型

| 回流类型 | 主要观察点 | 常见对象 | 失败时表现 |
| --- | --- | --- | --- |
| 状态回流 | 主表状态、阶段日志、发布日志 | OJ 提交、学习资产、版本、申请记录 | 页面卡在旧状态、列表不刷新 |
| 通知回流 | 通知中心、未读数、阅读记录 | 注册欢迎、审核结果、互动提醒、公告 | 用户不知道结果、未读数不对 |
| 计数回流 | 统计表、排行榜、Redis 计数 | 积分、热度、在线数、排行、浏览量 | 数值落后或与主表不一致 |
| 审计回流 | 登录日志、操作日志、失败日志 | 后台写操作、登录行为、配置变更 | 出事后找不到证据 |
| 实时回流 | WebSocket ACK、ERROR、广播事件 | 聊天、在线态、撤回、踢出 | 前端显示失败但后端已入库 |

## 主流程、回流和审计的关系

很多功能不是“做完就结束”，而是分三层：

| 层 | 作用 | 是否应该阻断主流程 |
| --- | --- | --- |
| 主流程 | 把核心数据写对 | 应该阻断 |
| 回流 | 让其他页面或系统知道结果 | 通常不阻断，失败要留痕 |
| 审计 | 方便事后追责和排障 | 通常不阻断，失败只记日志 |

一个好用的判断是：

- **主状态写错**，这是真问题。
- **通知没发出去**，大多不该回滚主状态。
- **日志没写入**，通常影响排障，不影响业务结果。

## 哪些回流应该优先落地

这几类回流最值得优先保证：

1. **状态回流**，因为它直接决定业务有没有真的推进。
2. **实时回流**，因为前端用户会立刻看到成功或失败。
3. **通知回流**，因为它决定用户和管理员是否知道结果。
4. **审计回流**，因为它决定后面能不能定位问题。

计数回流一般也重要，但通常允许短暂延迟，只要最终一致或可补算。

## 典型 best-effort 回流

这些副作用通常不应该阻断主流程：

- 通知发送失败
- 登录日志写入失败
- 操作日志写入失败
- 某些统计任务失败
- 某些热度或排行异步刷新失败

它们应该：

1. 记录 warn/error 日志。
2. 尽量保留补偿入口。
3. 不要反向把主流程也判失败。

## 最容易漏的 6 个回流点

| 漏点 | 常见后果 | 先看哪里 |
| --- | --- | --- |
| 只改主表，不写发布日志 | 后面不知道怎么推进的 | 日志表、阶段日志 |
| 只改状态，不发通知 | 用户以为没成功 | 通知中心、已读记录 |
| 只改列表，不改统计 | 用户端和管理端口径不一致 | 统计表、Redis |
| 只入库，不发 ACK | 聊天前端一直显示发送中 | WebSocket 事件 |
| 只改页面，不清缓存 | 页面刷新后又回到旧值 | 缓存、查询逻辑 |
| 只做成功路径，不补失败态 | 出错后没人知道该看哪 | [异常路径与失败态索引](/reference/failure-paths) |

## 一张按目标回流的速查表

| 想确认什么 | 第一入口 | 第二入口 |
| --- | --- | --- |
| 用户有没有知道结果 | 通知中心 | 对应模块页 |
| 管理员有没有看到结果 | 仪表盘与日志、管理端页面 | 对应后台模块页 |
| 状态有没有推进 | 状态机索引 | 对应模块页 |
| 实时消息有没有送达 | WebSocket 协议 | IM 聊天室 |
| 统计有没有更新 | 仪表盘与日志、对应统计表 | 对应模块页 |
| 审计有没有留下 | 登录日志、操作日志 | 仪表盘与日志 |

## 配套阅读顺序

如果你要把“回流”这件事系统补齐，建议这样读：

1. 先看本页，知道动作成功后应该出现什么。
2. 再看 [模块状态机与生命周期索引](/reference/module-state-machines)，确认主状态怎么走。
3. 接着看 [异常路径与失败态索引](/reference/failure-paths)，知道回流没到时该先查哪层。
4. 然后看 [通知中心](/modules/notification) 和 [仪表盘与日志](/modules/dashboard-logs)，把通知、日志、审计和统计的落点搞清楚。
5. 最后回到 [核心链路教程](/manuals/core-workflows) 或具体模块页，把“接口成功、用户端回流、管理端可见”写成稳定验收口径。

## 一句话总结

一个功能真正闭环，不是接口返回 `200`，而是：

```text
主状态写对了
+ 用户看见结果了
+ 管理员查得到
+ 日志留得住
+ 统计对得上
```
