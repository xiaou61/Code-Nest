# 事故响应

本页适合"已经影响用户，需要先止损再排查"的情况。
如果你还在判断问题属于哪一类，先看 [监控与观测](/operations/monitoring) 和 [问题定位流程](/operations/diagnosis-flow)；如果主要是本地开发或部署故障，再看 [常见问题排查](/operations/troubleshooting)。

## 这页解决什么

| 场景 | 先做什么 | 不要先做什么 |
| --- | --- | --- |
| 告警突然升高 | 先确认影响面和开始时间 | 先大改代码 |
| 用户大量失败 | 先止损，必要时降级或回滚 | 只看单条报错 |
| 服务不可用 | 先保住核心链路 | 先排查所有细节 |
| 变更后出问题 | 先对照最近发布和配置变化 | 先重启一切 |

## 先做三件事

1. **确认影响范围**：是用户端、管理端、后端接口、AI、OJ、文件、聊天，还是部署层。
2. **留下最小证据**：告警名、开始时间、请求 URL、状态码、日志片段、截图。
3. **先止损再定位**：能回滚就先回滚，能降级就先降级，能关入口就先关入口。

## 事故分级

| 等级 | 判断 | 常见处理 | 响应时间 |
| --- | --- | --- | --- |
| P0 | 大面积不可用，核心链路断了 | 立即止损、回滚、广播通知 | 5 分钟内 |
| P1 | 关键功能大面积异常 | 降级、限流、切换配置 | 15 分钟内 |
| P2 | 局部模块异常，但主链路还在 | 先定位，再择机修复 | 1 小时内 |

### 核心链路定义

以下为 P0 必保链路，任一中断即升级：

| 链路 | 涉及模块 | 判断标准 |
|------|----------|----------|
| 用户登录 | user, system | 登录接口 5xx > 10% |
| 社区浏览 | community, moment | 列表接口 5xx > 10% |
| OJ 提交 | oj | 提交接口 5xx > 10% |
| AI 对话 | ai | AI 接口 5xx > 20% |

## 10 分钟动作清单

```text
0:00  确认告警是否真实（排除误报）
0:02  查看最近变更：发版/改配置/改代理/改SQL/换模型/重启依赖
0:05  确定影响范围：单模块 or 全局
0:07  选择止损方式：回滚 > 降级 > 关入口 > 切配置
0:10  写下止损动作和结果，通知相关人
```

## 止损方式对照表

| 问题类型 | 首选止损 | 操作 | 回滚检查 |
|----------|----------|------|----------|
| 接口 5xx 飙升 | 回滚最近发布 | 保留上一个构建版本，`java -jar` 回退 | 确认回滚后 5xx 下降 |
| 登录失败 | 检查鉴权配置 | 对照 `application.yml` 检查 Sa-Token 配置 | 确认两端登录独立 |
| 数据库连接耗尽 | 限流 + 扩连接池 | `druid.maxActive` 临时调大 | 确认连接池恢复正常 |
| Redis 不可用 | 降级非核心功能 | 关闭缓存、使用数据库直接查询 | 确认 Redis 恢复后切回 |
| 文件访问失败 | 恢复代理/挂载 | 检查 Nginx alias 配置 | 确认文件可访问 |
| 聊天不可用 | 暂停实时入口 | 保留公告和离线消息 | 确认 WebSocket 恢复 |
| AI/OJ 不可用 | 暂停高风险入口 | 显示维护提示 | 确认外部服务恢复 |
| 内存溢出 | 重启 + 降级 | `-Xmx` 调大或关闭非常用功能 | 确认堆内存使用正常 |

## 模块级降级策略

| 模块 | 可降级 | 降级方式 | 不可降级时 |
|------|--------|----------|-----------|
| AI 对话 | 是 | 返回维护提示，跳过 AI 调用 | 检查 API Key 和网络 |
| OJ 判题 | 是 | 新提交排队，返回"排队中" | 检查 go-judge 连通性 |
| 聊天室 | 是 | 关闭 WebSocket，降级为轮询 | 检查 Redis 在线态 |
| 文件上传 | 部分 | 限制文件大小/类型 | 检查磁盘空间 |
| 社区/博客 | 否 | 核心内容，不可降级 | 直接回滚 |
| 用户登录 | 否 | 核心链路，不可降级 | 立即回滚 |

## 诊断命令速查

```bash
# 后端状态
curl -s http://localhost:9999/api/actuator/health | python3 -m json.tool

# Prometheus 指标
curl -s http://localhost:9999/api/actuator/prometheus | grep http_server_requests

# MySQL 连接数
mysql -u root -p -e "SHOW PROCESSLIST;" | wc -l

# Redis 内存
redis-cli info memory | grep used_memory_human

# 查看最近错误日志
tail -100 logs/code-nest.log | grep -i "exception\|error"

# 查看系统资源
# Linux:
top -b -n 1 | head -5
df -h
# Windows:
tasklist /fi "imagename eq java.exe"
```

## 恢复后检查清单

止损后不要立刻离开，按以下清单确认：

| 检查项 | 方法 | 通过标准 |
|--------|------|----------|
| 健康检查 | `/actuator/health` | `status: UP` |
| 核心接口 | 手动测试登录+浏览+提交 | 返回正常 |
| 错误率 | `/actuator/prometheus` | 5xx < 1% |
| 告警恢复 | 确认告警已消除 | 无新告警 |
| 日志干净 | `grep -i error` | 无新异常 |
| 数据完整 | 抽查关键数据 | 无丢失/不一致 |

## 复盘模板

| 字段 | 写法 | 示例 |
| --- | --- | --- |
| 事故标题 | 模块+现象+影响 | OJ 判题超时导致提交堆积 |
| 时间线 | 发现→止损→恢复→关闭 | 14:00 告警 → 14:03 降级 → 14:15 恢复 |
| 影响范围 | 模块+用户数+持续时间 | OJ 模块，约 200 用户，15 分钟 |
| 根因 | 最底层的原因 | go-judge 沙箱内存不足被 OOM Kill |
| 止损动作 | 做了什么 | 降级 OJ 提交为排队模式 |
| 修复 | 怎么修的 | 增加沙箱容器内存限制 |
| 预防 | 防止再发的措施 | 添加沙箱健康检查告警 |
| 文档更新 | 哪些文档要改 | 告警 runbook + OJ 模块页 |
