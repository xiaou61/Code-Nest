# 事故响应

本页适合“已经影响用户，需要先止损再排查”的情况。  
如果你还在判断问题属于哪一类，先看 [监控与观测](/operations/monitoring) 和 [问题定位流程](/operations/diagnosis-flow)；如果主要是本地开发或部署故障，再看 [常见问题排查](/operations/troubleshooting)。

## 这页解决什么

| 场景 | 先做什么 | 不要先做什么 |
| --- | --- | --- |
| 告警突然升高 | 先确认影响面和开始时间 | 先大改代码 |
| 用户大量失败 | 先止损，必要时降级或回滚 | 只看单条报错 |
| 服务不可用 | 先保住核心链路 | 先排查所有细节 |
| 变更后出问题 | 先对照最近发布和配置变化 | 先重启一切 |

## 先做三件事

1. 确认影响范围：是用户端、管理端、后端接口、AI、OJ、文件、聊天，还是部署层。
2. 留下最小证据：告警名、开始时间、请求 URL、状态码、日志片段、截图。
3. 先止损，再定位：能回滚就先回滚，能降级就先降级，能关入口就先关入口。

## 事故分级

| 等级 | 判断 | 常见处理 |
| --- | --- | --- |
| P0 | 大面积不可用，核心链路断了 | 立即止损、回滚、广播通知 |
| P1 | 关键功能大面积异常 | 降级、限流、切换配置 |
| P2 | 局部模块异常，但主链路还在 | 先定位，再择机修复 |

## 10 分钟动作清单

1. 看监控：先确认是不是已经超过阈值。
2. 看变更：最近是否发版、改配置、改代理、改 SQL、换模型、重启依赖。
3. 看范围：单模块问题还是全局问题。
4. 选止损：回滚、降级、关闭入口、切换备用配置。
5. 留记录：写清楚你做了什么、结果怎样、下一步是谁接手。

## 常见止损方式

| 问题 | 常见止损 | 注意点 |
| --- | --- | --- |
| 接口 5xx 飙升 | 回滚最近发布、临时关闭高风险入口 | 先确认是否是单模块 |
| 登录失败 | 检查鉴权配置，必要时恢复上一版配置 | 用户端和管理端登录域不同 |
| 文件访问失败 | 恢复代理、挂载或访问前缀 | 不要把私有文件直接暴露成静态资源 |
| 聊天不可用 | 暂停实时入口，保留公告和离线说明 | 重点看 WebSocket 和 Redis 在线态 |
| AI/OJ 不可用 | 暂停高风险入口或切换维护模式 | 保留失败样本和回归记录 |

## 恢复后要做什么

1. 重新验证核心链路。
2. 补一条复盘记录，写清根因和修复点。
3. 更新相关文档，把这次事故能教会别人的东西留下来。

## 记录模板

| 字段 | 写法 |
| --- | --- |
| 现象 | 用户端聊天发送图片后失败 |
| 影响范围 | `/chat`、文件访问 URL、消息展示 |
| 根因 | 返回路径和部署代理不一致 |
| 止损动作 | 回滚/降级/关闭入口 |
| 修复 | 调整代理或代码路径 |
| 验证 | 上传、发送、刷新、复查监控 |
| 文档 | 更新监控、排查、模块页和复盘记录 |
