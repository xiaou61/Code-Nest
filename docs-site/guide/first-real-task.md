# 首个真实任务接入指南

这页的目标很明确：让新人从“学会定位与验证”走到“能安全地接第一个真实任务”。

它不是替代 [功能开发流程](/guide/feature-development)，而是站在新人视角回答三个问题：

1. 第一个真实任务应该选什么类型？
2. 哪些任务不应该作为第一单？
3. 第一次交付要怎么做到“稳、可验证、可交接”？

如果你正在跑带教流程，建议先读：

- [带教执行剧本](/guide/onboarding-execution-playbook)
- [带教交付模板包](/guide/onboarding-deliverable-templates)

## 0. 首单任务的定义

所谓“首个真实任务”，不是“能合并到 master 的大功能”，而是满足下面三点的第一单：

1. 有明确输入输出：别人能复述你要改什么、为什么改。
2. 有最小验证：能跑命令、能走页面或接口，不靠“我觉得”。
3. 有可交接结论：别人拿着你的记录能继续做或回滚。

你要的不是一次做大，而是第一次把闭环跑完整。

## 1. 先选对任务：任务类型分级

为了让首单更稳，建议把任务按风险分级：

| 级别 | 适合作为首单 | 典型任务 |
| --- | --- | --- |
| L0 | 强烈推荐 | 文档断链修复、导航补入口、索引同步、脚本说明补齐 |
| L1 | 推荐 | 小范围前端页面 bug、文案/空态/失败态修复、管理端列表筛选小修 |
| L2 | 谨慎（需要带教陪跑） | 后端普通 CRUD、小范围 SQL 修复、DTO 字段补齐 |
| L3 | 不建议作为首单 | 鉴权登录域、文件公开读取边界、WebSocket 协议、积分规则、敏感词策略、AI Prompt/Schema、OJ 判题链路、部署与 Nginx 改动 |

首单的策略是：先把闭环跑完，再逐步上强度。

## 2. 哪些任务不要先碰（避雷清单）

下面这些任务不建议作为首单，原因不是你做不了，而是验证成本与影响面太大：

| 模块/能力 | 不建议原因 |
| --- | --- |
| 鉴权与登录域 | 一改就可能全站 401 或权限错乱 → 详见 [鉴权与用户体系](/modules/auth) |
| 文件访问与公开读取 | 最容易出现越权或 URL 映射问题 → 详见 [文件存储](/modules/file-storage) |
| WebSocket 与聊天协议 | 失败态复杂，联调与重连成本高 → 详见 [WebSocket 协议](/reference/websocket) |
| 积分与抽奖 | 幂等、库存、一致性验证要求高 → 详见 [积分与抽奖](/modules/points)、[幂等回滚与补偿](/reference/idempotency-rollbacks-compensation) |
| 敏感词风控 | 误杀/漏判都影响用户内容生产 → 详见 [敏感词风控](/modules/sensitive) |
| AI Prompt/Schema/RAG | "200 不等于正确"，回归与依赖要求高 → 详见 [AI Schema 与治理](/reference/ai-schemas) |
| OJ 判题/go-judge | 强依赖外部服务，状态机与回写链路长 → 详见 [OJ 判题系统](/modules/oj) |
| Docker/Nginx/监控 | 环境差异大，一旦出错难定位 |

如果你的首单落在这些领域，建议降级：先做文档校准或只读验证记录补齐，而不是直接改代码。

## 3. 首单任务怎么选（推荐决策流程）

推荐用下面 6 个问题筛选任务：

1. 我能在 1 小时内复现问题或确认需求边界吗？
2. 我能在 2 小时内定位到页面/Controller/表至少其中两项吗？
3. 我能列出最低验证项吗？（构建 + 页面/接口烟测）
4. 这次改动会牵连鉴权/文件/积分/AI/OJ/WebSocket 吗？
5. 如果牵连了，我能否把任务拆成一个更小的可验证子任务？
6. 如果改坏了，我能否快速回滚或恢复？

如果 1-3 答不上来，就不是好首单；如果 4-6 风险太大，就先拆小再做。

## 4. 首单交付标准（Definition of Done）

首单不要求“改得很漂亮”，但必须满足下面最小交付标准：

| 项 | 必须达到 |
| --- | --- |
| 边界 | 这次改动只解决一个明确问题，不做额外扩展 |
| 定位 | 写清楚入口、相关模块、涉及接口或表 |
| 风险 | 明确是否涉及高风险横切能力（鉴权/文件/AI/OJ/WebSocket 等） |
| 验证 | 至少跑构建，并明确写出业务烟测或未验证原因 |
| 文档同步 | 如果新增页面/接口/表，必须同步对应索引页 |
| 交接 | PR 描述里写清已验证和未验证项 |

最低验证怎么选，参考：

- [测试与回归](/guide/testing-regression)
- [发布前验证](/guide/release-verification)

## 5. 首单任务的推荐落点（从哪里找题）

如果你没有现成需求，首单最稳定的来源通常是“同步与校准”：

1. 文档断链或入口缺失：导航、模块总览、索引页互链。
2. 索引同步：前端路由索引 / API 路由索引 / 数据表索引补齐。
3. 文档与源码不一致：以源码为准，修正文档，并记录校准点。
4. 失败态缺失：某个模块页验证清单里缺“未登录/权限不足/空态/失败态”说明。

如果你准备做代码类首单，建议从这些方向选：

1. 小范围前端 bug：空态、错误提示、按钮禁用、加载态。
2. 管理端小修：列表筛选、分页重置、批量操作确认提示。
3. 后端低风险：字段校验补齐、错误码返回统一、日志补充（不涉及鉴权策略）。

## 6. 做首单时最稳的执行顺序

这是一条“新人更不容易翻车”的执行顺序：

1. 先写复现/边界：一句话说明问题与期望结果。
2. 再写定位：入口/模块/接口/表至少写出两项。
3. 再写风险：是否涉及横切能力、是否依赖外部服务。
4. 再写最小验证清单：构建 + 1 条业务烟测。
5. 最后再动手改代码或文档。
6. 改完立刻按清单验证，并把结果写进 PR 描述。

如果你发现需要动到高风险边界（鉴权、文件、积分、AI、OJ、聊天），先停一下，把任务拆小，或者先补“验证记录 + 文档校准”作为首单。

## 8. 结业信号：什么时候可以接第二单

如果你的首单满足下面三个信号，基本可以开始接第二单并逐步上强度：

1. 你能稳定写出定位与影响面，而不是靠猜。
2. 你能稳定给出最小验证，并诚实记录未验证边界。
3. 你能稳定把交接写清楚，别人能复用你的结论。

如果你做完首单仍然觉得“每一步都心里没底”，建议回到：

- [模块学习任务模板](/guide/module-learning-workbook)
- [模块学习评审表](/guide/module-learning-rubric)

把“定位、链路、风险、验证”这四项再练一次，再来接第二单会更稳。

如果你现在就要给新人分配具体任务，可以直接打开 [首单任务题库](/guide/first-task-backlog) 挑一题。

如果首单已经顺利做完，下一步不要只凭感觉决定“能不能接模块”，建议直接接着看 [模块接手升级路线](/guide/module-ownership-ladder)。
