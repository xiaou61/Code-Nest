# 带教执行剧本

这页把 Code Nest 的学习文档变成可执行的 onboarding 流程，适合团队带新人，也适合你自己用“每日交付物 + 评审节点”的方式快速进入项目。

如果你只想知道“该读什么”，先看：

- [学习路线](/guide/learning-paths)（按角色）
- [按模块学习路线](/guide/module-learning-paths)（按业务模块）

如果你想把学习变成可验收交付，配套使用：

- [模块学习任务模板](/guide/module-learning-workbook)
- [模块结业案例](/guide/module-learning-capstone)
- [模块学习评审表](/guide/module-learning-rubric)
- [带教交付模板包](/guide/onboarding-deliverable-templates)

## 使用原则

带教最怕两件事：

1. 新人“看完了文档”，但仍然不会定位、不敢验证、不知道怎么交接。
2. 带教“凭感觉评价”，导致标准不一致、反馈不聚焦。

所以本剧本只强调三件事：

1. 每天交付物必须可复用、可继续执行。
2. 每天必须有最小验证，不接受“我觉得可以”。
3. 每天反馈必须指向 rubric 的某一项维度，而不是泛泛而谈。

## 带教前准备（30-60 分钟）

### 1) 选路线

二选一即可，不要两条路线一起跑：

- 按角色：走 [学习路线](/guide/learning-paths) + [角色上手任务包](/guide/onboarding-tasks)
- 按模块：走 [按模块学习路线](/guide/module-learning-paths) + [模块学习任务模板](/guide/module-learning-workbook)

如果团队目标是“尽快能接一个业务模块”，更推荐按模块路线。

### 2) 确认环境与启动

至少完成文档站可构建：

- [本地完整启动剧本](/guide/startup-playbook)
- `docs-site` 构建命令见 [测试与回归](/guide/testing-regression)

如果带教目标包含真实业务验证，提前约定本次 onboarding 需要启动哪些依赖（例如 MySQL、Redis、AI sidecar、go-judge）。

### 3) 建立“作业存放位置”

推荐在团队内部用一个固定位置保存交付物（例如一个仓库目录或文档空间）。关键是：交付物要能被下一位继续复用。

本剧本不强制工具，但强制格式：交付物必须能套进 [模块学习任务模板](/guide/module-learning-workbook) 的四类模板。

如果你不想自己临时写日报、评审意见和返工通知，可以直接套用 [带教交付模板包](/guide/onboarding-deliverable-templates)。

## 两套节奏：3 天游与 5 天游

如果你时间紧，用 3 天游；如果希望更稳，用 5 天游。两套节奏的交付物格式完全一致。

| 节奏 | 适合场景 | 目标 |
| --- | --- | --- |
| 3 天游（冲刺） | 项目急需人手、已有强工程基础 | 让新人能接一个低到中风险模块任务 |
| 5 天游（标准） | 新人对业务不熟、模块多且复杂 | 让新人稳定具备定位、验证与交接能力 |

## 3 天游（冲刺版）

### Day 1：全局骨架 + 第一站作业

**目标**：建立全局地图，并完成“账号/安全边界”作业。

必读：

- [架构总览](/architecture/overview)
- [源码地图](/reference/source-map)
- [模块总览](/modules/)
- [按模块学习路线](/guide/module-learning-paths)
- [模块学习任务模板](/guide/module-learning-workbook)

交付物（必须）：

1. 一份模块画像卡（账号/安全边界）。
2. 一份链路追踪单（任选登录后接口链路）。
3. 一份验证记录单（至少包含 docs-site 构建结果）。

评审点（15-20 分钟）：

- 用 [模块学习评审表](/guide/module-learning-rubric) 快速打分，重点看“模块定位”和“验证设计”。
- 只给 2 条反馈：一条必须返工项，一条建议加强项。

### Day 2：选择一条主业务链路做深读

**目标**：完成一个典型主业务模块的链路拆解，并能给出最小验证清单。

推荐二选一：

- OJ 线：[OJ 判题系统](/modules/oj) + [积分与抽奖](/modules/points)
- AI 线：[模拟面试与求职作战台](/modules/mock-interview-job-battle) + [学习资产](/modules/learning-assets)

交付物（必须）：

1. 模块画像卡（所选链路）。
2. 链路追踪单（写清状态变化、落库、回流、失败态）。
3. 验证记录单（明确依赖是否启动，未验证项必须写原因）。

评审点（20-30 分钟）：

- 重点看“链路理解”“风险判断”“未验证项是否诚实”。

### Day 3：结业案例 1 + 真实小任务拆解

**目标**：完成一份结业案例级别输出，并能把一个真实小需求拆成开发流程。

必做：

- 从 [模块结业案例](/guide/module-learning-capstone) 任选 1 个案例先自己做，再对照示范答案。
- 选一个真实小任务（不一定要写代码），按 [功能开发流程](/guide/feature-development) 写一份“开发与验证计划”。

交付物（必须）：

1. 一份结业案例交付（四类模板齐全）。
2. 一份“开发与验证计划”（含影响面、验证范围、文档同步清单）。

评审点（30-45 分钟）：

- 全量按 rubric 打分，给出结论：通过 / 补充后通过 / 不通过。

## 5 天游（标准版）

### Day 1：全局骨架 + 账号/安全边界

内容与交付物同 3 天游 Day 1，但评审更细：必须补齐“文档回写清单”（哪怕本次不真的修改文档）。

### Day 2：学习成长基础盘（任选其一做深读）

推荐从下面选一个：

- [面试题库](/modules/interview)
- [闪卡](/modules/flashcard)
- [计划与学习小组](/modules/plan-team)

目标是训练“学习状态记录”和“统计回流”，不要只写页面行为。

### Day 3：内容发布闭环（社区/博客/动态任选其一）

推荐：

- [社区帖子](/modules/community)
- [动态广场](/modules/moments)
- [博客](/modules/blog)

要求必须包含：

- 一个敏感词或渲染安全边界点（参考 [前端渲染安全](/reference/frontend-rendering-security) 与 [敏感词风控](/modules/sensitive)）。
- 一个用户端 + 管理端联动验证清单。

### Day 4：高风险链路（OJ 或 AI 二选一）

同 3 天游 Day 2，但要求必须补 1 条失败态验证设计（哪怕暂时无法执行）。

### Day 5：结业案例 2 + 上线前验证意识

必做：

- 结业案例至少做 2 个（推荐 OJ + 内容，或 AI + 内容）。
- 按 [发布前验证](/guide/release-verification) 写一份“如果我改了 X，这次我会跑哪些最小验证”的清单。

## 带教评审会议怎么开（推荐 20-30 分钟）

议程固定，避免发散：

1. 学习者用 3 分钟复述本次交付物（只讲结论，不讲过程）。
2. 评审者用 5 分钟对照 rubric 打分并给出结论。
3. 共同确定 1 个必须返工项 + 1 个下一步挑战模块。
4. 明确“未验证项”是否可接受，以及补测计划。

## 返工规则（强烈建议）

返工不要整篇重写，只返工评审命中的最小范围。

| 问题类型 | 返工最小动作 |
| --- | --- |
| 定位不清 | 补齐入口、模块、接口前缀、核心表四项 |
| 链路不清 | 补齐状态变化、落库结果、回流结果三项 |
| 风险空泛 | 至少补 1 个横切能力 + 1 个失败态 |
| 验证不可信 | 把已验证/未验证/原因拆开写，并补一条可执行命令 |
| 交接不可复用 | 删除模糊指代，补“下一步建议”和“可执行入口” |

## 常见带教失败模式（以及解法）

| 失败模式 | 表现 | 解法 |
| --- | --- | --- |
| 只读不产出 | 说“看懂了”，但交不出模板 | 强制四类交付模板 |
| 只成功不失败 | 只写 happy path | 强制写至少一个失败态 |
| 只构建不业务 | 只说 build 过了 | 强制写真实链路验证或明确未验证边界 |
| 评审太主观 | “感觉还行/不行” | 统一用 rubric 的维度给反馈 |
| 反馈太泛 | “多看看源码” | 指向一个维度，给最小返工动作 |

## 完成标准：什么时候算“可以接模块了”

建议的最低门槛：

1. 至少 2 份模块作业达到 rubric 85 分以上。
2. 至少 1 份结业案例达到 rubric 85 分以上。
3. 能说清楚 1 个模块的“入口 -> 接口 -> 表 -> 状态 -> 验证”。
4. 能写出“已验证/未验证/原因/下一步”的诚实交接。

满足以上条件后，就可以开始接第一个真实小任务了。建议从低到中风险模块开始，不要一上来就接 AI/OJ/聊天这类高风险链路。

如果你准备把这套流程真的在团队内跑起来，下一步最省心的做法就是把 [带教交付模板包](/guide/onboarding-deliverable-templates) 直接拿去复制使用。

准备把“学习闭环”切换成“交付闭环”时，建议直接接 [首个真实任务接入指南](/guide/first-real-task)。

如果带教已经走到“该发第一单”的阶段，直接使用 [首单任务题库](/guide/first-task-backlog) 会更省事。

如果第一单已经跑完，接下来就不要再靠感觉升难度了，直接用 [模块接手升级路线](/guide/module-ownership-ladder) 判断应该继续停留在低风险任务，还是开始进入模块接手阶段。
