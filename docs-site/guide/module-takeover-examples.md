# 模块接手样例包

这页是给 [模块接手检查清单](/guide/module-takeover-checklist) 配的“示范答案”。

如果上一页解决的是“接手模块时该盘什么”，那这页解决的是：

同样是接手模块，内容类、状态机类、外部依赖类，到底应该怎么写出一份靠谱的接手记录。

## 这页怎么用

最好的用法不是直接照抄，而是按下面顺序看：

1. 先自己按 [模块接手检查清单](/guide/module-takeover-checklist) 填一版。
2. 再来对照本页，看自己漏了入口、表、边界、失败态还是文档同步点。
3. 最后回到 [模块接手升级路线](/guide/module-ownership-ladder)，判断自己当前更适合接哪一类模块。

这页故意选了三种不同类型：

| 样例 | 模块类型 | 适合阶段 | 主要训练什么 |
| --- | --- | --- | --- |
| 社区帖子 | 内容类 | `S2 -> S3` | 入口盘点、敏感词、渲染安全、用户端/管理端联动 |
| 学习资产 | 状态机与审核类 | `S3 -> S4` | 记录状态、审核流、发布回流、通知与目标模块映射 |
| OJ 判题系统 | 外部依赖高风险类 | `S4+` | 异步判题、外部依赖、状态机、赛事约束、失败态验证 |

## 样例 1：社区帖子接手记录

这是最适合第一次真正接模块时练手的类型之一。

它的好处是：

1. 用户端和管理端入口都清楚。
2. 写入、互动、运营三个层次都在一个模块里。
3. 有风险边界，但不至于像 OJ/AI 那样强依赖外部服务。

### 适合谁

- 已经能稳定完成 `L0/L1`。
- 准备从“单点任务”过渡到“模块连续维护”。
- 想先训练内容风控、交互回流和后台联动。

### 示例接手记录

```text
【模块接手记录】
模块名称：社区帖子
当前阶段：S2 / S3
接手目标：在陪跑下开始连续维护

一、入口地图
- 用户端入口：/community、/community/posts/:id、/community/create、/community/my-posts
- 管理端入口：/community/categories、/community/tags、/community/posts、/community/comments、/community/users
- 后端模块：xiaou-community
- 文档入口：社区模块页、前端渲染安全、模块总览、必要时用户/管理端操作手册

二、接口与数据
- 核心接口前缀：/community/posts、/community/posts/{postId}/comments、/admin/community/posts、/admin/community/comments
- 关键写接口：发帖、评论创建、评论回复、点赞、收藏、置顶、禁用、删除
- 核心表：community_post、community_comment、community_user_status
- 关键状态字段：community_post.status、community_comment.status、community_user_status.isBanned

三、边界与依赖
- 高风险边界：敏感词命中、Markdown/富文本渲染安全、封禁用户绕过、AI 摘要失败但主链路不能受阻
- 依赖能力：鉴权、敏感词、通知、可选 AI 摘要、前端渲染安全
- 外部服务：AI 摘要相关服务可能不可用，但不能影响发帖主流程

四、最低验证
- 成功路径：正常登录用户发帖 -> 详情页可见 -> 管理端帖子列表可见
- 失败态：被封禁用户尝试发帖；帖子正文包含敏感词；重复点赞/收藏
- 当前未验证：AI 摘要缓存命中与强制刷新
- 未验证原因：需要 AI 摘要能力已启用且本地依赖可用

五、已知问题
- 回复显示层级容易误解，所有回复都挂在一级评论下，关系靠 replyTo* 字段表达
- 帖子详情缓存和热门缓存存在 TTL，排查“列表没变”时要先区分缓存问题还是写操作未清缓存
- AI 摘要是可选能力，不能把摘要成功误当成帖子主链路成功

六、文档同步点
- 新页面或管理操作变化：前端路由索引、社区模块页、操作手册
- 新接口前缀或错误口径变化：API 路由索引、响应体与错误码
- 新风险边界或补测结果：验证记录与已知问题、必要时发布前验证

七、当前接手结论
- 现在能独立接：列表筛选、评论展示、帖子状态、后台运营入口、低风险交互问题
- 现在仍需陪跑：敏感词策略变化、AI 摘要链路、缓存一致性问题
- 下周优先补：评论树和通知回流的最小回归清单
```

### 这个样例为什么算合格

它不是只说“社区模块有发帖和评论”，而是已经把：

1. 用户端、管理端、后端入口都拉齐了。
2. 内容模块最怕出事的敏感词和渲染安全标了出来。
3. “能独立接什么”和“还不能独立接什么”分开写清楚了。

### 如果你在这个样例里最容易漏什么

| 漏项 | 为什么危险 |
| --- | --- |
| 只写发帖，不写后台入口 | 很容易把内容模块误当成纯用户侧功能 |
| 不写封禁或敏感词 | 一到真实运营场景就会踩边界 |
| 把 AI 摘要和发帖主链路混为一谈 | 依赖异常时容易误判故障范围 |

## 样例 2：学习资产接手记录

这个样例比社区再往前一步，更适合已经开始接“有记录状态和审核流”的人。

它的难点不在页面多少，而在于：

1. 一次转化会跨记录、候选、发布日志三层数据。
2. 用户确认和管理员审核是两个阶段。
3. 发布后还要落到别的目标模块。

### 适合谁

- 已经能稳定处理普通内容类或后台小写操作。
- 想开始练“状态推进 + 审核 + 回流 + 通知”。
- 准备从 `S3` 往 `S4` 走。

### 示例接手记录

```text
【模块接手记录】
模块名称：学习资产
当前阶段：S3 / S4
接手目标：开始承担同一模块的连续需求与缺陷

一、入口地图
- 用户端入口：/learning-assets
- 管理端入口：/learning-assets/review、/learning-assets/statistics
- 后端模块：xiaou-learning-asset
- 文档入口：学习资产模块页、模块依赖地图、验证记录与已知问题、必要时目标模块页（闪卡/知识图谱/面试题库/计划）

二、接口与数据
- 核心接口前缀：/user/learning-assets、/admin/learning-assets
- 关键写接口：发起转化、编辑候选、确认候选、发布、重试、审核通过、合并、拒绝
- 核心表：learning_asset_record、learning_asset_candidate、learning_asset_publish_log
- 关键状态字段：record.status、candidate.status、targetModule、targetId、sourceType、sourceId

三、边界与依赖
- 高风险边界：来源快照丢失、候选状态误推进、直接发布与审核发布混淆、发布成功但目标模块映射不完整
- 依赖能力：鉴权、通知、目标模块写入、人工审核、可能的 AI 内容辅助生成
- 外部服务：来源数据可能来自博客/社区/CodePen/模拟面试，某些来源内容权限不足时会导致转化内容不完整

四、最低验证
- 成功路径：用户发起转化 -> 产生候选 -> 用户确认 -> 发布闪卡成功
- 失败态：用户编辑 REVIEWING 候选；管理员通过知识节点但未指定 mapId；所有候选拒绝后记录转 FAILED
- 当前未验证：CodePen 付费源码权限不足时的转化内容缺失表现
- 未验证原因：需要带特定权限和真实来源数据联调

五、已知问题
- 记录状态和候选状态不是同一个层级，排查时很容易只看其中一层
- 知识节点和面试题属于审核发布，不能拿闪卡直接发布的路径来套
- 目标模块写入成功后，还要检查 targetModule/targetId 是否真的回写完整

六、文档同步点
- 新来源类型：学习资产模块页、对应来源模块页、必要时 API 索引和测试回归说明
- 新目标资产类型：学习资产模块页、目标模块页、覆盖矩阵
- 新失败态或审核口径变化：验证记录与已知问题、发布前验证

七、当前接手结论
- 现在能独立接：普通候选编辑、发布结果展示、审核页低风险交互、统计口径文档同步
- 现在仍需陪跑：新 sourceType、新 targetType、发布后目标模块写入异常、跨模块回流问题
- 下周优先补：审核发布路径的最小回归清单，以及通知结果回流检查
```

### 这个样例最值得学什么

这个样例最重要的不是记住所有表名，而是学会两件事：

1. **分层看状态**：记录状态、候选状态、发布日志状态不能混着看。
2. **分开看发布方式**：直接发布和审核发布不是一回事，验证路径也完全不同。

### 最容易漏掉的点

| 漏项 | 为什么危险 |
| --- | --- |
| 只写记录状态，不写候选状态 | 很容易误判“为什么用户还能编辑/为什么管理员不能审” |
| 只看发布成功，不看 targetId 回写 | 后续查不到目标内容，模块就像“发出去了但没落地” |
| 不区分来源权限问题 | CodePen 或其他来源权限不足时会出现“转化成功但内容不完整” |

## 样例 3：OJ 判题系统接手记录

这个样例是故意放在最后的。

不是因为它最重要，而是因为它最不适合一上来就独立接。

OJ 的典型难点是：

1. 写接口返回成功，不代表判题闭环完成。
2. 状态机长，且很多问题会卡在异步执行和外部依赖。
3. 用户提交、赛事约束、排行榜、积分回流可能是多段逻辑。

### 适合谁

- 已经能稳定接住普通状态机或审核模块。
- 对外部依赖、异步任务和失败态回写有基本经验。
- 至少已经接近 `S4`，或者在更高阶段陪跑。

### 示例接手记录

```text
【模块接手记录】
模块名称：OJ 判题系统
当前阶段：S4+
接手目标：在陪跑下参与高风险模块值守

一、入口地图
- 用户端入口：/oj、/oj/problem/:id、/oj/my-submissions、/oj/contests、/oj/ranking、/oj/playground
- 管理端入口：/oj/problems、/oj/contests、/oj/tags
- 后端模块：xiaou-oj
- 文档入口：OJ 模块页、模块依赖地图、API 调用示例、验证记录与已知问题、必要时积分模块页

二、接口与数据
- 核心接口前缀：/oj/run、/oj/test、/oj/submit、/oj/submissions/**、/oj/contests/**、/admin/oj/**
- 关键写接口：正式提交、赛事提交、题目维护、测试用例维护、赛事配置
- 核心表：oj_submission、oj_problem、oj_test_case、oj_contest、oj_contest_participant、oj_contest_problem
- 关键状态字段：oj_submission.status、contest.status、time_used、memory_used、error_message

三、边界与依赖
- 高风险边界：异步判题卡 pending/judging、go-judge 不可用、测试用例缺失、赛事时间窗不合法、首次 AC 积分回流异常
- 依赖能力：鉴权、积分、Markdown 渲染安全、赛事规则、外部判题服务
- 外部服务：docker/go-judge；必要时还要联动积分模块和日志

四、最低验证
- 成功路径：正常提交正确代码 -> pending -> judging -> accepted -> 我的提交页可见结果
- 失败态：编译错误；无测试用例题目；赛事未报名提交；go-judge 不可用导致 system_error
- 当前未验证：首次 AC 积分回流与赛事榜单联动
- 未验证原因：需要 go-judge、赛事数据和相关依赖都完整可用

五、已知问题
- 接口返回提交成功只代表记录写入成功，不代表判题和回写完成
- 自测通过不等于正式提交通过，隐藏用例和正式判题策略不同
- 排查 pending 和排查 judging 是两种问题：前者先看异步触发，后者先看判题服务和用例执行

六、文档同步点
- 新提交状态、赛事规则或管理能力变化：OJ 模块页、API 索引、必要时操作手册
- 新失败态或回归补测结果：验证记录与已知问题、发布前验证
- 积分回流或跨模块逻辑变化：OJ 模块页、积分模块页、覆盖矩阵

七、当前接手结论
- 现在能独立接：题目展示、普通列表问题、非核心运营入口、只读文档校准
- 现在仍需陪跑：正式判题主流程、go-judge 异常、赛事校验、首次 AC 奖励回流
- 下周优先补：pending/judging/system_error 三类问题的排查清单
```

### 这个样例为什么重要

这个样例最大的价值，是让你学会一句话：

**高风险模块不是不能接，而是不能在没分清“我能接哪一段”的时候乱接。**

你真正要学会的不是“把 OJ 全看完”，而是：

1. 哪些部分可以先独立接。
2. 哪些部分必须在陪跑下接。
3. 哪些验证没跑过就绝不能写成已验证。

## 三个样例放在一起，应该怎么看

对比这三份示例，你会发现一个规律：

| 模块 | 最怕漏什么 |
| --- | --- |
| 社区帖子 | 风控、渲染安全、后台可运营性 |
| 学习资产 | 状态分层、目标模块映射、审核与发布分流 |
| OJ 判题系统 | 外部依赖、异步回写、失败态与赛事约束 |

所以“接手模块”从来不是只背模板，而是：

1. 先判断模块是什么类型。
2. 再判断它最怕哪种错误。
3. 最后按类型补最小验证和接手结论。

## 你可以怎么自己练

如果你准备自己做一版练习，推荐顺序是：

1. 先按社区帖子做一份。
2. 再按学习资产做一份。
3. 最后再尝试 OJ 或 AI Runtime 这样的高风险模块。

更稳的原因很现实：

- 前两类更适合把“入口 -> 状态 -> 边界 -> 验证 -> 交接”这套动作练熟。
- 最后一类要求你已经能诚实处理“依赖没起、链路没跑完、不能乱下结论”。

## 最后的使用建议

如果你正在带新人，最省心的做法通常是：

1. 先让对方看 [模块接手升级路线](/guide/module-ownership-ladder)。
2. 再按 [模块接手检查清单](/guide/module-takeover-checklist) 自己写一版。
3. 最后拿本页的样例包对照补差距。

这样学习者不是“看懂一篇文档”，而是真的开始知道：

什么叫接手模块，什么又叫看起来接手了、实际上还没接住。
