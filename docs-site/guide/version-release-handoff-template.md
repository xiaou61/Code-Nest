# 版本公告与发版交接模板

这页不再讲“为什么要按 `git log` 写版本记录”，而是直接解决另一个更落地的问题：

**当你已经知道这版要发什么，怎么把它一次性写成 README 版本段、产品内版本墙文案，以及团队内部可继续接的发版交接记录。**

如果说 [按 Git Log 重写版本更新记录](/guide/git-log-release-notes) 解决的是“怎么找证据”，这页解决的就是“证据找完以后怎么交付”。

如果你不想只看抽象模板，而是想直接看 `v2.2.0`、`v2.1.2` 这种真实版本是怎么写出来的，可以直接继续看 [版本公告与交接实战样例](/guide/version-release-worked-examples)。

## 适合什么时候用

下面这几种情况，都可以直接套本页：

1. 要补 README 新版本段。
2. 要在 `/system/version` 里新建一条版本公告。
3. 要给 PR、群里同步或周报写一段发版交接。
4. 要把一个版本从“技术改动”翻译成“用户可感知变化”。

## 推荐使用顺序

最稳的顺序不是上来就写产品公告，而是：

1. 先按 [按 Git Log 重写版本更新记录](/guide/git-log-release-notes) 收集证据。
2. 再写这一版的“一句话主题”。
3. 先落团队内部交接单。
4. 再从交接单里裁剪出 README 段和产品内版本墙文案。

这样做的好处是：

- 内部版本记录更完整。
- 对外公告更简洁。
- 三套口径不容易互相打架。

## 模板 1：团队内部发版交接单

这份模板适合放在：

- PR 描述
- 发布群同步
- 周报 / 发版记录
- 版本归档页草稿

```md
## 版本交接：vX.Y.Z

一句话主题：

本次版本主要解决：

- 待补充
- 待补充
- 待补充

关键 Git 证据：

| 日期 | 提交 | 含义 |
| --- | --- | --- |
| 2026-05-20 | `abc1234` `feat: ...` | 这一版的核心落点 |

用户可感知变化：

- 待补充
- 待补充

管理端 / 运维 / 文档变化：

- 待补充
- 待补充

已完成验证：

- 待补充
- 待补充

未完成验证与原因：

- 待补充
- 待补充

需要同步的文档：

- [版本历史](/modules/version-history)
- [当前路线图](/roadmap/v2.2.1-docs-plan)
- 其他：

后续补测 / 跟进项：

- 待补充
- 待补充
```

### 这份模板最重要的不是“好看”，而是四件事不能缺

1. 这版到底在解决什么。
2. 哪些结论是真的有 Git 证据。
3. 哪些验证真的跑过。
4. 哪些还没跑，为什么没跑。

## 模板 2：README 版本段

README 适合写得更“主题化”，不要塞太多内部流程和验证细节。

可以直接套下面格式：

```md
## 🧩 vX.Y.Z 一句话版本名

`vX.Y.Z` 聚焦 **一句话主题**。

### 本次版本完成

- **能力 1**：用户或维护者能直接感知的变化。
- **能力 2**：尽量写结果，而不是只写改了哪个类。
- **能力 3**：必要时补一句适用范围或影响面。

### 这一版更适合怎么理解

- 如果它是大版本：写清主线升级。
- 如果它是补强批次：写清是在把哪条链路补稳。
- 如果它是文档或治理版本：写清是在提升什么维护能力。
```

### README 版本段怎么裁剪

内部交接单里常常会写很多“维护者需要知道”的信息，但 README 不需要全带过去。

README 一般保留：

- 版本主题
- 2 到 5 条最关键变化
- 对外可以公开说的能力或体验结果

README 一般不必保留：

- 很细的验证边界
- 过多 commit 细节
- 团队内部返工或补测安排

## 模板 3：产品内版本墙文案

产品内 `/version-history` 和 README 也不完全一样。它更像“用户在站内看到的正式公告”，可以更口语化一点，但仍然要准确。

```md
版本号：vX.Y.Z
标题：一句话更新标题
更新类型：重大更新 / 功能更新 / 修复更新 / 其他
发布时间：YYYY-MM-DD HH:mm:ss

简要描述：
这次更新主要围绕一句话主题展开，重点补了什么、优化了什么、修复了什么。

重点更新：

1. 待补充
2. 待补充
3. 待补充

修复与优化：

1. 待补充
2. 待补充

升级提醒：

1. 如果没有特别事项，可以写“本次更新无需用户额外操作”。
2. 如果有迁移、清缓存或重新登录要求，要单独写清楚。

延伸链接：

- PRD / 文档页 / 详细说明
```

### 产品内版本墙文案最容易犯的错

| 错法 | 更稳的写法 |
| --- | --- |
| 直接复制 commit 说明 | 改写成用户看得懂的变化 |
| 只写“优化体验” | 说清哪类体验、在哪个页面或链路 |
| 完全不写修复项 | 至少保留用户会感知到的稳定性变化 |
| 把内部敏感信息写进去 | 只放公开信息和安全可披露内容 |

## 模板 4：文档站版本页补写单

如果你不仅要发产品公告，还要补开发者文档，可以直接补一份更偏维护者视角的版本页草稿：

```md
## vX.Y.Z

主题：

这一版更像：

- 大版本升级 / 稳定性补强 / 治理批次 / 文档批次

关键提交：

| 日期 | 提交 | 含义 |
| --- | --- | --- |
|  |  |  |

为什么会有这一版：

- 待补充
- 待补充

这版最重要的不是：

- 待补充

这版最重要的是：

- 待补充
- 待补充

对应资料：

- README 段落
- 路线图
- 审计记录 / 模块页 / 计划页
```

这份模板很适合先写成草稿，再回填到 [版本历史](/modules/version-history)。

## 一套最省事的实际用法

如果你不想每次写三遍，最省事的方法是：

1. 先写“团队内部发版交接单”完整版。
2. 从里面裁 2 到 5 条，写到 README。
3. 再从 README 版本段改成更口语化的产品内版本墙文案。

也就是说：

- **交接单最全**
- **README 最概括**
- **产品公告最面向用户**

## 一份最小可用示例

下面给一个偏 `v2.2.x` 风格的例子：

```md
## 版本交接：v2.2.x

一句话主题：
继续补强工程治理与文档交付闭环。

本次版本主要解决：

- 当前工作线和历史路线图混在一起，不利于继续维护。
- 版本说明写法过度依赖口头记忆，缺少 Git 证据。
- 发版公告、README 和团队交接之间缺少统一模板。

用户可感知变化：

- 文档站当前维护主线更清晰。
- 版本说明和更新记录的入口更集中。

管理端 / 运维 / 文档变化：

- 新增 `v2.2.1` 文档计划。
- 新增 Git Log 版版本记录写法手册。
- 新增版本公告与发版交接模板。
```

如果你想看更完整、分成三层交付物的真实例子，继续看 [版本公告与交接实战样例](/guide/version-release-worked-examples)。

## 和哪些页面配合最好

本页最适合和下面几页一起用：

- [版本历史](/modules/version-history)
- [按 Git Log 重写版本更新记录](/guide/git-log-release-notes)
- [发布前验证](/guide/release-verification)
- [文档维护规范](/guide/documentation-maintenance)
- [v2.2.1 文档计划](/roadmap/v2.2.1-docs-plan)

它们的职责可以简单理解成：

- **版本历史**：版本线本身
- **Git Log 写法页**：怎么找证据
- **本页**：怎么把证据写成交付物
- **发布前验证**：这版到底验证了什么
- **路线图**：这批文档工作当前补到哪里

## 最后一个判断标准

如果你写完后，能同时满足下面三点，这份版本记录通常就够稳了：

1. 用户看得懂这版更新了什么。
2. 维护者看得懂这版为什么这样写。
3. 下一个接手的人看得懂哪些已经验证、哪些还没验证。
