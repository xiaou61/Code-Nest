# 模块接手检查清单

这页是 [模块接手升级路线](/guide/module-ownership-ladder) 的落地版。

如果上一页解决的是：

- 你现在处在哪个成长阶段
- 什么时候适合继续升难度

那么这页解决的就是更实际的问题：

当你真的准备开始接手一个模块时，第一天、第一周到底该盘什么，才不至于“看起来接手了，实际上没接住”。

## 这页适合什么时候看

最适合下面几种场景：

1. 你已经做完首单，正在从 `S2` 往 `S3/S4` 走。
2. 团队准备把一个普通模块交给你持续跟。
3. 你要带新人接模块，想给他一张能直接照着执行的清单。

如果你还停留在“第一个真实任务还没跑顺”，建议先回到：

- [首个真实任务接入指南](/guide/first-real-task)
- [首单任务题库](/guide/first-task-backlog)
- [模块接手升级路线](/guide/module-ownership-ladder)

先把首单和升级节奏跑稳，再用这页会更顺。

## 怎么使用这页

推荐把这页分三次用：

1. **接手前看一遍**：先判断这个模块是不是现在就适合你接。
2. **接手当天照着盘**：把入口、接口、表、状态、依赖和验证先拉齐。
3. **接手一周后回看**：检查还有哪些“看过但没真正验证”的盲区。

这页不是为了让你写一大堆材料，而是为了防止两个常见问题：

1. 只接了页面，不知道后端、表和依赖在哪。
2. 只接了 happy path，不知道失败态和高风险边界在哪。

## 一页式总清单

先给你最核心的 8 项：

| 必盘项 | 最低要求 |
| --- | --- |
| 入口地图 | 知道用户端、管理端、后端模块分别在哪 |
| 接口与表 | 知道核心接口前缀、核心表、关键状态字段 |
| 状态与边界 | 知道这个模块最关键的状态流转和高风险边界 |
| 依赖能力 | 知道它依赖鉴权、文件、通知、积分、敏感词、AI、OJ、WebSocket 里的哪些 |
| 最低验证 | 能写出至少 1 条成功路径和 1 条失败态验证 |
| 已知问题 | 知道这个模块现在最容易出错的 2 到 3 个点 |
| 文档同步点 | 知道改它时要同步哪些索引、模块页、手册、验证记录 |
| 接手结论 | 能写清楚“我现在能接什么、还不能接什么” |

如果这 8 项里有 3 项以上答不上来，通常说明现在还不能算真正接住了这个模块。

## 接手前 30 分钟，先判断值不值得接

不是每个模块都应该在同一阶段接手。先用下面 6 个问题做快速判断：

1. 这个模块主要是普通 CRUD，还是带状态机、外部依赖、实时链路？
2. 它会不会碰鉴权、文件、积分、AI、OJ、聊天或部署配置？
3. 现在有没有现成模块文档、操作手册、验证记录可参考？
4. 我能不能在半天内先跑出一条最小验证路径？
5. 如果改坏，是否容易定位和回滚？
6. 团队是要我“持续接手”，还是只做一次性任务？

更稳的建议是：

| 判断结果 | 建议 |
| --- | --- |
| 文档全、闭环短、依赖少 | 可以作为第一次模块接手对象 |
| 依赖多但边界清晰 | 适合在陪跑下接手 |
| 高风险边界多、验证成本高 | 先不要独立接手，先做观察和补测 |

## 接手当天，至少把这 6 类信息盘出来

### 1. 盘入口

最低要拉齐这几项：

| 入口类型 | 你至少要知道什么 |
| --- | --- |
| 用户端入口 | 从哪个路由进入，入口页、详情页、编辑页分别在哪 |
| 管理端入口 | 管理员从哪里看、配、审、查这个模块 |
| 后端入口 | 属于哪个 `xiaou-*` 模块，核心 Controller 在哪 |
| 文档入口 | 模块页、操作手册、验证记录、索引页分别在哪 |

如果你只能说出“这个模块大概在社区那边”，还不算真正完成入口盘点。

### 2. 盘接口和表

接模块时，别只知道页面行为，至少还要知道：

1. 核心接口前缀是什么。
2. 哪几个写接口最关键。
3. 至少一张核心表和一个关键字段。
4. 有没有明细表、统计表、历史表或日志表。

推荐直接写成这种格式：

```text
模块：
核心接口前缀：
关键写接口：
核心表：
关键状态字段：
```

### 3. 盘状态和边界

每个模块至少找出下面两类东西：

| 类型 | 例子 |
| --- | --- |
| 主状态流转 | 草稿 -> 发布、待审核 -> 通过/驳回、待判题 -> AC/WA |
| 高风险边界 | 权限、文件公开、重复发奖、敏感词、外部依赖失败 |

你不一定要第一天就把所有状态讲得很细，但必须知道哪几个状态一旦改错，影响会很大。

### 4. 盘依赖能力

Code Nest 的模块很少是完全孤立的。接手时建议按这张表过一遍：

| 依赖能力 | 要问自己什么 |
| --- | --- |
| 鉴权 | 普通用户、管理员、匿名访问分别怎样进来 |
| 文件存储 | 有没有上传、预览、下载、公开/私有访问 |
| 通知 | 成功/失败/审核结果是否会通知用户 |
| 积分 | 是否有奖励、扣减、抽奖、回流展示 |
| 敏感词 | 标题、正文、评论、简介是否需要拦截 |
| WebSocket | 是否有实时状态、ACK、重连、在线态 |
| AI / OJ | 是否依赖外部服务、结构化输出、判题回写 |

只要碰到其中任意一项，验证范围就不能只停在“页面能打开”。

### 5. 盘最低验证

接手当天不用把所有场景全跑一遍，但至少要先写出：

1. 一条成功路径。
2. 一条失败态或边界态。
3. 一条当前无法验证但必须记录的依赖项。

推荐格式：

```text
成功路径：
失败态：
当前未验证：
未验证原因：
```

### 6. 盘文档同步点

你准备接住一个模块，就要知道改它时哪些文档会一起动。

| 变更类型 | 常见同步点 |
| --- | --- |
| 新页面或入口变化 | 前端路由索引、模块页、操作手册 |
| 新接口或前缀变化 | API 路由索引、模块页 |
| 新表或字段变化 | 数据表索引、数据库字段指南、模块页 |
| 新失败态或补测结果 | 验证记录与已知问题、发布前验证 |
| 新高风险链路 | 模块页、覆盖矩阵、必要时路线图 |

如果你接模块时完全不知道这些同步点，后面就很容易只留代码不留认知。

## 第一周结束前，至少补齐这 4 项

第一天盘清楚入口和边界还不够，第一周结束前更建议补齐下面四件事：

1. **一份模块现状总结**  
   写清当前模块做什么、最关键的入口和最危险的边界。
2. **一份已知问题清单**  
   不求全，但至少列出最容易踩的 2 到 3 个坑。
3. **一份最小回归清单**  
   下次改这个模块时，先跑哪几个构建、页面或接口。
4. **一份接手结论**  
   明确“我现在能独立接什么，还需要谁陪跑什么”。

这四项一补，你才算从“了解模块”真正走到“开始能维护模块”。

## 不同类型模块，要额外盯什么

不是所有模块都看一样的重点。下面这张表比较实用：

| 模块类型 | 额外要盯的重点 |
| --- | --- |
| 内容类：社区、博客、动态 | 敏感词、渲染安全、互动计数、后台可运营性 |
| 学习类：题库、闪卡、计划、小组 | 状态记录、统计回流、用户长期数据口径 |
| 后台写操作类：审核、配置、管理页 | 二次确认、操作日志、列表回流、权限边界 |
| 奖励/状态机类：积分、学习资产、通知 | 幂等、回流、重复触发、异常补偿 |
| 外部依赖类：AI、OJ、聊天 | 外部服务可用性、失败态、依赖未起时的验证结论 |
| 基础设施类：鉴权、文件、部署 | 全局影响面、回滚方式、绝不能凭感觉修改 |

如果你现在第一次真正接手模块，强烈建议先从前 3 类开始，而不是一上来碰最后 3 类。

## 带教或负责人怎么验收“这个模块算接住了吗”

推荐只问 5 个问题，基本就够了：

1. 他能不能在 3 分钟内说清这个模块从哪里进、落到哪里？
2. 他知不知道这个模块最怕改坏什么？
3. 他写不写得出最小验证，而不是只会说“我去测一下”？
4. 他知不知道改这个模块时要同步哪些文档？
5. 他能不能明确说出“我现在还不能独立接哪类任务”？

如果这 5 个问题里有 2 个以上答不稳，通常说明还不该把这个模块完全放手交出去。

## 最后的判断口径

“会改这个模块里的一个点”和“能接住这个模块”之间，差的通常不是代码能力，而是下面三件事：

1. 能不能持续定位，而不是只会定位一次。
2. 能不能持续验证，而不是只会跑一次 happy path。
3. 能不能持续交接，而不是只在自己脑子里知道风险。

如果你现在已经在 [模块接手升级路线](/guide/module-ownership-ladder) 里走到 `S3` 附近，这页就可以当成你的模块接手首周执行清单。

如果你已经准备开始动手写接手记录，下一步直接对照 [模块接手样例包](/guide/module-takeover-examples) 会最省力。
