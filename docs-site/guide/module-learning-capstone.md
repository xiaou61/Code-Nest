# 模块结业案例

这页给“按模块学习”补最后一块拼图。

[按模块学习路线](/guide/module-learning-paths) 告诉你应该怎么学，  
[模块学习任务模板](/guide/module-learning-workbook) 告诉你每一站应该交什么作业，  
而这页告诉你一份合格的跨模块结业案例，大概应该长什么样。

它适合下面三种场景：

1. 你已经按模块读完大半内容，想做一次综合演练。
2. 你在带新人，希望给对方一份“参考答案级别”的示范。
3. 你准备开始真实开发，想先练一次跨模块影响面判断和验证记录。

## 这页怎么用

最推荐的用法不是照抄答案，而是先自己按 [模块学习任务模板](/guide/module-learning-workbook) 做一遍，再回来看差距。

建议顺序：

1. 自己先选一个案例做完整拆解。
2. 写出模块画像卡、链路追踪单、验证记录单、文档回写清单。
3. 再对照本页，看是否漏了平台依赖、失败态、验证范围或文档同步项。

## 结业案例 1：OJ 首次 AC 到积分回流

这个案例适合训练四件事：

1. 状态机阅读。
2. 外部依赖识别。
3. 结果回写和奖励回流。
4. 单模块改动如何扩展到跨模块验证。

### 案例背景

用户在 OJ 详情页提交代码，判题通过后，如果满足“首次 AC”条件，会给用户发积分奖励，并在用户端积分页和流水页回流。

### 这次演练要覆盖的模块

| 类型 | 模块 |
| --- | --- |
| 主业务 | [OJ 判题系统](/modules/oj) |
| 横切能力 | [积分与抽奖](/modules/points) |
| 支撑参考 | [API 调用示例](/reference/api-examples)、[测试与回归](/guide/testing-regression)、[模块依赖地图](/reference/module-dependencies) |

### 示例模块画像卡

```text
模块组：OJ 与积分反馈闭环
本轮目标：弄清一次 OJ 提交从创建到首次 AC 奖励回流的完整链路
用户端入口：/oj/problem/:id、/oj/my-submissions、/points
管理端入口：/oj/*、/points/*
后端模块：xiaou-oj、xiaou-points
核心接口前缀：/oj/run、/oj/test、/oj/submit、/user/points/checkin（参考积分模块风格）
核心表或关键字段：oj_submission、oj_contest_participant、user_points_detail、首次 AC 相关状态字段
依赖的平台能力：鉴权、积分、可能的日志记录
高风险边界：go-judge 可用性、判题状态卡住、首次 AC 重复奖励
```

### 示例链路追踪单

```text
链路名称：OJ 首次 AC 奖励回流
起点动作（用户或管理员）：用户在 /oj/problem/:id 提交代码
中间处理（Controller/Service/外部依赖）：
1. OJ Controller 接收提交请求
2. Service 创建 submission 记录
3. 调用 go-judge 执行代码
4. 判题结果写回 submission
5. 如果满足首次 AC 规则，触发积分奖励逻辑
状态变化：
submission 从待处理 -> 判题中 -> AC/WA/CE/TLE 等结果态
落库结果：
oj_submission 更新结果字段，积分流水表新增奖励记录
回流结果（页面、通知、积分、统计、日志等）：
我的提交页和提交详情页展示结果；积分页展示余额和新增流水；榜单或统计可能回流
失败态：
go-judge 不可达、状态未写回、首次 AC 判断重复、积分流水成功但余额未同步
```

### 这份答案为什么算合格

它不是只说“判题成功会加分”，而是明确写出了：

- 结果回写和奖励回流是两段逻辑。
- go-judge 是外部依赖，不能只靠构建成功判断。
- 积分流水和积分余额要分开看。

### 最小验证清单示例

```text
本轮验证目标：确认一次 AC 提交能正确写回，并且首次 AC 奖励只发一次
运行命令：
1. 后端构建
2. docs-site 构建（如果只改文档则运行此项）
3. 真实提交 AC/WA 各一次（依赖 go-judge）
关键结果：
- AC 提交在详情页可见
- 我的提交记录更新
- 首次 AC 后积分余额增加
- 重复 AC 不重复发放相同奖励
已验证项：
- 文档阅读和链路拆解已完成
- 如果依赖已启动，则应补真实 AC/WA 样例
未验证项及原因：
- 未启动 go-judge 时，无法完成真实判题和奖励回流验证
下一步补测建议：
- 补一次重复 AC 行为验证
- 补一次失败态（编译错误或超时）验证
```

### 这类案例最容易漏掉什么

| 常见漏项 | 为什么危险 |
| --- | --- |
| 只看 submission 是否成功 | 积分奖励可能没有回流 |
| 只看积分余额 | 流水可能没落库，后续无法追查 |
| 只测 AC，不测失败态 | 真实线上更容易出错的是 WA/CE/TLE 和外部依赖异常 |
| 不看重复 AC | 最容易引入重复发奖 bug |

## 结业案例 2：模拟面试到学习资产审核发布

这个案例适合训练：

1. AI 结构化结果理解。
2. 业务链路跨模块沉淀。
3. 用户端和管理端双端联动。
4. “页面显示没问题，但业务可能已经错了”的识别能力。

### 案例背景

用户在模拟面试完成一轮会话后，系统生成报告和总结，并把内容沉淀为学习资产候选项，随后由管理端审核、通过、驳回或合并。

### 这次演练要覆盖的模块

| 类型 | 模块 |
| --- | --- |
| 主业务 | [模拟面试与求职作战台](/modules/mock-interview-job-battle) |
| 沉淀模块 | [学习资产](/modules/learning-assets) |
| 横切能力 | [AI Runtime](/modules/ai-runtime) |
| 支撑参考 | [AI Schema 与治理](/reference/ai-schemas)、[验证记录与已知问题](/manuals/verified-scenarios) |

### 示例模块画像卡

```text
模块组：AI 面试、求职作战台与学习资产
本轮目标：弄清面试会话、报告生成、学习资产候选沉淀和后台审核之间的完整关系
用户端入口：/mock-interview、/mock-interview/session、/mock-interview/report、/learning-assets
管理端入口：/learning-assets/review、/system/ai-config、/system/ai-governance
后端模块：xiaou-mock-interview、xiaou-ai、xiaou-learning-asset
核心接口前缀：/user/mock-interview/*、学习资产审核相关接口、后台 AI 配置接口
核心表或关键字段：mock_interview_session、mock_interview_qa、learning_asset_candidate、learning_asset_record、候选状态字段
依赖的平台能力：AI Runtime、鉴权、通知（若审核结果回流）、可能的日志与治理统计
高风险边界：Schema 结构错误、RAG/模型依赖异常、候选资产生成成功但审核链路断裂
```

### 示例链路追踪单

```text
链路名称：模拟面试 -> 报告 -> 学习资产候选 -> 后台审核
起点动作（用户或管理员）：用户创建模拟面试会话并完成作答
中间处理（Controller/Service/外部依赖）：
1. 用户端创建 session
2. 面试问答与 AI 评分结果写入 session/qa 相关数据
3. 结束会话后生成报告与总结
4. 学习资产模块接收沉淀结果，生成 candidate
5. 管理端对 candidate 执行通过、驳回或合并
状态变化：
面试会话状态推进；学习资产候选从待审核 -> 通过/驳回/合并
落库结果：
session、qa、candidate、publish_log 等相关表写入记录
回流结果（页面、通知、积分、统计、日志等）：
用户端可查看报告和资产结果；管理端可见候选清单；治理或统计面板可能更新
失败态：
模型调用成功但 Schema 字段缺失、候选资产未生成、后台审核成功但用户端未回流
```

### 这份答案为什么算合格

合格点不在于写得长，而在于它能明确分开下面三层：

1. AI 返回了什么。
2. 业务是否把这些结果正确沉淀成候选资产。
3. 后台审核以后，用户端和记录层是否都同步更新。

### 最小验证清单示例

```text
本轮验证目标：确认一次模拟面试可以形成报告，并生成可被后台审核的学习资产候选
运行命令：
1. 后端构建
2. 如依赖可用，执行一次真实会话与报告生成
3. 管理端打开学习资产审核页检查候选项
关键结果：
- session 创建成功
- 报告页可正常展示
- 候选资产在后台可见
- 审核操作后状态更新
已验证项：
- 文档链路已拆解
- 可在依赖可用时补真实双端验证
未验证项及原因：
- 未启动真实模型/RAG 依赖时，无法证明结构化输出与沉淀质量
下一步补测建议：
- 补一次字段不完整或模型失败态验证
- 补一次后台审核后用户端回流验证
```

### 这类案例最容易漏掉什么

| 常见漏项 | 为什么危险 |
| --- | --- |
| 只看报告页展示正常 | 报告可能只是前端拿到了一段文本，不代表候选资产正确生成 |
| 不看 Schema 字段 | 前端可能能渲染，但后端落库字段已经错位 |
| 不看后台审核链路 | 资产沉淀是跨模块闭环，不做审核验证就不完整 |
| 不写未验证项 | AI 依赖经常不在本地完整启动，必须诚实记录边界 |

## 结业案例 3：社区发帖到风控、通知与后台运营

这个案例适合训练：

1. 内容写入链路的标准拆法。
2. 平台能力在内容模块中的复用。
3. 用户端与管理端双向验证。

### 案例背景

用户在社区发布帖子并触发评论、收藏或互动，系统需要经过登录校验、敏感词检查、可能的内容渲染与摘要处理，同时支持后台对帖子、评论和用户状态进行运营管理。

### 这次演练要覆盖的模块

| 类型 | 模块 |
| --- | --- |
| 主业务 | [社区帖子](/modules/community) |
| 横切能力 | [敏感词风控](/modules/sensitive)、[通知中心](/modules/notification)、[前端渲染安全](/reference/frontend-rendering-security) |
| 支撑参考 | [社区与内容矩阵](/modules/community-content)、[模块依赖地图](/reference/module-dependencies) |

### 示例模块画像卡

```text
模块组：内容社区与创作矩阵
本轮目标：弄清一次社区帖子从发布到后台运营的完整闭环
用户端入口：/community、/community/create、帖子详情页
管理端入口：/community/posts、/community/comments、社区用户相关运营入口
后端模块：xiaou-community
核心接口前缀：社区发帖、评论、收藏、详情、管理端帖子/评论接口
核心表或关键字段：社区帖子表、评论表、互动计数字段、状态字段
依赖的平台能力：鉴权、敏感词、通知、AI 摘要、前端渲染安全
高风险边界：敏感词误判/漏判、Markdown 或 HTML 渲染风险、用户端成功但后台不可运营
```

### 示例链路追踪单

```text
链路名称：社区发帖 -> 风控/渲染 -> 互动 -> 后台运营
起点动作（用户或管理员）：用户在 /community/create 发布帖子
中间处理（Controller/Service/外部依赖）：
1. 登录态校验
2. 发帖内容写入前执行敏感词或内容校验
3. 发帖成功后进入详情页和列表页
4. 后续评论、点赞、收藏触发互动计数变化
5. 管理端对帖子、评论或用户状态执行运营动作
状态变化：
帖子状态、评论状态、互动计数字段更新
落库结果：
帖子和评论等相关表写入或更新
回流结果（页面、通知、积分、统计、日志等）：
用户端列表和详情回流；互动通知可能生成；后台列表可见与可管理
失败态：
敏感词拦截、渲染异常、评论树异常、后台统计不同步
```

### 这份答案为什么算合格

因为它没有把“内容发布”看成单点功能，而是明确写出了：

- 发布前有风控边界。
- 发布后有渲染边界。
- 发布完成还要能被互动、通知和后台运营接住。

### 最小验证清单示例

```text
本轮验证目标：确认一篇社区帖子可以完成发布、展示、互动和后台可见
运行命令：
1. 前端构建
2. 后端构建
3. 真实发布一篇帖子并打开详情
4. 管理端查看帖子和评论列表
关键结果：
- 帖子发布成功
- 列表页和详情页都可见
- 如有评论或收藏，互动数据能更新
- 管理端能检索到该内容
已验证项：
- 文档链路已拆解
- 可在本地环境具备时补真实发布链路
未验证项及原因：
- 未补充恶意内容样例时，不能证明风控与渲染边界都安全
下一步补测建议：
- 补一次敏感词命中验证
- 补一次 Markdown/HTML 边界渲染验证
```

### 这类案例最容易漏掉什么

| 常见漏项 | 为什么危险 |
| --- | --- |
| 只在用户端看见帖子就算完成 | 后台可能完全找不到这条内容 |
| 不测敏感词 | 内容类模块最容易在线上踩审核边界 |
| 不看渲染安全 | Markdown 或 HTML 内容可能引入 XSS 风险 |
| 只测发帖，不测评论和互动 | 很多统计和通知都在第二段链路里出错 |

## 做完这页以后，应该具备什么能力

如果你能自己独立完成上面任意两个案例，并且写出与本页质量接近的结论，基本就具备了下面这些能力：

| 能力 | 表现 |
| --- | --- |
| 跨模块思维 | 不会把业务模块和平台能力割裂看待 |
| 验证意识 | 会把成功路径、失败态和外部依赖分开记录 |
| 交接能力 | 能给别人留下可继续执行的模块画像、链路追踪和验证记录 |
| 文档维护意识 | 知道什么时候要同步索引、模块页、验证记录和路线图 |

## 下一步建议

做完结业案例后，最适合进入下面两个方向之一：

1. 按 [功能开发流程](/guide/feature-development) 做一次真实的小改动。
2. 按 [问题定位流程](/operations/diagnosis-flow) 练一次从现象到根因的排障闭环。

如果你带的是新人，建议顺序就是：

1. 先走 [按模块学习路线](/guide/module-learning-paths)。
2. 再做 [模块学习任务模板](/guide/module-learning-workbook)。
3. 最后完成本页任意两个结业案例。

这样学到最后，读者基本就不是“看过文档”，而是“已经能接模块了”。

如果你要把结业案例交给带教或团队评审，建议直接使用 [模块学习评审表](/guide/module-learning-rubric) 统一打分口径。
