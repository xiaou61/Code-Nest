# 模块值守与回归手册

这页是给已经开始“接手模块”的人准备的。

如果前面的页面解决的是：

- [模块接手升级路线](/guide/module-ownership-ladder)：什么时候开始接模块
- [模块接手检查清单](/guide/module-takeover-checklist)：第一天和第一周该盘什么
- [模块接手样例包](/guide/module-takeover-examples)：一份合格接手记录长什么样

那么这页解决的就是下一步：

当你已经开始持续接一个模块以后，怎么把它稳稳守住，而不是只会改几个点。

## 这页适合谁

最适合下面三类人：

1. 已经开始连续接某个模块需求、缺陷或文档维护的人。
2. 准备从 `S3` 走向 `S4`，开始承担模块 owner 候选责任的人。
3. 需要给新人做“模块值守交接”的带教或负责人。

如果你还在首单阶段，这页不是第一优先级。建议先走完：

- [首个真实任务接入指南](/guide/first-real-task)
- [模块接手升级路线](/guide/module-ownership-ladder)
- [模块接手检查清单](/guide/module-takeover-checklist)

## 先讲一个最容易被误解的点

“接手模块”不等于“以后这个模块所有问题都归你背”。

更准确地说，模块值守意味着你至少要开始承担下面四件事：

1. **持续定位**：出了问题，知道先去哪里看。
2. **持续回归**：改完之后，知道最少要跑哪一组验证。
3. **持续同步**：文档、索引、验证记录不会只改代码不留痕。
4. **持续交接**：你不在的时候，别人能接得上。

这四件事如果缺任意一项，你就还只是“做过模块任务”，不是“在守模块”。

## 值守视角下的四个时间尺度

守模块最好按四个时间尺度去看：

| 时间尺度 | 要做什么 | 典型产物 |
| --- | --- | --- |
| 每天 | 处理新增改动、已知问题、最低回归 | 当日验证结论、任务交接说明 |
| 每周 | 复盘高频问题、补回归清单、补文档缺口 | 周回顾、已知问题更新、回归补充 |
| 发版前 | 按影响面做组合验证 | 发版验证清单、依赖状态说明 |
| 临时交接时 | 把当前状态交给别人 | 交接记录、未完成项、风险说明 |

你不需要每次都写大文档，但要在这四个时间尺度里都有最小闭环。

## 每天要守什么

### 1. 先判断今天的改动属于哪一类

一天里最容易犯的错，是见一个问题修一个问题，但不先判断它属于什么类型。

推荐先分成下面几类：

| 改动类型 | 先注意什么 |
| --- | --- |
| 文案/样式/低风险前端 | 页面回流、空态、失败态、构建 |
| 普通 CRUD / 管理端写操作 | 权限、列表回流、表字段、操作日志 |
| 内容类模块改动 | 敏感词、渲染安全、后台可运营性 |
| 状态机/奖励/审核流 | 状态推进、幂等、通知、异常补偿 |
| 外部依赖类 | 依赖在线状态、失败态、未验证边界 |

不先分类型，后面的回归范围很容易选错。

### 2. 每次改动后，都至少留下一个最小结论

日常值守最实用的，不是写长篇说明，而是写清这四项：

```text
本次改动：
已验证：
未验证：
未验证原因：
```

这四行很短，但极其关键。因为你第二天再看，或者别人来接，就不会把“没跑过”误看成“已经没问题了”。

### 3. 每天至少问自己这 5 个问题

1. 我今天改的是模块主链路，还是外围体验？
2. 这次有没有碰到权限、文件、积分、通知、敏感词、AI、OJ、WebSocket 这些横切能力？
3. 我今天跑的是构建验证，还是业务验证？
4. 有没有新增一个“以后还会踩”的已知问题？
5. 我今天的结论，别人明天能不能接着用？

如果这 5 个问题里有 2 个以上说不清，通常说明当天的值守结论还不够可靠。

## 每周至少补一次这三类东西

守模块如果只顾当天的任务，通常一周后就会开始失控。更稳的方式，是每周至少补一次下面三类东西。

### 1. 高频问题清单

把本周出现过两次以上，或者虽然只出现一次但风险大的问题记下来。

推荐格式：

| 问题 | 现象 | 初步判断 | 当前状态 |
| --- | --- | --- | --- |
| 列表回流不及时 | 改完后台状态，前端列表不刷新 | 可能是缓存或聚合统计回写 | 待补测 |

这样下次再遇到，不会又从零开始。

### 2. 最小回归清单

每个模块都应该慢慢长出自己的“最小回归组”，而不是每次都重新想。

例如：

| 模块类型 | 最小回归组 |
| --- | --- |
| 社区/博客/动态 | 列表、详情、写入、后台可见、敏感词失败态 |
| 学习资产/审核流 | 记录状态、候选状态、发布/审核、目标模块回流、通知 |
| OJ / AI / 聊天 | 构建、依赖可用性、成功路径、至少一条失败态 |

这一点和 [测试与回归](/guide/testing-regression) 的关系是：

- 那一页告诉你“按类型怎么选验证层级”
- 这一页告诉你“作为模块 owner，怎么把它沉淀成固定回归组”

### 3. 文档缺口清单

一周下来，你几乎一定会发现某些东西“代码里有，但文档里不好找”。

建议把它们分成三档：

| 类型 | 说明 |
| --- | --- |
| 必须补 | 不补就影响接手或回归判断 |
| 可以后补 | 不影响当前值守，但补了更顺 |
| 暂不补 | 当前收益不高 |

这个动作很像“给未来的自己扫路”，但它的收益非常高。

## 发版前，模块 owner 至少要做到什么

发版前不要只看通用清单。作为模块 owner，你还要知道“我这个模块最怕发坏什么”。

### 建议的发版前四步

1. 先看这次改动是不是碰到了模块主链路。
2. 再看有没有碰横切能力或外部依赖。
3. 再把模块自己的最小回归组跑一遍。
4. 最后把依赖状态和未验证边界写清楚。

### 一个好用的模块发版口径

```text
模块：
本次变更范围：
模块主链路是否受影响：
横切能力是否受影响：
已跑回归：
未跑回归：
依赖状态：
发版风险：
```

如果你能稳定写出这 8 项，发版前的讨论会轻松很多。

### 哪些情况必须提高警惕

| 情况 | 为什么危险 |
| --- | --- |
| 只改了一个按钮，但其实带写操作 | 往往会误伤权限、状态回流或通知 |
| 接口返回 200 就算通过 | 很多模块的问题不在接口层，而在状态和回流层 |
| 外部依赖没起但仍写“已验证” | 后面的人会基于错误前提做判断 |
| 改了文档但不改验证记录 | 下次回归时会误以为已有结论 |

## 真正出现线上或联调问题时，先怎么收口

模块值守最怕慌。越是问题来了，越要先把问题收口，而不是一股脑到处试。

### 推荐的四步法

1. **先定范围**：是页面问题、接口问题、数据问题、依赖问题，还是权限问题？
2. **再定链路位置**：卡在入口、处理中间层、状态回写，还是结果回流？
3. **再定证据**：日志、Network、表字段、依赖健康状态分别是什么？
4. **最后才定修复动作**：别反过来先改。

这一步和 [问题定位流程](/operations/diagnosis-flow) 是一套逻辑，但从 owner 角度更强调：

你不只是定位出问题，还要把“这个问题以后怎么更快定位”顺手留下来。

## 临时交接时，最少要写什么

模块值守很现实的一点是：你总会有不在的时候。

临时交接如果写得不好，别人最容易踩坑。

推荐最小交接模板：

```text
【模块临时交接】
模块：
当前在做：
当前已完成：
当前未完成：
当前风险：
已验证：
未验证：
下一步建议：
相关文档/入口：
```

这个模板看起来普通，但它比“我大概已经改好了，你再看看”强很多。

## 什么时候说明你已经开始像 owner 了

这通常不是因为你接了最难的需求，而是因为你开始稳定表现出下面这些特征：

1. 看到问题会先判断影响面，不会立刻乱改。
2. 改完会主动补最小回归，不靠别人提醒。
3. 会主动把新问题沉淀进已知问题或验证记录。
4. 会把能独立接和仍需陪跑的边界说清楚。
5. 别人来接你手里的模块时，不需要重新从零摸索。

如果这五件事已经开始稳定出现，你基本就在从“能接任务”真正走向“能守模块”。

## 一份可以直接复制的值守周报模板

```text
【模块值守周报】
模块名称：
本周处理事项：
- 
- 

本周新增已知问题：
- 

本周补充回归项：
- 

本周文档同步：
- 

当前最大风险：
- 

下周优先事项：
- 
```

这个模板不要求长，但要求真。尤其“当前最大风险”这一行，往往最能体现 owner 意识。

## 和现有页面怎么配合

最后给一个最顺的配套顺序：

1. 用 [模块接手升级路线](/guide/module-ownership-ladder) 判断现在是不是该开始承担模块责任。
2. 用 [模块接手检查清单](/guide/module-takeover-checklist) 完成第一天和第一周接手盘点。
3. 用 [模块接手样例包](/guide/module-takeover-examples) 对照不同类型模块的写法。
4. 用本页把日常值守、每周回顾、发版前回归和临时交接跑起来。
5. 再回到 [测试与回归](/guide/testing-regression)、[发布前验证](/guide/release-verification) 和 [验证记录与已知问题](/manuals/verified-scenarios) 沉淀证据。

这样你守模块就不是“靠记性和经验”，而是逐渐形成一套可复用的方法。

如果你已经把模块自己的日常节奏建立起来，下一步就把最低验证范围固定到 [模块最小回归矩阵](/reference/module-regression-matrix) 这种查表口径里。这样不管是你自己、带教对象还是临时接手的人，都能更快知道“改完这个模块至少先回归什么”。
