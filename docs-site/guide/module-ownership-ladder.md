# 模块接手升级路线

这页是给已经跑完 onboarding 前半程的人准备的。

如果说：

- [首个真实任务接入指南](/guide/first-real-task) 解决的是“第一单怎么接”，
- [首单任务题库](/guide/first-task-backlog) 解决的是“第一单具体接什么”，

那么这页解决的就是下一步：

做完首单以后，怎么一步步从“能做任务”，成长到“能稳定接手一个模块”。

它特别适合三类人：

1. 学习者本人，想知道自己现在处在哪一档、下一步该补什么。
2. 带教同学，想更稳地决定什么时候该升难度、什么时候该继续保护。
3. 技术负责人，想把“能不能接手模块”从感觉判断变成有证据的判断。

## 这页怎么用

最推荐的顺序是：

1. 先用 [模块学习评审表](/guide/module-learning-rubric) 判断当前作业质量。
2. 再用 [首个真实任务接入指南](/guide/first-real-task) 和 [首单任务题库](/guide/first-task-backlog) 选当前合适的任务层级。
3. 每完成一批任务，就回到这页判断是否可以进入下一阶段。

这页不要求你“每一档都待很久”，但要求每一次升级都有证据。

## 六个成长阶段

先给一个全局图：

| 阶段 | 说明 | 核心目标 |
| --- | --- | --- |
| S0 观察期 | 还在读文档、跟着别人跑验证 | 先学会定位、复述链路、诚实记录未验证项 |
| S1 首单执行 | 已开始做第一单 | 先把一个低风险闭环跑完整 |
| S2 连续低风险交付 | 能连续做 2 到 3 个低风险任务 | 从“做完一次”变成“稳定做完” |
| S3 中风险链路接手 | 能接一个普通业务模块的小闭环 | 开始独立判断影响面、回归范围和文档同步 |
| S4 模块 Owner 候选 | 能持续处理一个模块的日常需求、缺陷与文档 | 不只会做任务，还能主动守住模块边界 |
| S5 稳定 Owner / 高风险值守 | 能接高风险链路或承担模块值守 | 能处理复杂回归、灰区边界和跨模块协同 |

真正有用的，不是“名字听起来升级了”，而是你是否真的能承担下一档的责任。

## 每一阶段到底能接什么

### S0 观察期

这一阶段还不算真正接模块，更像“准备接模块”。

| 项 | 要求 |
| --- | --- |
| 适合做什么 | 读文档、做 [模块学习任务模板](/guide/module-learning-workbook)、补只读校准、整理验证记录 |
| 暂时不要做什么 | 改业务逻辑、碰状态机、改权限或高风险写操作 |
| 必须拿出的证据 | 至少 1 份模块画像卡 + 1 份链路追踪单 + 1 份验证记录单 |
| 验证要求 | 至少能跑文档站构建，能把已验证/未验证分开写 |
| 升级信号 | 能说清一个模块的入口、接口前缀、核心表和一个失败态 |

S0 的重点不是“看了多少”，而是“有没有从阅读进入可复用的交付物”。

### S1 首单执行

这是从学习闭环切到真实交付闭环的第一步。

| 项 | 要求 |
| --- | --- |
| 适合做什么 | `L0` 首单、低风险文档同步、低风险前端小修、索引校准 |
| 暂时不要做什么 | 鉴权、文件、积分、AI、OJ、WebSocket、部署类改动 |
| 必须拿出的证据 | 一份完整首单记录：边界、定位、风险、最低验证、文档同步 |
| 验证要求 | 至少构建通过，并有 1 条可信业务烟测或明确未验证原因 |
| 升级信号 | 能把一单独立做完，且交接记录别人能继续用 |

S1 的目标不是追求难度，而是第一次把“任务 -> 修改 -> 验证 -> 交接”完整走通。

### S2 连续低风险交付

如果说 S1 证明你“做成过一次”，那 S2 要证明你“不是碰巧做成一次”。

| 项 | 要求 |
| --- | --- |
| 适合做什么 | 连续完成 `2-3` 个 `L0/L1` 任务，开始接一个普通模块里的小写操作 |
| 暂时不要做什么 | 横切能力核心规则修改、跨多个高风险模块联动改动 |
| 必须拿出的证据 | 连续任务记录 + 至少 2 次评审结果 + 文档同步闭环 |
| 验证要求 | 能自己列出最低验证，不再只等别人告诉你跑什么 |
| 升级信号 | 能稳定区分“已验证 / 未验证 / 依赖未起 / 风险待补测” |

S2 最核心的变化是：你开始有“交付稳定性”，而不只是“完成了一次任务”。

### S3 中风险链路接手

到了这一步，才开始接近“能接手一个普通模块的小闭环”。

| 项 | 要求 |
| --- | --- |
| 适合做什么 | 普通 CRUD、小范围审核流、小范围状态流转、用户端 + 管理端的联动小改 |
| 暂时不要做什么 | 改登录域、文件公开策略、积分规则、AI Schema、OJ 判题主流程 |
| 必须拿出的证据 | 至少 1 个 `L2` 级任务 + 影响面评估 + 回归范围说明 |
| 验证要求 | 除了构建，要能覆盖至少 1 条成功路径和 1 条失败态 |
| 升级信号 | 能独立写出“为什么这次只影响 A，不应该误伤 B/C” |

S3 的门槛不是代码量，而是你开始能控制影响面。

### S4 模块 Owner 候选

这一档才是真正意义上的“准备接手模块”。

| 项 | 要求 |
| --- | --- |
| 适合做什么 | 连续处理同一模块的需求、缺陷、文档同步、回归补测、已知问题跟踪 |
| 暂时不要做什么 | 在没有陪跑的情况下，独自改高风险横切规则 |
| 必须拿出的证据 | 同一模块连续 `2-4` 次有效交付 + 1 份模块维护视角总结 |
| 验证要求 | 能按影响面设计回归，而不是每次只测手头那个点 |
| 升级信号 | 遇到新任务时，会主动先补模块页、验证记录、操作手册或索引同步点 |

S4 的关键变化是：你开始从“做任务的人”变成“守模块的人”。

### S5 稳定 Owner / 高风险值守

这一步不是人人都必须马上到，但它代表你已经能承担复杂模块或高风险边界。

| 项 | 要求 |
| --- | --- |
| 适合做什么 | 高风险链路陪跑、复杂回归设计、跨模块协同、模块值守、线上问题收口 |
| 暂时不要做什么 | 在缺少上线验证条件时，凭感觉改核心策略 |
| 必须拿出的证据 | 多次跨模块交付、复杂失败态记录、风险决策与回滚方案 |
| 验证要求 | 要能覆盖成功路径、失败态、依赖异常和回滚可行性 |
| 升级信号 | 团队已经愿意把一个高风险模块或值守窗口交给你负责 |

S5 的本质不是“更忙”，而是“团队对你的边界判断和交接质量有稳定信任”。

## 每一档升级时，至少要看什么证据

为了避免“感觉差不多可以升了”，推荐用下面的证据口径：

| 从哪一档升到哪一档 | 最低建议证据 |
| --- | --- |
| S0 -> S1 | 1 份模块作业达到 80 分以上，能独立复述一个模块闭环 |
| S1 -> S2 | 完成 1 个首单，且首单交付记录完整、可复用 |
| S2 -> S3 | 连续完成 `2` 个 `L0/L1` 任务，至少 `1` 次评审 85 分以上 |
| S3 -> S4 | 完成 `1` 个 `L2` 任务，且能独立写影响面与文档同步清单 |
| S4 -> S5 | 同一模块连续 `2-4` 次交付稳定，能主动设计回归与回滚 |

如果你们团队已经在用 [带教交付模板包](/guide/onboarding-deliverable-templates)，这些证据完全可以直接从日报、评审记录和结业结论里提取，不需要另起一套表。

## 哪些信号说明现在不该升难度

升级不是只看完成数量，也要看是否出现了明显回退信号。

常见的“先别升”信号有这些：

1. 连续两次交付都没有把已验证和未验证分开写。
2. 仍然经常把“接口 200”当成“业务正确”。
3. 改完只会说“应该没问题”，说不清影响面。
4. 明明碰到了文件、权限、积分、AI、OJ、聊天等高风险边界，却没有显式标出来。
5. 每次都只顾改代码，不同步模块页、索引页、验证记录。
6. 返工后仍然解决不了同一类问题，例如定位总是泛、验证总是虚。

出现这些情况时，更好的做法通常不是继续加难度，而是回到：

- [模块学习任务模板](/guide/module-learning-workbook)
- [模块学习评审表](/guide/module-learning-rubric)
- [首个真实任务接入指南](/guide/first-real-task)

先把短板补扎实，再升档会更稳。

## 按模块类型，推荐怎样升级更顺

不是所有模块都适合用同样顺序接手。更稳的路径一般是下面这样：

| 顺序 | 推荐模块类型 | 典型页面 |
| --- | --- | --- |
| 第 1 批 | 文档、索引、轻量前端体验、低风险运营页 | 文档站、列表筛选、空态、提示、索引页 |
| 第 2 批 | 普通内容与学习模块 | 社区、博客、动态、面试题库、闪卡、计划/小组 |
| 第 3 批 | 普通后台写操作与联动页 | 审核页、管理端配置、低风险 CRUD |
| 第 4 批 | 有状态回流的业务模块 | 学习资产、通知、积分展示、简历、代码工坊 |
| 第 5 批 | 高风险或强依赖模块 | 鉴权、文件存储、IM、积分规则、AI Runtime、OJ、部署运维 |

原因很简单：

1. 前几批适合练定位、验证和交接，不容易因为一个点改坏全站。
2. 中间几批开始练“用户端 + 管理端 + 后端 + 表结构”的完整闭环。
3. 最后一批才去碰权限、文件、外部服务、状态机和部署，因为这些地方验证成本最高。

## 带教同学怎么用这页分配任务

推荐把这页当成一个“发任务前的闸门”。

### 如果对方在 S0-S1

优先发：

1. 文档同步类。
2. 索引校准类。
3. 低风险前端体验小修。

不要发：

1. 需要读状态机的大闭环。
2. 会碰权限、文件、AI、OJ、聊天的任务。

### 如果对方在 S2-S3

优先发：

1. 普通模块的小写操作。
2. 低风险后台联动。
3. 需要补一条失败态验证的任务。

这时候最适合训练的是：影响面判断、最小回归设计、文档同步意识。

### 如果对方在 S4 以后

可以开始让对方承担：

1. 同一模块的连续需求。
2. 模块页和操作手册维护。
3. 已知问题收口和补测计划。
4. 带一个新人做低风险首单。

这时你评估的重点也该从“会不会做”切换成“能不能稳定守住模块边界”。

## 学习者本人怎么自查

如果你想快速判断自己现在大概在哪一档，先问自己这 8 个问题：

1. 我最近做的是一次性完成，还是已经连续稳定交付？
2. 我能不能在动手前写清边界、定位、风险和验证？
3. 我碰到失败态时，会不会主动记下来，而不是略过？
4. 我是否知道哪些地方绝对不能用“应该没问题”来替代验证？
5. 我改一个点时，能不能说明为什么不会误伤相邻模块？
6. 我是否会主动同步索引、模块页、操作手册或验证记录？
7. 我能不能独立给别人分配一个低风险任务，并说明为什么适合对方？
8. 团队现在信不信任我去守一个模块，而不只是做一张工单？

如果前 4 个问题还经常答不上来，通常还在 S1-S2；
如果前 6 个问题已经比较稳，通常开始接近 S3-S4；
如果 8 个问题大多都能给出具体例子，基本就进入模块接手阶段了。

这个模板不追求漂亮，追求一件事：下周再看时，大家能很快判断是不是该升档。

## 最后给一个现实一点的判断口径

什么时候可以说“这个人已经能接手一个模块”？

更稳的标准通常不是“他已经做过一个难任务”，而是下面四件事同时成立：

1. 连续交付稳定，不靠运气。
2. 风险边界诚实，不会把未知当已知。
3. 文档同步主动，不会只留代码不留说明。
4. 交接可复用，别人拿着记录就能继续往下做。

如果这四件事还没同时出现，继续积累任务证据通常比贸然升难度更值。

已经做完首单，准备从“会做一单”继续升级到“能稳稳接住模块”时，就把这页当成你接下来 2 到 4 周的路线图来用。

当你确认自己准备开始真正接一个模块时，下一步直接接 [模块接手检查清单](/guide/module-takeover-checklist)。

如果你想看不同难度模块的接手记录到底应该写成什么样，再接 [模块接手样例包](/guide/module-takeover-examples)。
