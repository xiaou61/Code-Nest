# 带教交付模板包

这页不是再讲“怎么学”或“怎么带”，而是把带教过程中最常用的交付物直接做成可复制模板。

如果说：

- [带教执行剧本](/guide/onboarding-execution-playbook) 解决的是节奏问题，
- [模块学习评审表](/guide/module-learning-rubric) 解决的是评审标准问题，

那么这页解决的就是最后一步：

每天到底发什么、评审记录怎么写、返工通知怎么发、结业结论怎么落。

## 适合什么时候用

| 场景 | 用哪一类模板 |
| --- | --- |
| 每日同步进度 | 日报模板 |
| 带教会议后记录反馈 | 评审记录模板 |
| 要求补充或返工 | 返工通知模板 |
| 判断是否可接真实任务 | 结业结论模板 |
| 团队内部统一口径 | 模板包全套一起用 |

## 使用建议

最稳的组合是：

1. 用 [带教执行剧本](/guide/onboarding-execution-playbook) 确定第几天要交什么。
2. 用本页模板直接写出交付物。
3. 用 [模块学习评审表](/guide/module-learning-rubric) 打分。
4. 把评审结论和返工动作继续回写到本页模板里。

这样下来，学习、评审、返工、结业会形成一个完整闭环。

这个模板适合学习者每天结束前提交，重点不是写很多，而是让带教同学快速看到：

### 带教视角怎么看这份日报

带教同学不用逐字改，只看 4 个问题：

1. 这份日报能不能看出学习者真的落到具体页面、模块、接口、表。
2. 已验证和未验证有没有分开。
3. 卡点是不是具体问题，而不是“还有点不懂”。
4. 明日计划是不是能执行。

### 这个模板适合怎么说话

建议反馈口径尽量具体：

- 不要说：`再看看源码`
- 更好的是：`这次少了核心表和状态字段，先只补这两项，不要整份重写`

这样反馈才会让人真的能动起来。

这个模板适合在“补充后通过”或“不通过”时发给学习者。重点不是施压，而是把返工范围压缩到最小，避免把人打回重做整篇。

### 返工通知的关键原则

| 原则 | 说明 |
| --- | --- |
| 返工范围越小越好 | 避免把人打回“全部重做” |
| 明确哪些不用改 | 减少焦虑，也减少重复劳动 |
| 返工完成标准必须可验证 | 不能只写“补完整一点” |

### 结论怎么选更稳

| 结论 | 适用情况 |
| --- | --- |
| 可接真实任务 | 至少 2 份模块作业和 1 份结业案例都比较扎实 |
| 可在保护下接任务 | 大方向没问题，但仍需师兄/师姐陪跑首个任务 |
| 暂不建议接任务 | 还停留在文档复述层或验证意识不足 |

## 推荐组合用法

### 组合 A：最轻量

适合时间很紧的带教：

1. 日报模板
2. 评审记录模板
3. 结业结论模板

### 组合 B：标准版

适合多数团队：

适合要沉淀带教体系的团队：

1. 日报模板
2. 评审记录模板
3. 返工通知模板
4. 结业结论模板
5. 带教周总结模板

## 模板包常见误用

| 误用 | 问题 | 更好的做法 |
| --- | --- | --- |
| 日报写成长篇散文 | 带教看不动，也看不出重点 | 用字段化模板，短而准 |
| 评审只写结论不写返工范围 | 学习者不知道怎么改 | 至少补“必须返工项”与“最小返工范围” |
| 返工通知过于抽象 | 容易让人陷入重复劳动 | 明确哪些不用改，哪些必须补 |
| 结业结论只写“还行” | 没法拿来分配真实任务 | 必须落到可接 / 保护下可接 / 暂不建议接 |

## 和现有页面怎么搭配

最后给你一条最顺手的使用顺序：

1. 用 [带教执行剧本](/guide/onboarding-execution-playbook) 排节奏。
2. 用 [模块学习任务模板](/guide/module-learning-workbook) 规定交付结构。
3. 用本页模板包记录每天的实际交付。
4. 用 [模块学习评审表](/guide/module-learning-rubric) 打分。
5. 用 [模块结业案例](/guide/module-learning-capstone) 做对照补强。

这样这套 onboarding 文档体系就不只是“看起来完整”，而是真的可以拿去执行。
