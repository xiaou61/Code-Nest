# 按 Git Log 重写版本更新记录

这页解决的是一个很具体的问题：

**当你觉得现有版本说明太像口号、太像拍脑袋总结时，怎么基于 `git log`、README 和真实代码证据，把版本记录重新写扎实。**

这不是只给文档维护者看的。只要你要做：

1. 版本公告整理。
2. README 更新日志补写。
3. 产品内 `/version-history` 文案重写。
4. 某个版本的回顾总结。

都可以直接按这页来。

## 什么时候应该用这套方法

下面这些情况，基本都不要再凭记忆写版本说明了：

| 场景 | 为什么要用 Git Log 写法 |
| --- | --- |
| 版本跨度大 | 口头记忆很容易漏掉真正关键的批次 |
| 一版不是单个提交，而是一串持续推进 | 需要靠提交序列还原版本主题 |
| README 有版本段，但 Git 没有清晰 release tag | 要把“版本语义”和“Git 锚点”拆开写清楚 |
| 用户希望按实际提交重写更新记录 | 说明现有版本说明已经不够可信或不够具体 |

## 先看原则

按 `git log` 重写版本记录，核心不是“把 commit 原样抄一遍”，而是做三层翻译：

1. **先找 Git 证据**：确认这个版本真实发生过哪些关键提交、分支合入或版本号切换。
2. **再提炼版本主题**：判断这些提交共同在解决什么问题。
3. **最后翻译成读者能理解的话**：区分“维护者视角的版本演进”和“用户视角的更新公告”。

所以它不是“日志罗列”，而是“有证据的重构”。

## 最低证据清单

每次写一个版本，最低建议同时看这几类证据：

| 证据 | 主要回答什么 |
| --- | --- |
| `git log --oneline --decorate` | 这一版附近有哪些关键提交、合并和分支 |
| `git tag` | 有没有正式 tag 锚点 |
| `README.md` | 这一版对外是怎么定义主题的 |
| `pom.xml` / 前端 `package.json` | 版本号有没有实际切换 |
| 文档计划 / 审计记录 / PRD | 这一版背后的技术或文档主题是什么 |

如果只看其中一类，很容易偏：

- 只看 README，容易过于抽象。
- 只看 commit，容易变成流水账。
- 只看 tag，容易漏掉没有打 tag 的真实版本批次。

## 推荐命令

最常用的是这几组：

```powershell
git log --oneline --decorate --max-count=50
git tag --sort=-creatordate
git log --grep="v2.2.0" --oneline --decorate
git log --grep="release" --oneline --decorate
git log --graph --oneline --decorate --max-count=80
```

如果你已经知道某个版本号，还可以补一组定向检查：

```powershell
git log --oneline --decorate --grep="v2.2.1"
git log --oneline --decorate --grep="bump version"
git log --oneline --decorate --grep="docs: update"
```

## 一版版本记录怎么拆

最稳的结构通常不是“功能点无序罗列”，而是下面这四段：

1. 这一版的主题是什么。
2. 这个主题有哪些 Git 证据支撑。
3. 对用户或产品来说，能感知到什么变化。
4. 对维护者来说，后续该回哪几页继续看。

可以直接套这个模板：

```md
## vX.Y.Z

一句话主题：

关键 Git 证据：

| 日期 | 提交 | 含义 |
| --- | --- | --- |
| 2026-05-20 | `abc1234` `feat: ...` | 说明这是这一版的核心落点 |

对外变化：

- 用户能感知到什么
- 管理端或维护流程有什么变化

延伸阅读：

- README 对应段落
- 路线图 / 审计页 / 模块页
```

## 怎么判断“这是一个版本”还是“只是一个补丁批次”

不是每一段提交都值得写成强锚点版本。可以按下面思路判断：

| 现象 | 更像什么 |
| --- | --- |
| 有版本号切换、README 更新、分支合入 | 强锚点版本 |
| 只有一串围绕同一主题的持续提交 | 补强批次 |
| README 明确写了一个版本，但 Git 没有单独 release commit | 产品语义版本 |
| 只有零散修复，没有共同主题 | 不要硬写成一个正式版本段 |

这里最重要的一条是：

**可以诚实地写“这是稳定化批次，不是单独 release commit”。**

不要为了看起来整齐，就假装每一版都有完美锚点。

## Git 演进视角和产品公告视角怎么分开

这两个视角最好不要混着写：

| 视角 | 主要回答什么 |
| --- | --- |
| Git 演进视角 | 仓库主线到底怎么演进，哪一版解决了什么系统性问题 |
| 产品公告视角 | 用户在产品里会看到哪些新功能、修复和体验变化 |

比如：

- `v2.2.0` 在 Git 演进视角里，重点是工程质量和文档治理。
- 但如果给用户写公告，可能只写“版本历史更清晰、部分页面稳定性优化、运维可观测补强”。

这并不矛盾，反而更准确。

## 推荐写作顺序

每次重写版本记录，建议按这个顺序来：

1. 先收集 Git 证据，不要急着下结论。
2. 再看 README 和版本号切换，确认对外语义。
3. 把提交按主题分组，而不是按时间机械罗列。
4. 写一句话主题。
5. 再补“关键提交表”和“用户能感知的变化”。
6. 最后把延伸阅读链到路线图、模块页或审计页。

## 这套方法最容易踩的坑

| 坑 | 更稳的处理方式 |
| --- | --- |
| 直接复制一堆 commit 当版本说明 | 先抽主题，再保留少量关键提交做证据 |
| 把 README 的营销摘要当成全部事实 | 用 Git 证据回校 |
| 没有 release tag 就说“没有版本” | 版本语义和 Git 锚点可以分开存在 |
| 新版主线已经切换了，还一直往旧路线图补 | 把当前工作线单独立出来 |
| 给用户写版本公告时塞太多内部术语 | 内部术语留给文档站，用户公告只写可感知结果 |

## 一个最小实践

如果你今天要重写一版记录，最低可以先完成这四步：

1. 跑 `git log --oneline --decorate --max-count=50`。
2. 找出和目标版本最相关的 3 到 6 条关键提交。
3. 用一句话概括这版主线。
4. 把这句话和关键提交表同步到 [版本历史](/modules/version-history) 或产品版本墙草稿里。

做到这一步，通常就已经比“凭印象写一段更新说明”稳很多了。
