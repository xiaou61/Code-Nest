# 按模块学习路线

如果 [学习路线](/guide/learning-paths) 回答的是“我是后端、前端、运维还是文档维护者，应该先看什么”，那这页回答的是另一件事：

我想按业务模块把 Code Nest 这套系统真正学明白，应该从哪一块开始，一块一块怎么往下走。

这种读法更适合下面三类人：

1. 你要接手整个项目，而不是只改一个页面或一个接口。
2. 你想先弄懂“业务闭环”，再回头分前后端和运维细读。
3. 你准备带新人，希望给对方一条像教材一样的学习顺序。

如果你希望每一站学习后都有固定交付物，可以配合 [模块学习任务模板](/guide/module-learning-workbook) 使用。它把每一站拆成必做任务、过关标准和统一回写模板。

## 先搭全局骨架

正式按模块读之前，先把下面 6 页过一遍。它们的作用不是记细节，而是帮你建立“这个项目大概长什么样”的第一张地图。

| 顺序 | 文档 | 先建立什么认识 |
| --- | --- | --- |
| 1 | [架构总览](/architecture/overview) | 先知道后端多模块、用户端、管理端、文档站、AI sidecar、OJ 沙箱和监控分别是什么 |
| 2 | [源码地图](/reference/source-map) | 知道页面、Controller、Service、SQL、脚本、文档素材分别去哪里找 |
| 3 | [模块总览](/modules/) | 先看到所有功能按学习、内容、平台能力、工具和后台是怎么分组的 |
| 4 | [端到端业务链路图](/manuals/business-flow-map) | 先看清账号、学习、OJ、AI、社区、聊天、运营之间怎么串成闭环 |
| 5 | [模块依赖地图](/reference/module-dependencies) | 明白哪些能力是横切能力，改一个点会牵连哪些模块 |
| 6 | [术语表](/reference/glossary) | 避免一开始就被 Sa-Token、RAG、Graph、ws-ticket、go-judge 这些词卡住 |

如果这 6 页读完后，你已经能说出“用户端、管理端、后端模块、数据库和运维配置分别在哪”，就可以开始真正的模块化学习。

## 怎么使用这页

按模块学习时，建议每一站都保持同一个节奏：

1. 先读“总览页”，知道这个模块为什么存在。
2. 再读“细分模块页”，看入口、接口、表和状态流转。
3. 然后补“横切能力页”，弄清权限、文件、通知、AI 或 WebSocket 之类的公共依赖。
4. 最后做一个 15 到 30 分钟的小练习，把阅读变成能复述、能验证、能交接的理解。

你不需要一天读完整个文档站。更好的节奏是每次吃透一个模块组，再进入下一站。

建议做法：每完成一站，就回到 [模块学习任务模板](/guide/module-learning-workbook) 交付一份模块画像卡、链路追踪单和验证记录单。

如果八站走完后你想做一次综合演练，可以继续看 [模块结业案例](/guide/module-learning-capstone)。它把 OJ、AI 学习资产、社区运营这些跨模块链路做成了示范答案。

如果你需要把学习结果做成可评审交付，最后一步建议套用 [模块学习评审表](/guide/module-learning-rubric)。

如果你是团队带教或想给自己排一个学习周期，直接继续看 [带教执行剧本](/guide/onboarding-execution-playbook)。

## 模块学习总顺序

| 阶段 | 模块组 | 为什么先学这一组 |
| --- | --- | --- |
| 1 | 账号、身份与安全边界 | 几乎所有功能都建立在登录态、当前用户和权限边界之上 |
| 2 | 学习成长基础盘 | 这是 Code Nest 最核心的业务主线，决定“用户为什么持续回来” |
| 3 | OJ 与积分反馈闭环 | 这是最典型的高反馈链路，能训练你读状态机、排行和奖励回流 |
| 4 | AI 面试、求职作战台与学习资产 | 这是项目里最复杂的高风险链路，涉及 Prompt、Schema、RAG、报告和资产沉淀 |
| 5 | 内容社区与创作矩阵 | 它最适合训练“用户写入 -> 平台能力 -> 管理端运营”这类完整闭环 |
| 6 | 平台能力与横切系统 | 文件、通知、聊天、敏感词这类能力被很多模块复用，后学更容易看出共性 |
| 7 | 后台运营、日志与发布交付 | 这部分决定系统能不能持续运营、排障和发版 |
| 8 | 工具、版本与长期维护补充 | 这一组更像项目尾部能力，适合最后补充全景认识 |

## 第一站：账号、身份与安全边界

先把账号体系弄清楚，后面读任何业务模块都不会迷路。

### 推荐阅读顺序

| 顺序 | 文档 | 读完后要能回答什么 |
| --- | --- | --- |
| 1 | [鉴权与用户体系](/modules/auth) | 为什么普通用户和管理员是两套登录域 |
| 2 | [用户账户与个人中心](/modules/user-account) | 当前用户资料、验证码、注册、副作用和状态字段怎么流转 |
| 3 | [权限与安全边界](/guide/security-boundaries) | 当前用户、后台权限、文件访问和敏感边界怎么控 |
| 4 | [API 路由索引](/reference/api-routes) | `/user`、`/admin`、公共接口分别在哪一侧 |
| 5 | [数据库字段阅读指南](/reference/database-field-guide) | 用户表、状态字段、逻辑删除和归属字段怎么读 |

### 这一站要盯住的重点

- 登录和鉴权到底是用户端、管理端两套会话，还是一套共享会话。
- 后端是不是从当前登录态取 `userId`，而不是信任前端传参。
- 用户状态、资料、头像、验证码这些看起来小的功能，是否会触发通知、文件或日志副作用。

### 这一站的小练习

任选一个需要登录的用户端接口，回答下面 4 个问题：

1. 它的前端入口页面在哪。
2. 它的 Controller 和 Service 在哪个模块。
3. 它依赖当前登录用户的哪几个字段。
4. 如果把这里改坏，最少要补哪些登录和权限验证。

## 第二站：学习成长基础盘

这一组是 Code Nest 的主业务地基。先学它，你会更容易理解后面的 OJ、AI 面试、学习资产和内容沉淀为什么都围着“成长闭环”打转。

### 推荐阅读顺序

| 顺序 | 文档 | 读完后要能回答什么 |
| --- | --- | --- |
| 1 | [题库与成长闭环](/modules/interview-and-growth) | 整个学习链路是怎么从题库、复习、成长分一直串起来的 |
| 2 | [面试题库](/modules/interview) | 题单、掌握度、复习算法、学习记录怎么落地 |
| 3 | [闪卡](/modules/flashcard) | 另一种轻量学习闭环如何组织卡组、卡片和学习记录 |
| 4 | [计划与学习小组](/modules/plan-team) | 计划、打卡、小组成员、任务和讨论如何形成协作学习 |
| 5 | [知识图谱](/modules/knowledge) | 图谱和节点如何承载知识结构化沉淀 |
| 6 | [核心链路教程](/manuals/core-workflows) | 用用户视角把学习主线再走一遍 |

### 这一站要盯住的重点

- 学习状态是如何被记录的，比如掌握度、复习记录、打卡、学习热力图。
- 不同学习模块之间到底是独立存在，还是会把统计、成长分、资产沉淀回流到同一个闭环。
- 管理端是否只做配置和审核，还是也会影响用户学习侧的数据。

### 这一站的小练习

从 [面试题库](/modules/interview) 或 [闪卡](/modules/flashcard) 里任选一条用户链路，写出：

1. 用户端页面入口。
2. 后端核心接口。
3. 至少一张核心表。
4. 学习完成后会更新的统计或状态。

## 第三站：OJ 与积分反馈闭环

OJ 是这个项目里最适合训练“完整业务链路思维”的模块。它同时包含题面、提交、判题、状态回写、排行和积分反馈。

### 推荐阅读顺序

| 顺序 | 文档 | 读完后要能回答什么 |
| --- | --- | --- |
| 1 | [OJ 判题系统](/modules/oj) | 题目、提交、判题状态机、赛事和榜单怎么跑通 |
| 2 | [积分与抽奖](/modules/points) | 首次 AC 奖励、签到、流水、余额和抽奖消费怎么回流 |
| 3 | [API 调用示例](/reference/api-examples) | OJ 提交、分页、登录态和积分接口怎么调试 |
| 4 | [测试与回归](/guide/testing-regression) | 改 OJ 或积分时最低该跑什么验证 |
| 5 | [验证记录与已知问题](/manuals/verified-scenarios) | go-judge、状态卡住或回归缺口过去是怎么记录的 |

### 这一站要盯住的重点

- 提交从创建到判题完成，中间经历了哪些状态。
- 判题结果写回后，哪些页面、排行、积分或统计会跟着变化。
- 这条链路里哪些步骤依赖外部服务，比如 go-judge，哪些是本地数据库状态推进。

### 这一站的小练习

把“用户提交一道 OJ 题”拆成 6 到 8 个步骤，至少写出：

1. 用户在哪个页面发起提交。
2. 提交记录什么时候创建。
3. 判题结果什么时候写回。
4. 首次 AC 奖励在什么条件下触发。

## 第四站：AI 面试、求职作战台与学习资产

这一站是 Code Nest 里最复杂、也最值得认真学的一块。它把 AI Runtime、结构化输出、求职闭环和学习资产沉淀都绑在一起。

### 推荐阅读顺序

| 顺序 | 文档 | 读完后要能回答什么 |
| --- | --- | --- |
| 1 | [AI Runtime](/modules/ai-runtime) | Prompt、Schema、RAG、Graph、回归和指标分别在哪一层 |
| 2 | [AI Schema 与治理](/reference/ai-schemas) | AI 场景是怎么注册、结构化和治理的 |
| 3 | [模拟面试与求职作战台](/modules/mock-interview-job-battle) | 会话、追问、报告、Job Battle 和 Career Loop 怎么串起来 |
| 4 | [学习资产](/modules/learning-assets) | AI 产出的内容如何转为可审核、可发布的学习资产 |
| 5 | [模块依赖地图](/reference/module-dependencies) | 这条链路依赖了哪些横切能力和高风险边界 |
| 6 | [验证记录与已知问题](/manuals/verified-scenarios) | sidecar、外部模型和结构化字段回归过去踩过哪些坑 |

### 这一站要盯住的重点

- AI 接口返回 `200` 绝对不等于功能正确，结构化字段、阶段状态和报告内容都要看。
- 学习资产不是单独一页，它是面试、求职、总结之后的“沉淀出口”。
- 这一组模块经常会牵连环境变量、外部模型、RAG sidecar、Schema 解析和前端展示层。

### 这一站的小练习

任选 [模拟面试与求职作战台](/modules/mock-interview-job-battle) 里的一个场景，写出：

1. 它使用了哪个 AI 场景或 Schema。
2. 结果是直接展示给用户，还是还会继续沉淀到学习资产。
3. 如果要回归验证，除了页面显示，还必须检查哪两个环节。

## 第五站：内容社区与创作矩阵

这一组最适合训练“用户写入 -> 内容审核/风控 -> 通知/互动回流 -> 管理端运营”的阅读方法。

### 推荐阅读顺序

| 顺序 | 文档 | 读完后要能回答什么 |
| --- | --- | --- |
| 1 | [社区与内容矩阵](/modules/community-content) | 社区、动态、博客、代码工坊在内容生态里分别扮演什么角色 |
| 2 | [社区帖子](/modules/community) | 发帖、评论树、热门缓存、AI 摘要和后台运营怎么配合 |
| 3 | [动态广场](/modules/moments) | 动态、互动、收藏、浏览去重和统计怎么维护 |
| 4 | [博客](/modules/blog) | 博客开通、草稿、公开页、分类标签和管理后台怎么联动 |
| 5 | [代码工坊](/modules/codepen) | 代码作品、模板、可见性和收藏/付费 Fork 怎么组织 |
| 6 | [前端渲染安全](/reference/frontend-rendering-security) | Markdown、代码片段、`v-html` 和高亮内容如何安全展示 |
| 7 | [敏感词风控](/modules/sensitive) | 文本发布前的风险控制和版本策略怎么接入 |

### 这一站要盯住的重点

- 内容模块看起来很多，但底层套路非常统一：登录、风控、文件、互动、通知、运营。
- 只验证“能发出来”远远不够，还要确认刷新后可见、详情页能看、管理端可检索或审核。
- 任何涉及 Markdown、HTML、代码片段、Diff 或 AI 摘要的地方，都要额外过一遍渲染安全。

### 这一站的小练习

从社区、动态、博客、CodePen 里任选一个模块，列出：

1. 用户写入入口。
2. 至少一个风控或渲染风险点。
3. 至少一个管理端入口。
4. 最低该做的“用户端 + 管理端”联动验证。

## 第六站：平台能力与横切系统

学到这里再回头读平台能力，你会更容易看出它们不是独立功能，而是被多个业务模块反复复用的公共底座。

### 推荐阅读顺序

| 顺序 | 文档 | 读完后要能回答什么 |
| --- | --- | --- |
| 1 | [文件存储](/modules/file-storage) | 文件上传、公开读取、本地映射、迁移和高风险操作怎么控 |
| 2 | [通知中心](/modules/notification) | 业务事件、公告、已读/未读和模板变量如何流转 |
| 3 | [IM 聊天室](/modules/chat) | ticket、WebSocket 连接、消息 ACK、禁言和踢人怎么跑 |
| 4 | [敏感词风控](/modules/sensitive) | 敏感词引擎、白名单、来源同步和检测策略怎么复用 |
| 5 | [WebSocket 协议](/reference/websocket) | 聊天协议层面有哪些事件和失败态 |
| 6 | [模块依赖地图](/reference/module-dependencies) | 这些平台能力分别被哪些业务模块调用 |

### 这一站要盯住的重点

- 文件、通知、敏感词、聊天这些能力通常不是“独立完成一个模块”，而是“每个业务都接一点”。
- 横切能力改动时，验证范围不能只停留在当前页面，必须回到调用它的业务链路去看。
- 文件 URL、WebSocket ticket、通知接收人、敏感词策略这些地方，都是上线后很容易出问题的点。

### 这一站的小练习

任选一个平台能力模块，回答：

1. 至少三个调用它的业务模块。
2. 改它时最容易被忽略的回归范围。
3. 一条最小但可信的验证路径。

## 第七站：后台运营、日志与发布交付

这一站的目标不是学会写某个后台页面，而是明白项目为什么能持续运营、排障和发版。

### 推荐阅读顺序

| 顺序 | 文档 | 读完后要能回答什么 |
| --- | --- | --- |
| 1 | [系统运营后台](/modules/system-ops) | 后台配置、AI 治理、版本维护和系统类运营入口都在哪里 |
| 2 | [仪表盘与日志](/modules/dashboard-logs) | 仪表盘聚合、健康检查、登录/操作日志怎么定位问题 |
| 3 | [环境变量总表](/operations/env-vars) | 本地、联调、生产环境最关键的变量有哪些 |
| 4 | [监控与观测](/operations/monitoring) | Actuator、Prometheus、Grafana 和 AI 指标怎么看 |
| 5 | [问题定位流程](/operations/diagnosis-flow) | 遇到未知问题时怎么从现象一路缩小到模块 |
| 6 | [发布前验证](/guide/release-verification) | 发版本前怎么选构建、烟测、回归和记录范围 |
| 7 | [文档维护规范](/guide/documentation-maintenance) | 功能变更后哪些文档必须同步更新 |

### 这一站要盯住的重点

- 后台和发布相关文档不是“最后补充看看”，它们决定你能不能把项目真正接住。
- 看到问题时，先定位属于页面、接口、权限、数据、外部依赖还是部署配置，再决定往哪一层查。
- 发布验证的本质不是多跑命令，而是按影响面选择最小但足够可信的验证组合。

### 这一站的小练习

假设你刚改了一个“带登录、带文件上传、带管理端审核”的功能，试着列出：

1. 需要检查的环境变量。
2. 需要跑的最低验证。
3. 需要同步更新的文档页。

## 第八站：工具、版本与长期维护补充

这一组不是项目主业务，但它很适合帮助你补全“这个系统还能承载哪些轻量能力、如何持续演进”的认识。

### 推荐阅读顺序

| 顺序 | 文档 | 读完后要能回答什么 |
| --- | --- | --- |
| 1 | [开发者工具](/modules/dev-tools) | 哪些是纯前端工具页，哪些有后台或数据依赖 |
| 2 | [摸鱼工具](/modules/moyu) | 外部热榜、薪资计算、程序员日历等轻工具如何落地 |
| 3 | [版本历史](/modules/version-history) | 版本记录、发布状态和用户可见更新怎么维护 |
| 4 | [角色上手任务包](/guide/onboarding-tasks) | 如果要把学习结果转成练习任务，该怎么做 |

### 这一站要盯住的重点

- 轻工具页面虽然简单，但也会暴露渲染安全、外部数据源、缓存和后台配置这些问题。
- 版本历史页能帮你理解“项目不是一堆静态模块”，而是一个持续发布、持续变更的产品。

### 这一站的小练习

任选一个工具页，判断它更接近下面哪一类：

1. 纯前端本地工具。
2. 依赖后台配置或外部数据源的工具。
3. 需要版本记录和运维关注的功能。

并说明你的判断依据。

## 推荐完整顺序

2. 学习成长基础盘。
3. OJ 与积分反馈闭环。
4. AI 面试、求职作战台与学习资产。
5. 内容社区与创作矩阵。
6. 平台能力与横切系统。
7. 后台运营、日志与发布交付。
8. 工具、版本与长期维护补充。

这个顺序的好处是：先学所有业务都会用到的底层规则，再学主业务，再学高风险复杂链路，最后补横切能力和发布维护。这样你越往后读，越容易把前面的理解重新串起来。

## 读到什么程度算过关

| 能力 | 过关标准 |
| --- | --- |
| 模块定位 | 看到一个功能名，能说出它属于哪个模块组、前端入口大概在哪、后端模块大概在哪 |
| 依赖判断 | 知道它会不会牵连鉴权、文件、通知、积分、AI、敏感词、WebSocket 或部署配置 |
| 数据理解 | 至少能说出一张核心表、一个关键状态字段或一个典型接口前缀 |
| 验证意识 | 知道这个模块最低要怎么验证，不会只停留在“页面能打开” |
| 文档维护 | 知道改这个模块后，应该回写哪些索引、模块页、验证记录或发布说明 |

如果你已经能做到这些，再回头按 [学习路线](/guide/learning-paths) 分角色深化，或者直接做 [角色上手任务包](/guide/onboarding-tasks)，效率会高很多。

如果你已经走完模块学习、评审和首单接入，下一步建议直接看 [模块接手升级路线](/guide/module-ownership-ladder)，把“学会模块”继续推进到“稳稳接住模块”。
