# 模块学习评审表

这页给模块化学习补上最后一个闭环：评审标准。

现在文档体系已经有三层：

1. [按模块学习路线](/guide/module-learning-paths)：告诉读者先学什么。
2. [模块学习任务模板](/guide/module-learning-workbook)：告诉读者每一站交什么作业。
3. [模块结业案例](/guide/module-learning-capstone)：告诉读者合格答案大概长什么样。

而这页要解决的是另一件事：

当一个人把作业交上来以后，团队怎么判断这份作业到底是“看过了文档”，还是“真的已经能接模块了”。

## 这页适合谁用

| 角色 | 怎么用 |
| --- | --- |
| 学习者本人 | 做完作业后自评，先找出明显短板再交给别人看 |
| 带教同学 | 用统一标准评阅，减少“感觉差不多”的主观波动 |
| 技术负责人 | 用它判断新人是不是可以开始接真实模块任务 |
| 文档维护者 | 用评审结果反推哪些文档还不够清楚 |

## 评审原则

评审模块学习作业时，最容易出现两个误区：

1. 只看字数，不看是否真的能落到源码、接口、表和验证。
2. 只看答案是否“看起来懂了”，不看对失败态、依赖和边界是否诚实。

所以这份评审表只看四类证据：

1. 能不能定位。
2. 能不能串链路。
3. 能不能判断风险。
4. 能不能给出可信验证和交接结果。

## 快速结论标准

如果你不想逐项细打分，先用这个快速口径：

| 结论 | 说明 |
| --- | --- |
| 通过 | 已经具备接一个低到中风险模块的基础能力 |
| 补充后通过 | 理解方向基本对，但定位、验证或交接还不够扎实 |
| 不通过 | 仍停留在文档复述层，不能支撑真实接手 |

推荐阈值：

| 总分 | 建议结论 |
| --- | --- |
| 85-100 | 通过，可开始接真实小任务 |
| 70-84 | 补充后通过，需返工 1 到 2 处关键短板 |
| 0-69 | 不通过，建议回到任务模板或结业案例重做 |

## 标准评分维度

总分 100 分，建议按下面 6 个维度评分。

| 维度 | 分值 | 看什么 |
| --- | --- | --- |
| 模块定位 | 20 | 是否能准确找到前端入口、后端模块、接口前缀、核心表 |
| 链路理解 | 20 | 是否能把起点动作、中间处理、状态变化、落库和回流串起来 |
| 风险判断 | 15 | 是否能识别鉴权、文件、AI、敏感词、WebSocket、外部依赖等关键边界 |
| 验证设计 | 20 | 是否能给出成功路径、失败态、依赖条件和未验证项 |
| 交接表达 | 15 | 是否能留下别人可以继续执行的记录，而不是只有自己看得懂 |
| 文档维护意识 | 10 | 是否知道哪些索引、模块页、验证记录或路线图需要同步更新 |

## 详细评分说明

### 1. 模块定位（20 分）

| 得分档位 | 标准 |
| --- | --- |
| 16-20 | 页面、后端模块、接口前缀、核心表都能准确落到具体位置 |
| 10-15 | 大方向对，但有 1 到 2 个位置描述过泛或不够准确 |
| 0-9 | 只能复述模块名，不能真正落到代码和数据结构 |

评审时重点看：

- 有没有把用户端和管理端入口分开。
- 有没有把后端模块写成具体 `xiaou-*` 模块。
- 有没有写出至少一张核心表或关键字段。

### 2. 链路理解（20 分）

| 得分档位 | 标准 |
| --- | --- |
| 16-20 | 能完整写出起点、处理中间层、状态变化、落库和回流 |
| 10-15 | 能说出主流程，但状态变化或回流环节不完整 |
| 0-9 | 只会描述页面行为，不能解释中间链路 |

评审时重点看：

- 是否把“页面显示”和“业务落库”区分开。
- 是否知道回流可能发生在列表、详情、通知、积分、日志、统计等多个地方。
- 是否能指出至少一个失败态。

### 3. 风险判断（15 分）

| 得分档位 | 标准 |
| --- | --- |
| 12-15 | 能主动识别高风险边界，并说明为什么危险 |
| 7-11 | 能识别部分风险，但遗漏关键横切能力 |
| 0-6 | 基本没提边界，或只写了“需要注意安全”这类空话 |

评审时重点看：

- 内容类是否提到了敏感词和渲染安全。
- AI/OJ/聊天类是否提到了外部依赖和失败态。
- 文件、鉴权、通知、积分是否被当成横切能力看待。

### 4. 验证设计（20 分）

| 得分档位 | 标准 |
| --- | --- |
| 16-20 | 成功路径、失败态、依赖条件、未验证项和补测建议都写清楚 |
| 10-15 | 有验证意识，但未验证项或依赖边界写得不够诚实 |
| 0-9 | 只写“跑一下看看”，或只写构建通过没有业务验证思路 |

评审时重点看：

- 是否把“能构建”与“真实业务可用”分开写。
- 是否写了未验证项及原因。
- 是否有至少一个失败态或边界条件验证想法。

### 5. 交接表达（15 分）

| 得分档位 | 标准 |
| --- | --- |
| 12-15 | 下一位同学拿到这份记录就能继续做，不需要大量口头补充 |
| 7-11 | 大体可读，但仍有关键上下文只存在作者脑中 |
| 0-6 | 输出高度主观，别人难以复用 |

评审时重点看：

- 表达是否具体，不用“这里”“那个接口”这种模糊指代。
- 已验证和未验证是否分开写。
- 是否明确给出下一步建议。

### 6. 文档维护意识（10 分）

| 得分档位 | 标准 |
| --- | --- |
| 8-10 | 能准确指出要同步哪些文档和为什么 |
| 4-7 | 知道要补文档，但范围不完整 |
| 0-3 | 基本没有文档同步意识 |

评审时重点看：

- 新页面是否想到前端路由索引。
- 新接口是否想到 API 路由索引。
- 新表字段是否想到数据表索引和模块页。
- 高风险链路是否想到验证记录、发布前验证或路线图。

## 评审示例 1：OJ 首次 AC 回流案例

下面给一个简化示例，帮助你理解怎么打分。

### 示例评语

```text
评审对象：OJ 首次 AC 到积分回流
评审模块：OJ 与积分反馈闭环
评审人：带教同学

一、模块定位（18/20）：
能准确指出 /oj/problem/:id、/oj/my-submissions、/points 等入口，也能识别 xiaou-oj 与 xiaou-points。
建议再把首次 AC 对应的具体状态字段或规则位置补得更具体。

二、链路理解（19/20）：
已把 submission 创建、go-judge、结果回写、积分回流拆成两段逻辑，理解比较完整。

三、风险判断（14/15）：
识别了 go-judge、重复发奖、状态卡住等关键风险，较好。

四、验证设计（16/20）：
知道要测 AC、WA、重复 AC，但还缺少一次“积分流水成功但余额未同步”的思考。

五、交接表达（13/15）：
记录基本可继续执行，建议把“未启动 go-judge 时无法验证”的前置条件写得更醒目。

六、文档维护意识（8/10）：
知道要回写模块页和验证记录，建议再补 API 示例或测试回归页的联动意识。

总分：88
结论：通过
必须返工项：无
建议加强项：补一条余额与流水不一致的失败态验证思路
推荐下一步模块：AI 面试到学习资产审核
```

### 为什么这个分数合理

它不是因为“写得多”，而是因为：

- 关键定位都对。
- 结果回写与奖励回流被分开看。
- 已经有明确的未验证边界。

扣分点也很清晰：

- 还可以更具体到规则或字段级。
- 失败态验证仍有提升空间。

## 评审示例 2：常见不过关情况

下面这种作业，表面上看写了很多话，但其实不能算通过：

```text
这个模块主要是用户提交代码，然后系统判题，通过以后会有积分奖励。
需要注意判题和积分的正确性。
验证方面需要跑一下功能，看是否能正常使用。
后续如果有问题再补充。
```

### 为什么不通过

因为它几乎在每个维度都缺证据：

| 维度 | 问题 |
| --- | --- |
| 模块定位 | 没写入口、模块、接口、表 |
| 链路理解 | 没拆中间处理和状态变化 |
| 风险判断 | “注意正确性”不是风险判断 |
| 验证设计 | 没写具体验证方式，更没有未验证项 |
| 交接表达 | 别人无法拿着这份记录继续做 |
| 文档维护意识 | 完全缺失 |

这种情况下，更合适的结论通常是：`不通过，回到模块学习任务模板重做`。

## 自评清单

在把作业交出去之前，建议先用这 8 个问题自查：

1. 我有没有写出具体页面入口，而不是只写模块名？
2. 我有没有写出具体后端模块、接口前缀和至少一张核心表？
3. 我有没有把状态变化写出来？
4. 我有没有识别至少一个横切能力和一个高风险边界？
5. 我有没有把已验证和未验证分开？
6. 我有没有诚实写出依赖没启动时的验证边界？
7. 我有没有写出别人可继续执行的下一步建议？
8. 我有没有说明这份作业会牵连哪些文档同步？

只要这 8 个问题里有 3 个以上答不上来，通常说明这份作业还不该提交评审。

## 评审后的推荐动作

| 评审结果 | 建议动作 |
| --- | --- |
| 通过 | 开始接一个低到中风险真实小任务 |
| 补充后通过 | 只返工关键短板，不建议整份重写 |
| 不通过 | 回到 [模块学习任务模板](/guide/module-learning-workbook) 重做，并对照 [模块结业案例](/guide/module-learning-capstone) 查漏补缺 |

如果你是带教同学，尽量不要只说“多看看源码”。更有帮助的反馈通常是：

1. 你漏了哪一个维度。
2. 为什么这个维度会影响真实接手能力。
3. 最小返工范围是什么。

这样评审才会变成真正的成长反馈，而不是抽象打分。

如果你已经准备把这套路线真正跑起来，可以继续看 [带教执行剧本](/guide/onboarding-execution-playbook)。它把 3 天游、5 天游、每日交付物和评审节点都串好了。
