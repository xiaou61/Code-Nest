# 角色上手任务包

这页不是新的目录索引，而是一组可以直接执行的小任务。适合新人第一次接手 Code Nest，也适合团队把“熟悉项目”变成可验收的 onboarding 流程。

建议先读完 [学习路线](/guide/learning-paths)，再从本页选择与你角色最接近的任务包。每个任务都尽量控制在半天以内，目标不是一次看懂所有源码，而是学会“从现象定位到页面、接口、表、验证记录”的方法。

如果你希望按业务模块而不是按角色训练，可以改走 [按模块学习路线](/guide/module-learning-paths)，并使用 [模块学习任务模板](/guide/module-learning-workbook) 作为交付格式。

## 怎么使用这页

每个任务包都按同一种节奏执行：

1. 先读指定文档，建立上下文。
2. 打开对应源码目录或页面入口，确认文档和代码能对上。
3. 写下你找到的入口、风险和验证命令。
4. 用 [测试与回归](/guide/testing-regression) 选择最低验证项。
5. 如果发现文档缺口，按 [文档维护规范](/guide/documentation-maintenance) 补齐。

完成一个任务包后，不要求你记住每个类名，但要能向别人说清楚“这个功能从哪里进、后端在哪里、数据落在哪、怎么验证”。

## 通用热身任务

适合所有角色，建议第一个完成。

| 项目 | 内容 |
| --- | --- |
| 目标 | 能把项目从目录结构、启动方式、模块分组和验证口径串起来 |
| 先读 | [架构总览](/architecture/overview)、[源码地图](/reference/source-map)、[本地完整启动剧本](/guide/startup-playbook)、[模块总览](/modules/) |
| 重点目录 | `xiaou-*` 后端模块、`vue3-user-front`、`vue3-admin-front`、`docs-site`、`docker` |
| 输出 | 一段 5 到 8 行的项目说明，说明后端、用户端、管理端、文档站、AI sidecar、OJ 沙箱分别负责什么 |

执行步骤：

1. 在源码地图里找到后端、用户端、管理端、文档站和部署目录。
2. 在模块总览里任选一个模块，确认它属于成长学习、内容社区、平台能力、运营后台还是基础设施。
3. 在启动剧本里选择“文档模式”或“最小业务模式”，记录对应命令。
4. 在测试与回归页里找到只改文档时的最低验证项。

验收标准：

| 自测问题 | 合格答案应该包含 |
| --- | --- |
| 文档站怎么启动 | `cd docs-site`，然后运行文档站脚本或构建脚本 |
| 后端业务代码放哪里 | 对应 `xiaou-*` 模块，不放到启动聚合模块里 |
| 用户端和管理端有什么区别 | 两个前端目录、两套路由入口、不同登录域和权限边界 |
| 改文档至少要验证什么 | `git diff --check` 和 `docs-site` 构建 |

## 后端开发任务包

适合要改 Controller、Service、Mapper、权限、表结构或业务状态的人。

| 项目 | 内容 |
| --- | --- |
| 目标 | 能从一个页面或接口定位到 Controller、Service、Mapper 和核心表 |
| 先读 | [后端模块](/architecture/backend-modules)、[API 路由索引](/reference/api-routes)、[API 调用示例](/reference/api-examples)、[数据表索引](/reference/database-tables)、[数据库字段阅读指南](/reference/database-field-guide)、[权限与安全边界](/guide/security-boundaries) |
| 推荐模块 | [OJ 判题系统](/modules/oj) 或 [社区帖子](/modules/community) |
| 输出 | 一张“接口前缀 -> Controller -> Service -> Mapper -> 表 -> 验证项”的小表 |

执行步骤：

1. 在模块页找到用户端入口和 API 分组。
2. 到 `xiaou-*` 模块里找对应 Controller，确认路由前缀和文档一致。
3. 继续顺着 Service、Mapper 找核心表。
4. 用 [模块依赖地图](/reference/module-dependencies) 判断它是否会牵连当前用户、管理员权限、敏感词、文件、积分、AI 或 WebSocket。
5. 按测试与回归页选择构建、单测或烟测范围。

验收标准：

| 自测问题 | 合格答案应该包含 |
| --- | --- |
| 这个接口属于用户端还是管理端 | 登录域、鉴权方式、是否需要 `@RequireAdmin` 或当前用户 |
| 表结构在哪里维护 | 数据表索引和 SQL 脚本入口 |
| 写接口时不能相信什么 | 前端传来的 `userId`、未校验的状态、未过滤的内容 |
| 最低验证怎么选 | 后端构建，加对应接口或页面烟测 |

建议记录模板：

```text
模块：
入口页面：
接口前缀：
Controller：
Service：
Mapper：
核心表：
权限边界：
最低验证：
```

## 前端开发任务包

适合要改用户端页面、管理端页面、API 封装、上传、Markdown 渲染或 WebSocket 的人。

| 项目 | 内容 |
| --- | --- |
| 目标 | 能从一个路由定位到页面组件、API 封装、模块页和安全边界 |
| 先读 | [前端应用](/architecture/frontend-apps)、[前端路由索引](/reference/frontend-routes)、[前端渲染安全](/reference/frontend-rendering-security)、[文件存储](/modules/file-storage) |
| 推荐入口 | 用户端社区、博客、OJ、聊天或管理端任意列表页 |
| 输出 | 一份页面风险清单，说明是否涉及登录、上传、`v-html`、WebSocket、分页、空态和失败态 |

执行步骤：

1. 在前端路由索引里选一个真实页面。
2. 找到 `vue3-user-front/src/views` 或 `vue3-admin-front/src/views` 中的页面文件。
3. 找到它使用的 API 封装和接口前缀。
4. 判断页面是否展示 Markdown、富文本、代码高亮、Diff、RAG 命中片段或后端 HTML。
5. 如果涉及上传，回到文件存储页确认公开读取和登录上传边界。
6. 如果涉及实时通信，回到 WebSocket 协议页确认 ticket、事件和失败态。

验收标准：

| 自测问题 | 合格答案应该包含 |
| --- | --- |
| 页面属于哪个前端 | `vue3-user-front` 或 `vue3-admin-front` |
| API 封装在哪里 | 对应 `src/api` 文件和后端前缀 |
| 内容渲染是否安全 | 是否走 Markdown 渲染、DOMPurify、`escapeHtml` 或统一工具 |
| 页面改完怎么验证 | 前端构建，打开真实路由，检查加载、空态、失败态和关键操作 |

## AI 与 OJ 任务包

适合要改 Prompt、Schema、RAG、Graph、模拟面试、SQL 优化或 OJ 判题的人。

| 项目 | 内容 |
| --- | --- |
| 目标 | 能分清普通 CRUD 和高风险外部依赖链路的验证差异 |
| 先读 | [AI Runtime](/modules/ai-runtime)、[AI Schema 与治理](/reference/ai-schemas)、[OJ 判题系统](/modules/oj)、[验证记录与已知问题](/manuals/verified-scenarios) |
| 推荐任务 | 梳理一个 AI 场景或一次 OJ 提交流程 |
| 输出 | 一份“输入 -> 处理 -> 外部依赖 -> 结构化结果 -> 失败态 -> 回归样例”的链路说明 |

执行步骤：

1. 任选一个 AI 场景，例如模拟面试、求职作战台、SQL 优化或社区摘要。
2. 找到 Prompt、Schema、调用入口、结果字段和失败处理。
3. 判断是否依赖 RAG、Graph、模型服务、缓存或指标上报。
4. 任选一个 OJ 提交链路，确认提交、排队、判题、结果回写和积分奖励之间的关系。
5. 在验证记录里确认哪些链路已有真实验证，哪些需要补测。

验收标准：

| 自测问题 | 合格答案应该包含 |
| --- | --- |
| AI 改动为什么不能只看 200 | 结构化字段、失败态、Schema、回归样例可能都出问题 |
| OJ 改动最怕漏什么 | go-judge 可用性、状态推进、结果回写、首次 AC 积分 |
| 怎么写验证结论 | 写清楚依赖是否启动、样例是否跑过、哪些只完成构建 |
| 文档要同步哪里 | AI Runtime、AI Schema、OJ 模块页、验证记录 |

## 运维与发布任务包

适合要部署、联调、排障、维护环境变量或发版本的人。

| 项目 | 内容 |
| --- | --- |
| 目标 | 能从启动失败或线上异常定位到服务、端口、代理、环境变量和日志 |
| 先读 | [本地开发](/guide/local-dev)、[Docker 与服务部署](/operations/docker)、[环境变量总表](/operations/env-vars)、[问题定位流程](/operations/diagnosis-flow)、[常见问题排查](/operations/troubleshooting)、[发布前验证](/guide/release-verification) |
| 推荐任务 | 选一个故障现象，按诊断流程写排查路径 |
| 输出 | 一份“现象 -> 证据 -> 可能模块 -> 验证命令 -> 结论”的排障记录 |

执行步骤：

1. 从常见问题排查里选一个现象，例如 401、CORS、WebSocket 连接失败、文件 URL 不对或后端启动失败。
2. 根据问题定位流程收集浏览器、终端、接口返回和服务日志证据。
3. 回到 Docker 与服务部署页确认端口、Nginx 路径、环境变量和外部依赖。
4. 用发布前验证页判断修复后应该跑哪些构建和烟测。
5. 把结论写成能放进验证记录的格式。

验收标准：

| 自测问题 | 合格答案应该包含 |
| --- | --- |
| 401 先查什么 | 登录域、Token、接口前缀、管理端权限 |
| CORS 先查什么 | Origin、代理路径、后端允许来源配置 |
| WebSocket 先查什么 | ws-ticket、Nginx 转发、Origin、事件错误 |
| 文件访问先查什么 | URL 前缀、`/api/files` 映射、`is_public` 和登录边界 |

## 文档维护任务包

适合继续补 v2.2.0 文档、写模块页、补索引、维护手册或做版本交接的人。

| 项目 | 内容 |
| --- | --- |
| 目标 | 能让新增代码、文档索引、操作手册和验证记录保持一致 |
| 先读 | [文档维护规范](/guide/documentation-maintenance)、[全功能覆盖矩阵](/reference/feature-coverage)、[核心链路教程](/manuals/core-workflows)、[v2.2.0 文档计划](/roadmap/v2.2.0-docs-plan) |
| 推荐任务 | 任选一个模块页，检查它是否有入口、源码、API、表、流程、风险和验证清单 |
| 输出 | 一份文档缺口清单，按“必须补、可以后补、无需补”分类 |

执行步骤：

1. 在全功能覆盖矩阵里选一个功能。
2. 打开对应模块页，检查是否能定位用户端入口、管理端入口、API 前缀和核心表。
3. 如果有页面或接口但索引没写，记录为必须补。
4. 如果模块页没有状态机、常见坑或验证清单，记录为可以后补。
5. 修改文档后，运行 `git diff --check` 和文档站构建。

验收标准：

| 自测问题 | 合格答案应该包含 |
| --- | --- |
| 新页面要同步哪里 | 前端路由索引、模块页、操作手册 |
| 新接口要同步哪里 | API 路由索引、模块页、错误码 |
| 新表要同步哪里 | 数据表索引、模块页、SQL 脚本说明 |
| 新高风险链路要同步哪里 | 发布前验证、验证记录与已知问题 |

## 任务完成记录模板

完成任意任务包后，建议用下面格式交接。短一点也可以，但“已验证”和“未验证”必须分开写。

```text
任务包：
选择的模块或页面：
读过的文档：
定位到的源码：
涉及的接口或表：
发现的风险：
已运行验证：
未运行验证及原因：
需要补充的文档：
下一步建议：
```

## 常见卡点

| 卡点 | 处理方式 |
| --- | --- |
| 不知道从哪个模块开始 | 从模块总览选一个你要改的业务，再回到学习路线找角色路线 |
| 页面能找到但接口找不到 | 用前端 API 封装里的路径回查 API 路由索引，再去后端 Controller |
| 接口能找到但表找不到 | 继续看 Service、Mapper 和数据表索引，不只停在 Controller |
| 文档和源码不一致 | 以源码为准，修正文档，并在路线图或验证记录写清楚校准内容 |
| 验证链路太多 | 先按测试与回归页选最低验证，再把没跑的依赖明确写出来 |
