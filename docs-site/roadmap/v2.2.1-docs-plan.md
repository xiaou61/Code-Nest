# v2.2.1 文档计划

`v2.2.1` 不是重新从零搭文档站，而是在 `v2.2.0` 已经完成“独立文档中心”之后，继续把**版本节奏、持续维护和新批次入口**补成可长期接力的形态。

如果说 `v2.2.0` 解决的是“有没有一套能用的文档中心”，那 `v2.2.1` 更关注三件事：

1. 当前工作线要有单独入口，不能所有新增批次都继续堆在 `v2.2.0` 下。
2. 版本更新记录要能按 `git log`、README、发布页面和真实提交证据持续重建。
3. 文档站要能明确区分“历史沉淀页”和“当前维护主线”。

## 这一阶段想补什么

| 方向 | 目标 |
| --- | --- |
| 当前计划入口 | 建立 `v2.2.1` 独立路线图，承接新的文档批次 |
| 版本记录方法 | 把“按 git log 重写版本更新记录”的方法固化成文档 |
| 站点版本语义 | 首页、导航、学习路径、维护规范等页能区分历史阶段和当前阶段 |
| 文档持续维护 | 让后续每次文档批次都知道该写去哪里，而不是继续堆旧计划页 |

## 与 `v2.2.0` 的关系

`v2.2.0` 路线图不会删除，它继续承担两个职责：

1. 作为“独立 VitePress 文档站建设期”的完整历史归档。
2. 作为已经完成的文档批次证明，方便回头追溯“这套体系最初是怎么补起来的”。

`v2.2.1` 则从“文档站已经存在”这个前提出发，继续补当前阶段的维护机制。

## 第一阶段：把 `v2.2.1` 立起来

| 文档 / 入口 | 状态 | 目标 |
| --- | --- | --- |
| `v2.2.1` 文档计划 | 已完成首版 | 给当前工作线一个明确入口 |
| 首页 / Nav / Sidebar 当前计划入口 | 进行中 | 让“当前主线”不再固定指向 `v2.2.0` |
| 学习路线 / 文档维护规范 / 发布前验证中的“当前计划”引用 | 进行中 | 把“新版本同步到哪里”切到当前主线 |

## 第二阶段：把版本记录方法写清楚

| 文档 | 状态 | 目标 |
| --- | --- | --- |
| 版本历史 | 进行中 | 继续承担“Git 演进视角 + 产品内版本墙视角”的双视角主页面 |
| Git Log 版更新记录写作手册 | 已完成首版 | 把“怎么根据提交、README 和版本墙重构 release notes”写成可复用方法 |
| 版本公告与发版交接模板 | 已完成首版 | 把版本证据直接落成 README、产品公告和团队交接单 |
| 版本公告与交接实战样例 | 已完成首版 | 用 `v2.2.0`、`v2.1.2` 示范完整的版本交付写法 |

这一阶段的重点不是多写一页“版本说明”，而是让以后每一版都能按统一方法去写，不靠记忆和口头同步。

## 第三阶段：把当前线和历史线分层

| 主题 | 预期结果 |
| --- | --- |
| 历史页 | `v2.2.0` 等旧路线图继续保留，作为完成批次归档 |
| 当前页 | `v2.2.1` 承接现在和后续的文档补强 |
| 版本页 | 版本历史讲演进，路线图讲当前写作与维护节奏 |

## 这一阶段的验收标准

做到下面这些，说明 `v2.2.1` 这条线就算真正立住了：

1. 首页能把“当前文档主线”指向 `v2.2.1`，而不是继续停留在 `v2.2.0`。
2. 新人看到“当前计划”时，不会误以为现在还在补最初骨架阶段。
3. 维护者想写版本更新记录时，能直接找到“按 git log 怎么写”的方法页。
4. `v2.2.0` 历史内容仍然保留，方便追溯文档站建设过程。

## 后续写作节奏

`v2.2.1` 之后，建议每次文档批次都按同一节奏推进：

1. 先判断这次是“补模块细节”还是“补维护机制”。
2. 如果属于新阶段，优先更新当前路线图，而不是只改零散页面。
3. 如果涉及版本语义、发布说明或 README 更新，同步更新版本历史或写作手册。
4. 完成后跑文档构建，并把本批次意义写回当前路线图。
